## 1. 文档目的

本文档在既有问题分析的基础上，正式引入 **「原型模板」** 概念，定义 **默认通用模板** 的形态与职责，并将 **模板壳 + 模型填肉**、**动效与状态 token**、**演示皮肤**、**提示词与少样本** 等优化方向收拢为可执行的优化路线图。  

**说明**：本文仅描述产品与交互层面的设计意图；接口字段、数据库表结构、具体代码改动在后续技术设计中展开。

---

## 2. 现状与问题摘要（结论回顾）

当前快原型在 **受约束 Spec** 与 **有限交互类型**（如 Tabs、Panel、`setTab`、`togglePanel`）下工作，存在：

- **交互词汇表偏薄**，页面切换多表现为 Tab 换区，观感偏「呆板」；
- **校验优先于展示质量**，模型输出易向「能过校验的及格结构」收敛；
- **缺少动效与细粒度 UI 状态**（加载、禁用、主次按钮层级等）的统一表达；
- **单根布局树**，与多页路由的心理模型存在差距。

优化不是单纯「换更大模型」，而是 **抬高表达上限**、**改变目标函数（在过校验前提下追求展示下限）**、**把重复结构交给模板**。

---

## 3. 新概念：原型模板（Prototype Template）

### 3.1 定义

**原型模板** 指：在生成 Spec（及可选的配套说明）时，预先注入的 **固定结构壳 + 样式与交互约定 + 可填充区域（槽位）**。  

- **模板负责**：整体信息架构骨架（如顶栏、侧栏占位、主内容区、默认 Tabs 容器）、合规的节点 id 约定、推荐 spacing/token、基础 interactions 骨架（若需要）。  
- **大模型负责**：在槽位内填充 **文案、Tab 标签、列表项、Badge、表单字段描述级内容** 等与需求强相关的部分，**不再从零拼装整棵易错且单调的树**。

### 3.2 价值（与前期建议对齐）

| 维度 | 无模板（现状倾向） | 有模板 |
|------|-------------------|--------|
| 输出稳定性 | 结构随机性大，易简陋或不合规 | 骨架固定，模型专注内容，**过校验率与观感下限同时提高** |
| 展示质量 | 依赖模型「一次画全图」 | **模板保证留白、分区、层级**，模型只填肉 |
| 迭代成本 | 改提示词难以系统性提升 | **换模板或调模板参数**即可批量影响新生成结果 |
| 与《页面设计文档》的关系 | 文档指导易落空 | 可约定「文档第 4～5 节 → 映射到模板槽位」的规则 |

### 3.3 模板的分层（建议）

1. **布局模板（Layout Shell）**：节点树固定层级，占位 id 稳定，便于渲染器做 **演示专用皮肤** 或动效绑定。  
2. **交互模板（Interaction Scaffold）**：预置合法的 `interactions` 片段（如主 Tabs 的 `setTab` 绑定），模型仅补充 `tabItems` 与 `contentId` 指向的子树。  
3. **视觉模板（Visual Preset）**：与 `theme` / `tokens` 或演示 CSS 绑定，定义「中后台演示风」「浅色卡片风」等，可与模板 ID 关联。

### 3.4 产品形态（建议）

- 用户或系统可选 **模板类型**（首期仅 **默认通用模板**，后续可扩展「仪表盘向」「表单向导向」等）。  
- 模板可由 **配置（JSON/YAML）+ 内置版本号** 管理，便于回滚与 A/B。  
- **导入报价《页面设计文档》** 时，默认绑定 **默认通用模板**，保证与现有流程无缝衔接。

## 4. 默认通用模板（Default Universal Template）

以下为 **首期建议的默认模板** 的产品级描述（非最终实现 JSON）。目标是：**通用中后台 / 工具型产品** 演示场景，兼顾 Spec 校验与展示下限。

### 4.1 设计目标

- 一屏内呈现 **「应用感」**：顶栏（产品名/占位操作）+ 可选侧栏占位 + **主内容区以 Tabs 组织多业务视图**（与现有语义校验「须有 Tabs」兼容）。  
- 每个 Tab 内容区内预留 **Card + 标题区 + 列表/徽章区**，便于展示《页面设计文档》中的模块能力，而不强制「一模块一 Tab」的机械映射（由提示词与文档约束，模板只提供 **可容纳多种分区的容器**）。  
- 主操作与次操作在结构上 **有层级**（例如一个 Primary Button 区 + 若干次要 Text/Button），减少「一片平铺文字」的呆板感。

### 4.2 结构骨架（逻辑层级）

1. **Page（根）**  
   - 子级：**顶栏 Container**（内含产品名 Text、占位操作区）。  
   - 子级：**主体 Row**（Grid 两列或「侧栏 + 主区」）：  
     - **侧栏**（窄）：导航占位 Text 或简化 Button 列表（可由模型填 2～4 项，或模板写死「概览 / 业务 / 设置」占位）。  
     - **主区**：**主 Tabs** —— 作为全页核心切换；`tabItems` 由模型根据《页面设计文档》的信息架构填写（2～6 个 Tab 为宜，提示词限幅）。  

2. **每个 Tab 的 content 根节点**  
   - **Card**：内含标题 Text、说明 Text（可选）。  
   - **Grid 或纵向 Container**：放置 **Badge 列表**（功能点/状态）、**Button**（操作占位）、**Panel**（可折叠详情，与 `togglePanel` 配合）。  

3. **Interactions**  
   - 主 Tabs：按钮或 Tab 头与 `setTab` 已由模板或生成器写好绑定关系，模型 **只填 tabKey 与 label**，避免漏绑。  
   - 可选：一个 **展开/收起** 的 Panel 与 Button 的 `togglePanel` 预连接，增强「可互动」观感。

### 4.3 与模型的分工边界

| 模板固定 | 模型生成 |
|----------|----------|
| Page 层级、主 Tabs 容器 id、侧栏/顶栏存在性 | Tab 的 `label`、`key`、各 `contentId` 下 Card 标题与说明文案 |
| 主 Tabs 的 interaction 骨架（source/target 固定） | 各 Tab 内 Badge 文本、Button label、Panel 内 Text |
| 默认 `theme`/`tokens` 或演示皮肤引用 | 与业务相关的具体措辞，须与《页面设计文档》第 2 节一致 |

### 4.4 默认模板的适用范围与非适用场景

- **适用**：B 端后台、运营工具、配置类、列表+详情类产品的 **早期对齐与演示**。  
- **非适用**（后续用独立模板）：营销落地长页、纯 C 端沉浸式单页、强地图/画布类产品（需另建模板或走 Mockup/HTML 通道）。

## 5. 与前期改进建议的整合方案

以下条目与《功能设计文档》中本地 EXE 等中长期方向 **不冲突**；本节聚焦 **Web 端 Spec 快原型** 可落地优化。

### 5.1 模板壳 + 模型填肉（核心）

- 生成流程调整为：**加载原型模板 → 将《页面设计文档》或用户 prompt 作为 user 内容 → 模型输出「模板槽位填充片段」或与模板合并后的完整 Spec**（实现上二选一：片段合并更安全）。  
- **验收**：同一需求下，对比「无模板 / 有默认模板」的预览截图与校验通过率。

### 5.2 动效与状态 Token（展示层）

- 在 Spec 允许字段中 **扩展枚举**（示例方向，具体枚举在技术设计中定稿）：  
  - 组件级：`variant`（primary / secondary / ghost）、`loading`、`disabled`；  
  - 过渡：`tabTransition`（none / fade）、`panelTransition`（none / height）。  
- **渲染器**（含导出 `preview/index.html` 内联脚本）实现 **有限几种 CSS 过渡**，避免模型输出任意 CSS。  
- **演示模式**（可选）：同一 Spec 在「编辑预览」与「客户演示导出」使用 **不同皮肤强度**（演示模式略增强阴影与过渡）。

### 5.3 提示词与少样本（Prompt）

- 在 `system` / `user` 中明确：**禁止**把附录表格行机械映射为 Tab；**必须**以《页面设计文档》第 2 节为叙事主线。  
- 内置 **1～2 个高质量 Spec 片段**（脱敏）作为 few-shot，展示信息密度与 Tab 命名风格。  
- 明确 **Tab 数量建议区间** 与 **每 Tab 内至少 N 个 Badge 或等价信息点**，防止空壳 Tab。

### 5.4 生成后微调（产品后续迭代）

- 提供 **轻量 UI**：Tab 顺序拖拽、文案就地编辑、隐藏某 Tab（不改后端，仅前端改 spec 再保存版本）。  
- 降低「一次生成定终身」带来的演示压力。

### 5.5 与 Mockup（HTML+CSS）通道的关系

- Mockup 通道可 **引用同一套视觉 Preset**（与模板 ID 对齐），使两种模式 **观感一致**。  
- 图像生成若引入，建议定位为 **风格参考图**，输入到提示词或视觉 Preset，**不替代**模板化 Spec 的主路径（见前文分析结论）。

## 6. 实施阶段建议（路线图）

| 阶段 | 内容 | 成功判据（示例） |
|------|------|------------------|
| **P0** | 默认通用模板配置化 + 生成管线接入（合并或槽位填充） | 新生成 Spec 默认带模板骨架；校验通过率不低于现状 |
| **P1** | 动效/状态 token 扩展 + 渲染器与导出预览同步 | 客户可感知 Tab/Panel 过渡；无任意 CSS 注入 |
| **P2** | Few-shot 与提示词收紧 + 与《页面设计文档》映射说明写入模板注释/文档 | 同需求下 Tab 命名与文档章节对应性提升 |
| **P3** | 演示皮肤开关 + 生成后轻量编辑 | 演示录屏满意度提升；减少为演示手动改 JSON |

## 7. 风险与注意事项

- **模板僵化**：模板过多会维护困难；首期 **只保留默认通用模板 + 明确扩展条件**（新客户类型再上新模板）。  
- **合并逻辑错误**：若采用「模型输出片段 + 服务端合并」，需严格 **JSON 合并规则与 id 冲突检测**，避免静默覆盖模板骨架。  
- **令牌与长度**：模板增大 prompt 体积，需监控 token 与截断策略，必要时对《页面设计文档》附录表格继续限行。
