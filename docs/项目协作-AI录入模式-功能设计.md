## 1. 背景与目标

在现有「项目协作」模块中，每个项目都有一张「任务表」多维表，用于记录需求 / 问题 / 任务。目前任务的创建方式主要是：

- 产品经理 / 管理员手动整理客户口述或会议纪要
- 再人工拆分为若干条任务，逐条录入多维表

这带来几个问题：

- 客户表达往往模糊、不成体系，产品需要大量时间「翻译」成结构化任务
- 截图、原型图等附件中的关键信息难以复用
- 手工录入效率低，容易遗漏细节

**目标**：在协作任务表页面提供一个「AI 录入模式」，基于：

- 客户的自然语言描述（口述/打字）
- 附件图片（需求截图、Bug 截图、设计图等）
- 后续可选的项目代码上下文（暂不实现）

通过通义·千问多模态模型，将其整理为**一条或多条结构化、颗粒度清晰的任务草稿**。用户在前端确认 / 编辑后，一键插入当前项目的「任务表」中。

文本类分析仍沿用 DeepSeek，涉及图片时使用通义·千问多模态模型。

---

## 2. 业务流程

1. 产品经理在「项目协作 → 某项目任务表」页面点击 **「AI 录入模式」**。
2. 在弹窗中：
  - 粘贴客户原始描述（会议纪要、聊天记录、需求说明等）
  - 可选：上传或选择协作中的图片附件
3. 点击「生成任务草稿」：
  - 前端调用后端 `POST /api/collab/projects/{projectId}/ai-tasks/preview`
  - 后端调用通义·千问多模态 API，根据任务表结构生成若干条任务草稿
4. 前端展示任务草稿列表：
  - 每条任务可编辑标题、描述、重要程度、当前状态、验收结果、负责人、归属模块等字段
  - 可以删除不需要的草稿
5. 用户确认后点击「插入任务表」：
  - 前端调用 `POST /api/collab/projects/{projectId}/ai-tasks/commit`
  - 后端将这些任务批量写入当前项目的协作「任务表」
6. 任务创建完成后，普通协作成员即可像手动创建的任务一样查看和编辑。

AI 仅生成草稿，不直接改动正式数据，所有任务最终仍由人工确认。

## 3. 后端设计（概览）

### 3.1 通义·千问客户端（QwenClient）

- 新增 `QwenClient`（类比现有 `DeepSeekClient`），封装对通义·千问多模态 API 的 HTTP 调用。
- 输入：`apiKey`、`model`（如 `qwen-omni`）、messages（system + user）。
- 输出：返回一段字符串内容，约定为 **JSON 数组** 字符串，用于解析成任务草稿列表。
- 仅用于「协作模块 AI 录入」场景，不影响现有「单次提交 AI 分析」逻辑。

### 3.2 配置项

在配置文件 / 配置表中新增通义·千问相关配置：

- `qwen.enabled`：是否启用千问多模态功能
- `qwen.apiKey`：通义·千问 API Key（由管理员在配置页填写）
- `qwen.model`：多模态模型名称，建议默认 `qwen-omni` 或适合图片理解的模型

当 `qwen.enabled = false` 或 `qwen.apiKey` 为空时，AI 录入接口返回友好错误提示。

### 3.3 协作 AI 录入接口（CollabAiTaskController）

新增控制器 `CollabAiTaskController`，挂载在 `/api/collab` 下，复用协作模块鉴权（`CollabAuthFilter`）：

- `POST /api/collab/projects/{projectId}/ai-tasks/preview`
  - 请求体：
    - `rawText: string` 客户原始描述
    - `attachmentIds: number[]` 可选，协作附件 ID 列表（图片）
  - 步骤：
    1. 校验协作用户对项目的访问权限
    2. 获取当前项目绑定的表结构（列 + 选项组），构造「任务表 schema」描述
    3. 构造 prompt：
      - `system`：你是资深产品经理，负责把原始描述 + 截图拆成若干「原子任务」
      - `user`：包含原始文字、附件信息、任务表 schema 说明，以及**严格 JSON 输出格式约束**
    4. 调用 `QwenClient`，返回字符串内容
    5. 使用 Jackson 解析为 JSON 数组，映射到内部 DTO `AiTaskDraft` 列表
    6. 返回 `data.items = drafts` 给前端
- `POST /api/collab/projects/{projectId}/ai-tasks/commit`
  - 请求体：
    - `tasks: AiTaskDraft[]`：前端编辑后的任务草稿列表
  - 步骤：
    1. 再次校验协作用户对项目的访问权限
    2. 根据项目的任务表结构，将每个草稿映射为 `Map<String,Object> fields`（`c{columnId}` → 值）
    3. 循环调用现有记录创建逻辑 `createRecord(tableId, userId, fields)`
    4. 返回 `createdCount` 等统计信息

## 4. 前端设计（概览）

### 4.1 入口与弹窗

在 `CollabTableView.vue` 顶部按钮区域新增：

- 次级按钮「AI 录入模式」，点击后打开 AI 录入弹窗。

弹窗包含三部分：

1. **原始输入区**：多行文本框（客户原始描述），后续可扩展附件上传 / 选择。
2. **生成按钮与状态**：点击「生成任务草稿」，展示加载状态与错误提示。
3. **任务草稿列表**：展示 AI 返回的任务数组，每条均可编辑标题、描述以及关键字段，可删除不需要的条目。

### 4.2 接口调用

- 生成任务草稿：
  - 调用 `POST /api/collab/projects/{projectId}/ai-tasks/preview`
  - 成功时将 `data.items` 绑定到本地 `aiTasks` 数组
- 插入任务表：
  - 调用 `POST /api/collab/projects/{projectId}/ai-tasks/commit`
  - 成功后关闭弹窗并刷新任务表数据

## 5. 安全与使用限制

- **权限控制**：仅已登录的协作用户，在有权限访问项目的前提下，才能使用 AI 录入模式。
- **隐私与合规**：
  - 启用千问多模态后，原始描述与部分截图内容会经由后端发送到通义·千问 API。
  - 对于含有高度敏感代码或数据的项目，可关闭该功能或限制仅管理员使用。
- **费用与配额控制**：
  - 通过限制单次输入文字长度与附件数量，控制单次调用 token 消耗。
  - 后端可选记录每次调用的模型与大致 token，用于后续成本分析。

---

## 6. 迭代与扩展方向

本版本仅实现「文字 + 图片 → 结构化任务草稿 → 人工确认后写入任务表」。后续可考虑：

- 引入「项目代码上下文摘要」，让 AI 在拆分任务时参考实际项目结构与接口
- 支持从已有任务或提交记录中反查相关信息，自动关联到新任务
- 对 AI 草稿增加「风险提示」字段，例如标记需求中存在的不确定项或待澄清问题
