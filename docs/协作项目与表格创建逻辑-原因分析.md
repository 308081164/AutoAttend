# 协作项目与表格创建逻辑 — 原因分析

## 现象

- 系统内能看到「三个项目」（例如首页仓库下拉、统计里的仓库数）。
- 进入「项目协作」后，只有**一个**项目有协作表格，另外两个项目没有表格或无法进入表格页。

## 两套「项目/仓库」数据来源

| 来源 | 接口/表 | 含义 |
|------|--------|------|
| **提交表仓库** | `GET /api/admin/repos` → `CommitService.listRepos()` → `SELECT DISTINCT repo_full_name FROM aa_commit` | 凡在 `aa_commit` 里出现过提交的仓库都会出现在首页「项目」下拉、仓库数统计里。 |
| **协作项目** | `GET /api/collab/projects` → `biz_project` 表 | 协作侧的项目列表；每个项目对应一张 `biz_project_table`（协作表格）。 |

二者**没有自动同步**：提交表里多出来的仓库，不会自动变成协作项目。

## 协作项目与表格的唯一创建入口

- **代码位置**：`CollabSyncService.ensureProjectAndTable(String repoFullName)`
- **调用时机**：仅在被 **GitHub Push Webhook** 调用时触发，即 `GithubWebhookController.onGithubWebhook()` 里：
  1. 解析 `payload.getRepository().getFull_name()`
  2. 调用 `collabSyncService.ensureProjectAndTable(repoFullName)`
  3. 若该 `repoFullName` 在 `biz_project` 中不存在，则**在同一事务内**创建：
     - 一条 `biz_project` 记录
     - 一条 `biz_project_table` 记录（默认「任务表」）
     - 默认列与选项组（`createDefaultColumnsAndOptionGroups`）

因此：

- **只有收到过 Push Webhook 的仓库**才会在协作侧有「项目 + 协作表格」。
- 若某仓库的提交是通过其他方式进入 `aa_commit` 的（例如历史迁移、手动录入、或未配置/未投递 Webhook），则该仓库会出现在「三个项目」里，但**不会**出现在协作项目表 `biz_project` 中，也就没有协作表格。

## 为何会出现「三个项目里只有一个有协作表格」

典型情况：

1. **三个仓库**都在提交表中有记录（所以首页/统计看到 3 个「项目」）。
2. **只有一个仓库**配置了指向本系统的 GitHub Webhook，并且至少触发过一次 Push，因此只有它调用了 `ensureProjectAndTable`，在 `biz_project` 和 `biz_project_table` 中各有一条记录。
3. 另外两个仓库从未通过 Webhook 调用 `ensureProjectAndTable`，因此没有协作项目，也没有协作表格。

另一种可能（较少见）：  
若历史上曾存在「只写 `biz_project` 不写 `biz_project_table`」的代码路径，也可能出现有项目无表格；当前代码中**不存在**这种路径，`ensureProjectAndTable` 始终是「项目 + 表」一起创建。

## 结论与建议

- **根因**：协作项目（及表格）的创建**只**在 GitHub Push Webhook 中执行；与「提交表里有哪些仓库」无直接关系，因此会出现「系统里录入了三个项目（仓库），但只有一个有协作表格」。
- **建议**：
  1. **补齐已有仓库**：对提交表里已有、但协作侧尚未存在的仓库，做一次「按仓库补齐协作项目」的同步（见下文实现）。
  2. **新仓库**：为每个需要协作的仓库在 GitHub 上配置 Webhook，并至少触发一次 Push，这样会自动创建协作项目与表格。
  3. 若希望「凡有提交记录的仓库都有协作表格」，可在部署或运维流程中定期或一次性调用同步接口。

## 已做修改（同步逻辑）

- 新增 **POST /api/admin/team/sync-collab-projects**：  
  遍历 `CommitService.listRepos()` 返回的每个 `repoFullName`，调用 `CollabSyncService.ensureProjectAndTable(repoFullName)`。  
- 效果：所有在提交表里出现过的仓库，都会在协作侧拥有对应项目及默认协作表格（已存在的不会重复创建）。  
- 使用方式：管理员在后台调用该接口一次（或通过脚本/定时任务调用），即可为当前所有「有提交记录的仓库」补齐协作项目与表格。
