# 平台监测台（AutoAttend 监测台）— 现状、文档依据与 SaaS 化扩展设计（审阅稿）

> **文档类型**：产品设计 / 架构对齐（供产品与技术审阅）  
> **版本**：v0.1  
> **关联仓库**：`platform_monitor_frontend` + `atuo_attend_backend` 下 `platform` 包  
> **目标**：在**订阅制 SaaS** 方向下，将监测台从「轻量指标页」演进为**平台运营与商业化可用的控制面**，并与历史文档中的设想对齐。

## 二、当前监测台实现程度（基于代码事实）

### 2.1 前端（`platform_monitor_frontend`）

| 能力 | 实现状态 | 说明 |
|------|----------|------|
| 独立应用与路由 | ✅ | `/login`、`/`（看板）、`/tenants`；`platform_ops_token` 存 localStorage |
| 登录 | ✅ | 密码登录，Bearer Token |
| **看板 KPI** | ✅ | 今日 DAU、当月 MAU、总用户、AI Key / GitHub Token **配置租户占比** |
| **DAU 趋势** | ✅ | 折线图，默认近 30 天（后端支持 1～90 天） |
| **今日活跃作者排行** | ✅ | 按提交数，含邮箱、名称、最后提交时间 |
| **组件/功能使用统计** | ✅ | 依赖主应用上报 `POST /api/admin/ops/events/component-click`；展示点击数、usage、core API 明细 |
| **租户页** | ⚠️ **偏薄** | 仅展示 ID、名称、Slug、联系方式、创建时间；**未展示**后端报表中的成员数、提交量、存储等 |
| 国际化 | ❌ | 界面以中文硬编码为主 |
| 响应式 / 可访问性 | ⚠️ | 基础布局，未按现代控制台标准做完整适配 |
| 角色 / RBAC | ❌ | 单密码即「平台运维」，无细粒度权限 |

### 2.2 后端（`/api/platform/*`）

| 能力 | 实现状态 | 说明 |
|------|----------|------|
| 平台会话鉴权 | ✅ | `PlatformAuthFilter` + `aa_platform_session` |
| `GET .../ops/metrics/overview` | ✅ | 见上 KPI |
| `GET .../ops/metrics/dau-trend` | ✅ | 按天序列 |
| `GET .../ops/metrics/active-authors` | ✅ | 当日排行 |
| `GET .../ops/metrics/component-usage` | ✅ | 组件 + core API 聚合 |
| `GET .../ops/reports/tenants` | ✅ **数据富、展示穷** | `PlatformTenantOpsRow` 含成员数、GitHub 绑定数、24h/7d 提交、日活近似、diff 存储近似等；**前端未用全** |
| `GET /api/platform/tenants` | ✅ | 简单租户列表（与报表部分重叠） |
| **订阅 / 订单 / 欠费** | ❌ | 无独立领域模型与 API |
| **全站错误率 / SLO** | ❌ | 无 APM 或统一错误聚合接入 |
| **租户级冻结 / 配额调整** | ❌ | 文档有提，API 未产品化 |
| **审计日志（平台操作）** | ⚠️ | 业务侧有部分审计；平台侧「谁冻结了谁」未形成标准产品能力 |
| **导出 / 报表任务** | ❌ | 无 CSV/异步导出 |

### 2.3 与《系统产品化分析报告》第 4.5 节对照

| 报告中的模块 | 当前 |
|--------------|------|
| 租户总览（状态、套餐、最近活跃、存储） | **部分**：有报表 API，UI 只展示基础列 |
| 健康与用量（请求量、错误率、业务数量聚合） | **弱**：无 HTTP 错误率、无按租户 QPS；仅有提交/Diff 相关近似与组件点击 |
| 财务（订阅、续费、欠费） | **无** |
| 风控（异常登录、Webhook 风暴、滥用） | **无** |
| 运维动作（冻结、强制下线 Token、调配额） | **无**（或仅数据库可手工改） |
| 报表导出 | **无** |

**总体评价**：监测台已具备 **「运营看数」雏形**（活跃、配置渗透、趋势、功能使用），与 **SaaS 商业化所需的「计费健康 + 风险控制 + 运维处置」** 仍有明显差距；与 `docs` 中的目标相比，约处于 **MVP 前期 / 报告「阶段 3」的一小段**。

## 三、现代化视角：SaaS 平台监测台还应具备的扩展能力

以下按**对订阅制商业化的直接价值**与**实施复杂度**分层，便于排期。术语对齐常见 **Internal Admin / Ops Console / Control Plane** 实践。

### 3.1 商业化与订阅（高优先级，产品化核心）

| 能力 | 说明 | 依赖/备注 |
|------|------|-----------|
| **订阅与套餐视图** | 每租户：当前套餐、计费周期、下次扣款日、试用结束日、是否欠费 | 需 `plan`、订单或支付提供商 webhook 落库 |
| **MRR / 流失风险** | 月度经常性收入、续费失败列表、试用转付费漏斗 | 需计费事件与标签 |
| **配额与用量对齐** | 成员数、仓库数、存储、AI Token（若平台代管）、API 调用次数 vs 套餐上限 | 与 `TenantQuota` 类中间层对齐 |
| **发票与对账导出** | 运营/财务导出周期账单（可先 CSV） | 可晚于支付接入 |

### 3.2 租户健康与「可观测性-lite」（高优先级，运维与客诉）

| 能力 | 说明 |
|------|------|
| **租户详情页** | 单页聚合：`ops/reports/tenants` 已有字段 + 最近 7 天趋势小图 + 关键配置缺失告警（未配 Webhook、未配 AI Key） |
| **错误与可用性** | 按租户聚合 **5xx 率**、关键接口失败次数（需日志结构化 `tenant_id` + 接入或自建聚合）；最小可行：**每日汇总表** + 阈值告警 |
| **异步任务与队列** | AI 分析任务、日报生成、Webhook 投递失败重试次数；**失败队列**便于排障 |
| **存储与成本信号** | Diff 存储、附件存储、MinIO bucket 用量趋势（已有近似字段可先做展示） |

### 3.3 风控与安全（中高优先级，SaaS 必备）

| 能力 | 说明 |
|------|------|
| **登录异常** | 异地、短时间大量失败、撞库特征（IP + 账号维度） |
| **API 滥用** | 单租户 QPS、单 IP 多租户注册、Webhook 突发流量 |
| **数据导出审计** | 谁在何时导出哪些租户数据（合规） |

### 3.4 运维动作（中优先级，需严格审计）

| 能力 | 说明 |
|------|------|
| **租户状态机** | `active` / `suspended` / `trial_expired`；暂停时对外提示文案（`ops-sla-oncall` 已提到） |
| **强制会话失效** | 使某租户下所有 admin/collab token 失效（安全事件） |
| **配额临时调整** | 补偿性延期、临时提额（双人审批可二期） |

### 3.5 体验与工程化（持续迭代）

| 能力 | 说明 |
|------|------|
| **完整展示租户报表** | 优先：**不新增接口**，先把 `PlatformTenantOpsRow` 全量字段与排序、筛选做进 UI |
| **全局搜索** | 按租户名、slug、管理员手机搜索 |
| **国际化** | `en.js` / `zh.js` 抽离监测台文案 |
| **与大屏/BI 对接** | 导出到 Metabase、飞书多维表格（后期） |

### 3.6 与现有「快捷运维（Nexus）」的边界

- **Nexus**：面向**租户管理员**的云资源运维（ECS、指标、成本），已在主产品内。  
- **平台监测台**：面向**你方平台运营**，关注**租户是否付费健康、是否滥用、全站是否挂**。二者勿混在一个导航；可在监测台提供 **「跳转至某租户管理员视角」** 仅限**受控模拟登录**（强审计，通常晚于 MVP）。

## 四、建议实施顺序（不含日历估算）

1. **展示层补齐**：租户列表使用 `ops/reports/tenants` 全字段 + 排序/筛选 + 租户详情页（纯只读，零新业务风险）。  
2. **订阅域最小模型**：租户表或扩展表增加 `plan_code`、订阅状态、周期结束时间（可与现有 `aa_tenant` / plan 对齐）；监测台只读展示。  
3. **错误与任务**：选一两种最关键路径（如 Webhook 处理、AI 任务）做**按租户失败计数**日报。  
4. **风控与冻结**：`suspended` + 审计日志 + 运营手册话术。  
5. **支付与 MRR**：接入支付后再做欠费与 MRR 看板。
