# 机会角 — 功能设计文档

## 1. 文档说明

- **功能名称**：机会角（含「人才角」「项目角」及项目结束后的协作评价闭环）
- **版本**：v0.1（草案）
- **目标**：为「有项目缺人手」与「有技能找项目」的用户提供匹配入口；项目关闭后沉淀**可复用的协作评价**，供后续项目方参考开发者协作能力（非唯一决策依据）。
- **依赖**：可与现有 AutoAttend 账号体系统一；具体技术栈与表结构在技术方案中展开。

---

## 2. 背景与产品目标

### 2.1 背景

- **人才角**：独立开发者展示简历化信息（技术栈、项目经历等），便于有临时开发需求的用户快速筛选与联系。
- **项目角**：展示正在招募开发者的项目（背景、角色、收益、发起者等），便于开发者发现合适机会。
- **评价闭环**：项目关闭后，发起人对参与者打分/评价，结果供其他发起者参考该技术成员的综合表现。

### 2.2 产品目标（指标可在立项后量化）

- 降低「找人」与「找活」的匹配成本。
- 在合规前提下形成可追溯、可解释的协作评价数据，提升重复匹配效率。

---

## 3. 概念与术语

| 术语 | 含义 |
|------|------|
| **机会角** | 一级入口，下挂「人才角」「项目角」两个子版块。 |
| **人才角** | 开发者展示个人档案，供项目方浏览与联系。 |
| **项目角** | 项目方发布招募信息，供开发者浏览、申请。 |
| **项目发起人** | 创建并管理某条招募项目的用户。 |
| **参与者** | 经确认加入该项目的开发者（与「仅浏览/收藏」区分）。 |
| **协作评价** | 项目在**已关闭**后，发起人对已确认参与者打的分数/评语，用于他人参考。 |

---

## 4. 用户角色与权限（概要）

| 角色 | 典型能力 |
|------|----------|
| **访客** | 可浏览公开列表与部分详情（联系方式可见性由策略控制）。 |
| **登录用户（开发者）** | 维护人才角档案；在项目角申请/沟通；查看与自己相关的评价摘要。 |
| **登录用户（项目方）** | 发布/编辑/关闭项目招募；在项目结束后对参与者评价。 |
| **平台运营（可选）** | 审核、下架、处理举报；配置展示规则与敏感词。 |

---

## 5. 功能模块总览

```
机会角
├── 人才角
│   ├── 开发者档案（简历、技术栈、项目经历、可协作方式等）
│   ├── 列表与筛选、排序
│   ├── 详情与联系/沟通入口（站内信或既有能力对接）
│   └── （可选）收藏、举报
└── 项目角
    ├── 招募帖（背景、角色、收益、发起人信息、状态）
    ├── 列表与筛选、排序
    ├── 报名/申请与发起人处理（通过/拒绝）
    ├── 项目生命周期（招募中 → 进行中 → 已关闭）
    └── 关闭后：发起人对参与者「协作评价」
```

**与评价的衔接**：评价数据在人才角以**聚合展示**为主（如平均分、评价条数、标签）；详情可按权限展开，并附「仅供参考」说明。

---

## 6. 人才角 · 功能设计

### 6.1 目标用户

- 主要：独立开发者 / 可接临时活的工程师。
- 次要：需要找人的项目方（只读 + 联系）。

### 6.2 开发者档案（建议字段）

**基础信息**：展示名、头像、一句话简介、常驻城市/时区（可选）、可用协作方式（远程/驻场/混合）。

**技术栈**：技术标签（多选）+ 熟练度（了解 / 熟悉 / 熟练 / 主导过）；（可选）GitHub、作品集外链。

**项目经历**：多条记录——项目名称、角色、时间、简述、技术点、成果（可外链）；是否可公开。

**简历与附件（可选）**：富文本或 PDF（限制大小与格式）。

**协作偏好**：期望类型（短期单模块 / 整包交付 / 长期兼职等）；可投入时间（如每周小时数区间）。

**与评价体系的展示**：协作评分摘要（如近 N 次项目均分、评价条数）；标签/关键词（由评价结构化或运营规则生成）；说明文案：评分来自历史项目发起人，仅供参考。

### 6.3 列表与筛选

- **筛选**：技术栈、城市/时区、可用性、评分区间、经历类型等。
- **排序**：综合推荐（规则待定）、最新更新、评分等。
- **搜索**：关键词（技术栈、简介、项目名）。

### 6.4 关键流程

- 创建/编辑档案；首次进入人才角可引导完善；支持草稿与发布。
- 项目方查看详情；联系方式按策略展示或走站内沟通。

---

## 7. 项目角 · 功能设计

### 7.1 招募帖（建议字段）

**项目背景**：标题、详细描述、行业/领域标签、技术栈要求。

**需要角色**：可多角色（如前端/后端/全栈）；每角色人数、必备技能、加分项。

**时间与投入**：预计周期、每周期望投入、开始时间窗口。

**收益与方式**：预期收益（区间或面议）；结算方式（按小时 / 按里程碑 / 固定包等，枚举可配置）。**免责声明**：平台仅信息发布，签约与纳税由双方自行约定（文案需法务审核）。

**发起者信息**：昵称/认证标识、历史发起项目数、（可选）历史评价信誉。联系方式策略与人才角对称。

**状态**：`招募中` / `进行中` / `已关闭`（是否仍展示「进行中」由产品定）。

### 7.2 列表与筛选

- **筛选**：技术栈、收益类型、远程/驻场、周期、领域。
- **排序**：最新、急招、匹配度（后期）、发起人信誉（后期）。

### 7.3 报名与协作（MVP 建议）

- 开发者**申请**，发起人**通过/拒绝**；通过后即参与者（或「候选人→确认参与者」二段式，需二选一写死规则）。
- 站内通知：申请结果、项目状态变更。

### 7.4 项目关闭与评价时机

- 发起人将项目置为**已关闭**（或到期自动关闭，规则待定）。
- **仅对已确认参与者**开放评价入口。
- （可选）评价截止时间：关闭后若干天内可评，逾期关闭入口。

---

## 8. 协作评价 · 功能设计

### 8.1 评价维度（可配置）

示例：技术能力、沟通与响应、交付质量、守时与靠谱度；每项星级或分数 + 可选短评（字数限制）；可设总评。

### 8.2 展示规则（给其他发起人参考）

- **人才角档案**：聚合分、条数、最近若干条评价（发起人昵称是否展示需合规与防骚扰权衡）。
- **（可选）项目角发起者侧**：展示「作为发起人的评价质量」以防恶意刷分（规则后期细化）。

### 8.3 反作弊与争议（原则）

- 评价仅绑定**真实项目与参与者关系**入库。
- 举报与申诉入口；刷分、水军策略在技术方案中细化。
- 评价修改策略：原则上不可随意改，或仅短时内更正/平台仲裁（待定）。

---

## 9. 关键业务流程

1. 开发者完善人才角档案 → 在项目角浏览 → 申请 → 发起人通过 → 协作 → 项目关闭 → 发起人评价 → 评价聚合到开发者档案。
2. 发起人发布项目 → （可选）审核 → 招募中 → 选人 → 进行中 → 关闭 → 对参与者打分。
3. 其他发起人在人才角查看评分与经历 → 联系 → 闭环。

---

## 10. 数据与隐私（概要）

- **最小必要**：联系方式、实名、简历附件分级可见。
- **评价数据**：告知用途与展示范围；用户可查看与本人相关的评价（在权限内）。
- **合规**：劳务/招聘相关风险在界面与用户协议中提示（法务审核）。

---

## 11. 非功能需求（建议）

- **性能**：列表分页；接口响应与现网协作模块对齐或单独 SLA。
- **安全**：防 XSS、上传校验、接口频率限制。
- **可运营**：发布审核、敏感词、封禁与下架。

---

## 12. MVP 与迭代

**MVP 建议**

- 机会角入口 + 人才角核心档案 + 项目角核心招募帖 + 列表筛选。
- 状态：招募中 / 已关闭；关闭后发起人对参与者**单向评分 + 短评**。
- 人才角展示**平均分 + 条数**。

**迭代方向**

- 智能匹配、双向评价、合同/里程碑模板、更深 IM 集成等。
