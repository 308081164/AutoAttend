# 需求 → 报价计算器 → 合同 半自动化产出 — 功能设计文档

## 1. 文档说明

- **功能名称**：需求→报价计算器→合同半自动化产出（外包工作室报价与合同工具）
- **版本**：v0.3（补充「功能模块 AI 智能录入」：自然语言经 DeepSeek 解析为结构化模块/功能点，与手工、预设并存）
- **目标**：在 AutoAttend 中新增**需求→报价→合同**的半自动化链路，使**管理员**能够通过**点选/结构化方式**快捷录入或关联需求，系统按**人天基准 + 风险系数**计算报价并生成**报价单**，再通过 **AI 自动生成合同**，减少 Excel/Word 手工报价与合同撰写时间（目标：单项目从 2–3 小时压缩到 15–30 分钟），同时通过**结构化需求 + 置信度 + 人工审核**保障报价准确性。
- **定位**：面向外包工作室的「生存工具」——项目启动前必做的报价与合同环节，标准化、可追溯、可学习。
- **依赖**：现有 AutoAttend 技术栈（Vue、Spring Boot、MySQL）；大模型 API（合同生成）；可选与「多维表格」需求记录联动；PDF 生成（服务端或前端库）。

## 2. 功能目标与范围

### 2.1 目标

- **需求录入**：管理员通过**结构化点选**（项目类型、技术栈、功能模块与功能点+复杂度）、**从预设库点选**、**从多维表格需求记录一键带入**，以及可选的 **AI 智能录入**（自然语言 → DeepSeek → 结构化模块清单），快速形成可计算的需求清单。**原则**：AI 输出仅作**初稿**，必须落在与手工相同的表单结构中供人审阅后再保存/报价；禁止不经人工确认的「纯自由文本直出报价」。
- **报价计算**：基于**人天基准库**（按技术栈×复杂度）计算基础人天，再叠加**风险系数**（需求变更、技术风险、沟通成本、维护预留、紧急程度等）与**地域/市场单价**，输出报价金额与**置信度评分**；支持**人工调整**与**审核清单**确认后再生成报价单。
- **合同生成**：在报价确认后，将**报价摘要 + 关键条款**提交 AI，自动生成**合同正文**（可编辑），支持公司模板/水印，导出 Word 或 PDF。
- **准确性保障**：**三层机制**——（1）结构化需求拆解；（2）动态校准引擎（人天基准 + 风险系数 + 可选市场锚定）；（3）人工审核与反馈闭环（置信度、审核清单、实际成本回填以校准算法）。

### 2.2 范围（本期）

| 在范围内 | 不在范围内（可后续迭代） |
|----------|---------------------------|
| 管理员端：需求结构化录入（项目类型/技术栈/功能模块与功能点+复杂度）及 **AI 智能录入（自然语言解析为模块/功能点）** | 客户自助录入、多角色审批流 |
| 人天基准库（内置按技术栈×复杂度）、风险系数配置与计算 | 实时抓取外包平台价格、多地域实时单价 |
| 报价计算：基础人天 + 风险调整 + 单价 → 报价金额 | 多币种、多税率、复杂分成 |
| 置信度评分（基于需求完整度等）、报价审核清单（勾选确认） | 全自动报价无需人工确认 |
| 报价单生成（PDF/HTML，含公司信息与明细） | 在线电子签章、法律存证 |
| 基于报价单关键信息 + AI 生成合同正文（可编辑） | 合同多版本对比、法务审核流 |
| 可选：与多维表格「需求记录」关联，从表格带入需求摘要 | 从 Git/PR 自动识别需求 |
| 历史报价/合同归档、可选实际人天回填用于校准 | 多团队效率系数、市场价爬虫 |

## 4. 功能需求详述

### 4.1 需求录入（结构化）

- **原则**：以**结构化点选为主**，减少「客户说做个类似淘宝的页面」这类模糊输入导致的报价偏差。
- **项目基础**（必填/点选）：  
  - **项目类型**：企业官网、电商小程序、管理后台、APP、其他。  
  - **技术栈**：Vue+Node、React+Java、小程序原生、Flutter、其他。  
  - **设计需求**：已有设计稿、需 UI 设计、套用模板。  
  - **数据迁移**：无、需从旧系统迁移、需第三方接口对接。
- **功能模块与功能点**：  
  - 支持**多模块**（如用户系统、商品管理、订单系统）；每模块下**多条功能点**。  
  - 每条功能点：**名称** + **复杂度**（简单/标准/中等/复杂/极复杂）+ **数量**（默认 1）。  
  - 复杂度定义见 4.2；支持从预设库点选常用功能点（如「手机号登录」「微信 OAuth 登录」）快速添加。  
  - **AI 智能录入（补充）**：管理员可切换到「AI 智能录入」模式，**粘贴或上传**（`.txt` / `.md` 等纯文本）自然语言需求描述；后端调用 **DeepSeek**（与合同生成等共用管理后台 **AI 配置** 中的 API Key），返回与表单一致的 **`modules[]` 结构**（`name`、`items[].name`、`items[].complexity`、`items[].quantity`）。前端支持 **替换全部现有模块** 或 **追加到现有模块之后**；填入后用户在同一界面继续**手工增删改**复杂度与数量，再保存项目。**约束**：单次需求正文建议上限约 **2 万字**；请求**不落库**解析原文，仅返回结构化 JSON。复杂度枚举与系统一致：`simple` / `standard` / `medium` / `complex` / `extreme`。
- **非功能需求**（可选）：  
  - 并发用户量级：&lt;100、100–1000、1000–10000、10000+。  
  - 安全要求：普通、金融级、政府级。  
  - 部署方式：单机、云服务器、容器化/K8s。
- **与多维表格联动**（可选）：若启用，可从「多维表格」中选择某项目的需求记录，将**问题描述/归属模块**等摘要带入当前报价项目，再补充功能点与复杂度，避免重复录入。

### 4.2 复杂度与人天基准

- **复杂度定义**（内置行业基准）：

| 复杂度 | 定义 | 参考人天 | 示例 |
|--------|------|----------|------|
| 简单 | 纯展示，无交互，无后端 | 0.5–1 天 | 静态页、文章详情 |
| 标准 | 常规 CRUD，标准组件 | 1–2 天 | 用户注册、列表管理 |
| 中等 | 需定制开发，有业务逻辑 | 2–4 天 | 支付流程、权限系统 |
| 复杂 | 算法/性能/第三方对接 | 4–8 天 | 推荐算法、高并发优化 |
| 极复杂 | 创新功能，技术预研 | 8 天+ | 实时音视频、区块链 |

- **人天基准库**：按**技术栈 × 复杂度**存储参考人天（可配置），例如：

| 技术栈 | 简单 | 标准 | 中等 | 复杂 | 极复杂 |
|--------|------|------|------|------|--------|
| Vue+Node | 0.5 | 1.5 | 3 | 6 | 10 |
| React+Java | 0.6 | 1.8 | 3.5 | 7 | 12 |
| 小程序 | 0.4 | 1.2 | 2.5 | 5 | 8 |
| Flutter | 0.7 | 2 | 4 | 8 | 15 |

- **计算**：各功能点人天 = 基准人天 × 数量；**项目总人天** = Σ 功能点人天（可再乘以团队效率系数，见 4.5）。

### 4.3 风险系数与报价计算

- **风险系数**（可配置，百分比叠加）：  
  - **需求变更风险**：+10%～+20%（如首次合作、无明确 PRD、传统行业客户）。  
  - **技术风险**：+10%～+15%（新技术栈、需预研）。  
  - **沟通成本**：+5%～+10%（异地、时差、非技术背景客户）。  
  - **维护预留**：+10%～+15%（含 1 年维护、无明确验收标准）。  
  - **紧急程度**：-10%（标准周期）或 +20%～+30%（加急）。
- **单价**：支持按**地域/市场**配置人天单价（如一线 ¥1500–2500/天，新一线 ¥1000–1800/天），或固定单价。
- **公式**：  
  - **基础报价** = 总人天 × 人天单价。  
  - **风险调整** = 基础报价 × (1 + Σ 风险系数)。  
  - **最终报价** = 基础报价 + 风险调整（或合并为一口价展示）。

### 4.4 置信度与人工审核

- **置信度评分**（0–100）：  
  - 依据：需求是否含详细 PRD、功能点数量与粒度、是否存在「类似 XX」等模糊参照、是否有历史相似项目可参考、复杂度跨度等。  
  - **高（90–100）**：可直接生成报价，提示「基于充分信息」。  
  - **中（70–89）**：生成报价，标注「建议人工复核以下 X 点」。  
  - **低（50–69）**：生成区间报价（如 ±30%），建议人工确认。  
  - **极低（&lt;50）**：建议先补充信息再生成。
- **报价审核清单**（生成报价单前可强制勾选）：  
  - 是否限制需求变更次数（如 ≤3 次）；  
  - 是否明确验收标准（功能清单/测试用例）；  
  - 是否约定付款节点（如 3-4-3、5-5）；  
  - 是否包含部署上线费用；  
  - 是否明确维护期及范围。  
  未勾选时可提醒，但不阻断生成。

### 4.5 报价单与合同生成

- **报价单**：  
  - 内容：项目名称、需求摘要、功能模块与功能点明细（含人天）、风险说明、总人天、单价、总价、付款建议、公司信息与 Logo。  
  - 输出：PDF 或 HTML，支持公司水印/品牌。
- **合同生成**：  
  - **输入**：报价单中的项目名称、总价、付款节点、交付范围摘要、维护期等关键信息 + 公司名称、客户名称（管理员填写）等。  
  - **AI**：将上述结构化信息与合同模板要求（如「软件开发合同」「含保密与知识产权」）组装为 prompt，调用大模型生成**合同正文**。  
  - **输出**：富文本/Word 可编辑，管理员可修改后再导出 PDF/Word。  
  - **模板**：支持预设「软件开发合同」「维护服务合同」等模板类型，在 prompt 中指定。  
  - **信息完整性**：当前实现中传入 AI 的字段仍偏少，与「标准软件开发协议」常见要素存在差距；**对照表与迭代建议见第 11 节**。

### 4.6 实际成本反馈与校准（可选）

- **项目结束后**：管理员可录入**实际人天**、偏差原因（如需求变更、技术难点）。  
- **系统**：  
  - 记录「预估人天 vs 实际人天」，用于后续**团队效率系数**（实际/预估均值）校准。  
  - 可选：对「含某类功能点」或「某客户类型」的项目自动建议风险系数微调。  
- **本期**：可仅落库实际人天与原因，校准算法为后续迭代。

## 7. 前端流程与界面建议

- **流程**：**选择/创建报价项目** → **结构化需求录入**（项目基础 + 功能模块与功能点）→ **计算报价**（展示人天、金额、置信度、风险提示）→ **审核清单勾选** → **生成报价单** → **填写客户/公司信息** → **AI 生成合同** → **编辑合同** → **导出**。
- **需求录入页**：  
  - 项目类型/技术栈/设计/数据迁移等下拉或单选。  
  - **功能模块区**：提供 **录入方式** 切换——**手工录入**（默认）与 **AI 智能录入**。AI 模式下展示多行文本框、**上传文本文件**、**替换/追加**选项与「DeepSeek 分析并填入」按钮；成功后**同一页**下方模块表格立即更新，可继续编辑、从预设添加、删行。  
  - 功能模块列表，每模块下功能点列表；功能点支持从「预设库」点选（名称+复杂度）或手动输入。  
  - 实时或按需计算并展示「当前预估人天」与「置信度」简要提示。  
- **报价结果页**：  
  - 展示总人天、基础报价、各项风险系数与调整后金额、最终报价。  
  - 置信度等级与「建议复核」项。  
  - 审核清单（复选框），通过后允许生成报价单。  
- **合同页**：  
  - 表单：客户名称、公司名称、合同类型等。  
  - 一键「AI 生成」后展示富文本，可编辑。  
  - 导出 Word/PDF。
- **合同补充信息（报价项目同页，与需求/计算衔接）**：  
  - 除付款计划、质保/SLA、交付物勾选、验收异议期、验收标准补充说明、里程碑外，提供 **验收测试用例 / 测试清单** 表格，字段包括：**用例名称**、**归属模块**（与附件一模块名对齐）、**优先级**（高/中/低）、**前置条件**、**测试角色（系统角色）**、**测试步骤**、**预期结果**、**测试结果**（待测试 / 通过 / 驳回）、**备注**。  
  - 支持 **手工增删行** 与 **AI 生成**（`POST /api/admin/quote/ai/acceptance-test-cases`）：模型仅允许使用已提供的模块名与功能点上下文，并结合 PRD、验收补充说明输出可执行步骤；生成初稿的「测试结果」统一为「待测试」，**须人工核对**后再保存。数据存入 `quote_contract_context_json.acceptanceTestCases`。  
  - **附件二 HTML** 将上述用例渲染为合同级测试清单；**合同 AI Prompt** 通过 `formatContractContextForPrompt` 注入用例条数与摘要，减少正文与附件矛盾。

## 10. MVP 迭代建议

- **第一阶段**：项目类型/技术栈/功能模块与功能点（含复杂度）、人天基准表、基础人天计算、人天单价配置、报价金额计算、简单置信度（如按功能点数量与 PRD 是否填写）、报价单 PDF 生成、公司信息配置。

- **第四阶段**：多合同模板、市场价区间提示（若接外部数据）、多角色与审批流（若需要）。  
- **合同完整性专项**（可与第二～四阶段穿插）：按 **第 11.5 节** 分步落地——优先扩充合同 AI 的 Prompt 输入（功能清单、PRD、审核清单等），再结构化付款与质保，最后交付物/里程碑与多附件导出。

## 11. 标准软件开发协议要素与系统能力对照（合同信息完整性）

> **背景**：当前合同由 AI 根据**少量结构化字段 + 通用模板提示**生成，容易在「支付方式、交付与验收细度、需求变更、知识产权细节」等方面偏泛化或与项目实际不一致。本节结合**行业常见必备条款**（参考公开资料中常见的软件开发/外包合同结构与中国大陆实践）与**本系统已落库能力**做对照，便于产品与迭代排期。

### 11.1 行业常见的「正文」必备信息（中国大陆软件开发/外包语境）

下列为商务与法务场景中**高频出现、宜写入正文或明确引用附件**的要素（不同项目可删减，但缺失易导致履约争议）：

| 大类 | 常见应约定内容 |
|------|----------------|
| **合同首部与主体** | 合同名称、编号（可选）、签订日期与签订地点；甲乙双方**法定名称**、统一社会信用代码、住所、法定代表人/授权代表、**联系人及联系方式**；乙方**收款账户**（开户名、账号、开户行）。 |
| **标的与范围** | 软件开发/服务的对象描述；**包含与不包含**的边界（如不含硬件、不含第三方平台年费、不含超出附件需求的功能等）。 |
| **需求与交付标准** | 需求来源（如附件《需求说明书》/功能清单）；**功能实现程度**（如以附件清单为准、以双方确认 UI/接口为准）；**变更机制**（变更申请、评估、确认、对价款与工期的调整方式）。 |
| **交付物** | 源代码归属与交付形式（仓库/压缩包）、**环境部署说明、数据库脚本、接口文档、测试报告**等是否包含及交付时间点。 |
| **进度与里程碑** | 阶段划分（如设计确认、联调、UAT、上线）；各阶段**交付物与计划时间**（可与付款节点挂钩）。 |
| **验收** | 验收标准（功能清单、性能指标或测试用例）；**验收流程**（提测、反馈期限、复测、验收通过认定）；未通过时的整改期限与反复验收规则。 |
| **价款与支付** | 合同总价（是否含税、税率/发票类型）；**付款阶段、比例或金额、触发条件**（如签约、里程碑、验收、质保开始等）；发票开具时点与逾期付款责任。 |
| **质保与维护** | 免费质保/维护**期限**；范围（缺陷修复 vs 新需求）；响应时间（可选 SLA）；质保期外收费方式。 |
| **知识产权** | 委托开发成果归属（一般约定验收后归甲方）；**乙方保留的展示权**（案例宣传、脱敏）；**第三方组件、开源软件**的许可与合规责任分担。 |
| **保密** | 保密信息范围、期限、例外、违约后果。 |
| **双方协作** | 甲方提供资料、配合测试、环境账号；乙方项目经理与汇报机制等（可简可繁）。 |
| **违约责任** | 延期交付、质量不达标、需求单方变更、付款逾期等情形下的违约金或赔偿计算方式（注意合理上限）。 |
| **不可抗力** | 定义、通知、后果处理。 |
| **通知与送达** | 联系地址、电子邮件是否构成送达等。 |
| **争议解决与法律适用** | 协商 → 诉讼或仲裁；**管辖法院或仲裁机构**；适用法律（如中华人民共和国法律）。 |
| **生效、变更与终止** | 生效条件、份数；合同变更须书面；提前终止、已完工作结算方式。 |

**说明**：以上为**内容维度清单**，不等于法律意见；具体条款需由商务/法务根据公司模板与客户情况审定。

### 11.2 行业常见的「关键附件」

| 附件类型 | 作用 |
|----------|------|
| **附件一：需求说明书 / 功能清单** | 与合同正文「标的」挂钩；通常需**双方确认**（签字/盖章/邮件确认），作为验收依据。 |
| **附件二：报价单或费用明细** | 列明人天、单价、优惠、税费说明等（本系统报价单 HTML/PDF/Word 可承担此角色）。 |
| **附件三：项目实施计划 / 里程碑表** | 阶段、时间、交付物，常与付款节点对应。 |
| **附件四：验收标准 / 测试要点** | 可操作的验收条款，减少「是否算做完」争议。 |
| **附件五（可选）：保密协议 / 团队成员保密承诺** | 项目涉密时单独成附件或引用模板。 |

### 11.4 当前系统「尚未实现或明显薄弱」的环节（与标准要素对比）

| 缺口 | 说明 |
|------|------|
| **合同 Prompt 未使用完整需求** | AI 生成合同时**未注入**：功能清单明细、PRD 摘要、非功能维度、已选**风险项说明**、**审核清单**具体结论。易导致正文「标的与验收」泛化，与报价单不一致感强。 |
| **付款方式无结构化建模** | 审核清单仅有「是否约定付款节点」**布尔项**，无**比例、节点名称、触发条件、发票与税率**等字段，无法稳定生成「价款与支付」专条。 |
| **验收与实现程度** | **已支持**：`acceptanceTestCases[]` 测试用例表 + 附件二 HTML 导出 + 合同 Prompt 摘要。**仍薄弱**：完成定义（DoD）、性能指标、浏览器兼容等仍依赖人工写入补充说明或正文。 |
| **需求变更** | 无变更单流程与对合同正文/附件的引用方式。 |
| **交付物清单** | 未结构化勾选（源码、文档、脚本、培训等），合同交付条易默认空泛。 |
| **里程碑与日历** | 有预计工期（天），无**阶段划分与计划日期**字段，难以生成附件三式计划表。 |
| **质保与维护** | 审核清单仅有「维护期及范围明确」勾选，无**质保月数、响应时间、是否收费**等字段。 |
| **主体与账户信息** | 仅客户名/公司名，无法人、地址、统一社会信用代码、**收款账户**等，需人工在编辑器补全。 |
| **知识产权与开源** | 无开源/第三方组件清单；Prompt 仅原则性要求「含知识产权条款」，**无项目级事实输入**。 |
| **争议解决与签订地** | 无配置项，完全依赖 AI 泛写或人工改。 |
| **附件体系** | 未实现「主合同 + 多附件」编号、自动生成附件正文（如从功能清单生成《附件一》）及在合同中自动引用。 |
| **电子签章 / 法务流程** | 已在范围外（见 2.2），保持不变。 |

### 11.5 建议的产品迭代方向（对齐合同完整性）

1. **合同生成输入增强（高优先级）**  
   - 将 **PRD 摘要、模块/功能点表格（可摘要或限长）、项目类型与技术栈、非功能维度、审核清单勾选结果、已选风险标签及说明** 组装进 AI Prompt（必要时对超长清单做「表格 Markdown + 截断策略」）。  
   - 与报价单内容保持同源，减少「报价单很细、合同很粗」的落差。

2. **付款与质保结构化（高优先级）**  
   - 增加可选字段：**付款阶段**（名称、比例、条件）、**含税/税率/发票类型**（简化版即可）、**免费质保月数**、**维护响应级别**（可选）。  
   - 写入 Prompt 与/或生成独立「附件：付款计划表」段落。

3. **验收与交付物（中优先级）**  
   - 项目级：**交付物多选**（源码、部署文档、API 文档、测试报告等）；**验收异议期（工作日）** 简单数字字段。  
   - **（已部分落地）** 结构化 **测试用例/测试清单** 与 AI 初稿生成、附件二导出。  
   - 可选：**验收标准短文模板**（按项目类型预置，可编辑）作为补充说明默认值。

4. **主体信息模板（中优先级）**  
   - 维护「我方公司信息」字典：统一社会信用代码、地址、联系人、**默认收款账户**；合同页一键带入 Prompt 或正文占位区。

5. **里程碑与附件自动化（低优先级）**  
   - 里程碑表可手工填或根据「预计工期」生成粗粒度阶段；支持导出「附件一：功能清单」「附件三：进度表」为独立 HTML/Word 并在合同首部列明附件清单。

## 附录 A：报价准确性保障三层小结

| 层级 | 要点 |
|------|------|
| **第一层：结构化需求** | 强制点选项目类型/技术栈/功能点+复杂度；**AI 智能录入**将自然语言**转化为**同一套结构化字段，且须**人工核对**后保存；降低「需求模糊」导致的报价偏差。 |
| **第二层：动态校准** | 人天基准库（技术栈×复杂度）+ 风险系数 + 地域单价；可选历史实际人天反馈校准团队效率系数。 |
| **第三层：人工兜底** | 置信度评分 + 报价审核清单 + 最终报价与合同由管理员确认后生成；项目结束后可选录入实际人天。 |

**原则**：结构化输入 &gt; 智能算法 &gt; 人工兜底；第一层做不好，后续易「垃圾进垃圾出」。报价计算器的价值在于把隐性经验显性化、把遗漏风险提前暴露，**最终报价仍由人确认**，决策质量提升。
