# 流帮Project — 系统产品化分析报告

> **文档目的**：在现有「单部署、单组织」形态基础上，梳理向**多租户平台化**演进、并增设**平台运营监测后台**所需的分析结论、扩展项与落地路线。  
> **范围**：产品与架构层面；不涉及具体代码实现。  
> **依据**：当前仓库 README、后端管理鉴权与协作模块设计、公开接口划分及常见 SaaS 实践。

---

## 一、现状摘要（基于当前实现）

### 1.1 产品形态


| 维度        | 现状                                                                                               |
| --------- | ------------------------------------------------------------------------------------------------ |
| **部署假设**  | 典型为**一套后端 + 一套数据库**服务一个组织；README 与路由描述均按「单实例管理端」叙述。                                              |
| **管理端账号** | 管理员登录依赖配置中的账号密码（如 `application.properties` / 环境变量），**非多租户用户表体系**；鉴权为 Bearer Token，偏「单管理员/单团队」模式。 |
| **协作端**   | `BizUser`、项目、多维表、权限（super_admin / 子管理员 / 成员）已具备**组织内 RBAC**，但缺少**跨组织隔离的租户主键**在全部业务表上贯通。          |
| **业务数据**  | GitHub 集成、仓库与提交、报价与合同上下文、AI 配置、团队管理、Webhook 等，在逻辑上均为**当前实例全局数据**。                                |
| **AI 成本** | 用户自带 API Key 执行推理，**平台不承担模型推理费**，有利于平台规模化；仍需承担**调度、存储、日志与限流**带来的算力与存储成本。                         |


### 1.2 与「平台化」的主要差距

1. **租户隔离**：缺少统一的 `tenant_id` / `org_id` 及全链路传递策略（HTTP 上下文、ORM 拦截、对象存储路径前缀等）。
2. **注册与开通**：无自助注册、组织创建、邀请码/邮件域、套餐与订单等**平台级生命周期**。
3. **配置与密钥**：GitHub Token、AI 配置等现为**实例级**；多租户下需按租户隔离存储与轮换策略。
4. **Webhook 路由**：GitHub Webhook 需能映射到**唯一租户**（多路径、多密钥或签名中携带租户标识）。
5. **可观测性**：面向**平台运营者**的租户级用量、错误率、付费状态、健康度尚未独立为「监测后台」产品。

## 二、目标形态定义

### 2.1 「面向所有用户的平台」

建议明确定义三层主体，避免需求蔓延：


| 角色           | 说明                                                                 |
| ------------ | ------------------------------------------------------------------ |
| **平台运营方（你）** | 拥有「监测后台」：看全平台租户健康、商业化指标、风控与运维。                                     |
| **租户（组织）**   | 一家企业 / 一个工作室：拥有独立数据边界、管理员、成员、项目与计费主体。                              |
| **终端用户**     | 租户内的管理员与员工（协作账号），权限模型可延续现有 super_admin / 子管理员 / 成员，但**作用域限定在本租户**。 |


### 2.2 「监测后台」与「租户管理后台」的关系


| 系统                  | 使用者        | 核心目标                                                               |
| ------------------- | ---------- | ------------------------------------------------------------------ |
| **租户管理后台**（现有产品演进）  | 各租户的 admin | 配置 GitHub、报价、协作、团队等**本组织业务**。                                      |
| **平台监测后台**（新建或独立应用） | 平台内部运营/运维  | **跨租户**只读为主：DAU、存储、API 错误、欠费、异常行为、功能使用统计；必要时**受限运维动作**（冻结租户、重置配额）。 |


二者应 **URL、鉴权、数据模型分离**，避免平台高权限与租户日常操作混在同一套接口与 Cookie 中。

## 三、多租户架构选项与建议

### 3.1 三种常见模式


| 模式                   | 做法                        | 优点           | 缺点          | 与当前项目匹配度                   |
| -------------------- | ------------------------- | ------------ | ----------- | -------------------------- |
| **独立库**              | 每租户一库                     | 隔离最强、便于私有化交付 | 运维成本高、迁移复杂  | 适合**企业私有化大单**，不适合作为默认 SaaS |
| **共享库 + 租户列【选用此模式】** | 业务表统一 `tenant_id`，查询强制带条件 | 成本可控、单库备份简单  | 设计疏漏易导致数据串租 | **推荐作为 SaaS 默认路径**         |
| **Schema 分片**        | 每租户一 schema               | 介于两者之间       | 迁移与连接池管理复杂  | 租户数极大时再考虑                  |


**建议**：SaaS 公网平台采用 **共享库 + 租户 ID + 行级策略**；若未来签**私有化企业版**，可评估「独立库 + 同一代码多配置」的交付方式。

### 3.2 必须在「全链路」贯彻的隔离点

- **HTTP 层**：解析 JWT / Session 后得到 `tenantId`，写入 **ThreadLocal / RequestContext**（或等价机制），供 Service 与 DAO 使用。  
- **数据访问**：禁止手写 SQL 遗漏 `tenant_id`；优先 **全局拦截器 / MyBatis 插件 / JPA 过滤器**（需评估现有框架）。  
- **缓存与队列**：Redis Key、消息体均需带租户前缀，避免键冲突与跨租消费。  
- **对象存储**：MinIO bucket 前缀或 bucket-per-tenant，并配合 ACL。  
- **Webhook**：`POST /api/webhooks/github` 需能解析 **租户**（如子路径 `/webhooks/github/{tenantSlug}` 或签名 secret 映射表）。  
- **定时任务**：每日总结、统计任务等 **按租户分批** 或带租户过滤，避免单任务扫全表拖垮库。

## 四、平台化需扩展的功能清单（按域）

### 4.1 身份、租户与商业化

- **组织注册**：邮箱/手机验证、组织名称、子域或短链标识（slug）。  
- **邀请与成员**：管理员邀请成员加入组织（延续现有协作用户体系，但绑定 `tenant_id`）。  
- **套餐与配额**：仓库数、成员数、存储空间、AI 调用次数（若未来平台代扣）、Webhook 条数等；**免费档 + 付费档**。  
- **计费与订阅**：与支付渠道对接（国内微信/支付宝、海外 Stripe 等）、发票与对账（可后续阶段）。  
- **平台级管理员账号**：与租户管理员完全分离的账号体系与 RBAC（仅监测后台）。

### 4.2 配置与密钥（多租户）

- **GitHub**：每租户 PAT 或 GitHub App 安装，**加密存储**，展示脱敏。  
- **AI 配置**：延续「用户自带 Key」策略时，仍建议 **按租户存储** 与审计；可选「平台代填 Key」作为增值服务。  
- **邮件/短信**：用于通知、找回密码、邀请（可选第三方服务）。

### 4.3 安全与合规

- **审计日志**：租户内关键操作 + 平台侧敏感操作（冻结、导出数据）。  
- **数据导出与删除**：GDPR/个保法视角下的**用户数据导出与租户注销**。  
- **速率限制**：按租户 IP + 用户限流，防止滥用 Webhook 与登录接口。

### 4.4 可观测性与运维

- **结构化日志**：`tenant_id`、`request_id`、错误码统一。  
- **指标**：QPS、延迟、DB 连接、外部 API（GitHub）失败率，按租户聚合。  
- **告警**：单租户异常飙升、磁盘、任务失败重试耗尽。

### 4.5 监测后台（产品功能建议）

以下为**平台内部**使用，与租户后台分离：


| 模块        | 功能要点                                        |
| --------- | ------------------------------------------- |
| **租户总览**  | 列表：租户名、状态、套餐、创建时间、最近活跃、存储用量。                |
| **健康与用量** | 各租户请求量、错误率、存储、协作表数量、报价项目数（可选脱敏聚合）。          |
| **财务**    | 订阅状态、续费时间、欠费标记（若已接支付）。                      |
| **风控**    | 异常登录、Webhook 风暴、单 IP 多账号、API 滥用阈值。          |
| **运维动作**  | 只读为主；必要时「冻结租户」「强制下线 Token」「调整配额」（需双人审批或审计）。 |
| **报表导出**  | CSV/Excel 导出给运营分析（权限严格控制）。                  |


技术形态上可为：**独立前端子应用 + 独立后端 `/api/platform/*`**，或 **独立仓库 + 只读从库 + 事件流**；第一阶段可用 **同一库只读账号 + 强鉴权** 简化，但长期建议监测与业务库读写分离。

## 五、推荐落地路线（分阶段）

### 阶段 0：准备（1～2 周）

- 冻结当前单租户版本为 **LTS 分支/标签**，便于私有化交付与回滚。  
- 文档化现有表结构与 **所有「全局」API**，标出必须加 `tenant_id` 的表。  
- 定稿：**租户标识**（slug）、**域名策略**（是否多子域）、**计费模型**（免费额度）。

### 阶段 1：数据模型与隔离骨架（4～8 周，视人力）

- 引入 `tenant` / `organization` 表；核心业务表加 `tenant_id` 及索引。  
- 实现 **Request 级租户上下文** 与 **DAO 层强制过滤**（可先覆盖读多写少的模块验证）。  
- 管理端登录改为 **用户表 + 归属租户**（替代纯配置文件单账号，或并存迁移期）。  
- **迁移脚本**：单租户历史数据写入 `tenant_id = 默认租户`。

### 阶段 2：自助开通与基础商业化（并行 4～10 周）

- 注册、创建组织、基础邀请。  
- **套餐与配额** 中间层（可先硬编码档位，再接入支付）。  
- Webhook URL 改造与 **GitHub 侧配置文档** 更新。

### 阶段 3：监测后台 MVP（4～6 周）

- 独立路由与鉴权；仅平台角色可访问。  
- 先接 **日志 + 基础 SQL 报表**（租户列表、日活估算、存储大小），再迭代 **可视化仪表盘**。  
- 定义 **SLA 与 on-call 流程**（哪怕先人工）。

### 阶段 4： hardening

- 渗透测试与串租专项测试。  
- 备份与按租户恢复演练。  
- 性能与容量规划（连接池、分库分表评估）。

**说明**：各阶段周数为**粗估**，与团队规模与是否重构前端路由强相关。

## 六、风险与对策


| 风险      | 对策                           |
| ------- | ---------------------------- |
| 迁移期数据串租 | 灰度环境、自动化集成测试、代码审查清单、ORM 强制过滤 |
| 单点故障    | 多实例部署、数据库主从、对象存储多副本          |
| 平台密钥泄露  | KMS、轮换、最小权限、审计               |
| 合规      | 隐私政策、用户协议、数据处理协议分阶段上线        |
| 范围失控    | 严格区分「租户功能」与「监测后台」需求，分里程碑交付   |

## 七、执行层面检查清单（非代码）

- 确定 SaaS 是否 **必须** 多子域（影响 Cookie、CORS、Webhook）。  
- 确定 **免费版** 与 **付费版** 功能边界（与 MVP 未实现清单对齐）。  
- 法务：用户协议、隐私、若涉及合同生成内容的免责声明。  
- 运维：监控告警接收人、备份保留周期、灾难恢复 RTO/RPO。  
- 商务：定价、试用天数、企业私有化报价与主线关系。

## 八、结论

当前项目在 **协作、报价、GitHub 集成、AI 配置** 上已具备较完整的功能纵深，**单租户升级多租户**的核心工作是 **数据模型 + 全链路租户上下文 + Webhook/存储隔离**，而非重写业务逻辑。  
**监测后台** 应作为 **独立权限域与独立产品** 规划，先 **只读 + 核心指标**，再扩展运维动作与商业化联动。

建议 **优先完成「租户模型 + 隔离骨架 + 迁移」**，再并行 **注册付费与监测后台 MVP**，避免在未隔离数据前堆叠平台功能导致返工。
