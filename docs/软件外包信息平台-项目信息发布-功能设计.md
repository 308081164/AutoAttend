# 软件外包信息平台 —「项目信息发布」功能设计

> **文档类型**：功能设计（产品 / 后端 / 前端对齐）  
> **版本**：v0.1  
> **关联仓库**：`atuo_attend_backend`（`/api/*`）、`platform_monitor_frontend`（平台监测台）、主业务前端（租户侧，若单独仓库则仅约定契约）  
> **约束**：本模块为**非经营性信息中介**能力，**全程免费、平台不介入交易**；合规条款以法务定稿为准。

---

## 1. 平台定位与核心原则

| 项目 | 说明 |
|------|------|
| 平台性质 | 非经营性信息中介平台 |
| 核心原则 | 全程免费、平台不介入交易 |
| 服务对象 | 企业（发包方）+ 程序员（接包方） |
| 合规基础 | ICP 备案、用户协议与免责声明 |

## 2. 与现有系统对齐：功能可见性（监测台开关）

### 2.1 需求摘要

「项目信息发布」**不得在默认情况下对全量用户展示**。仅当**平台监测台**对**指定范围**打开开关后，对应用户或租户才能在产品内看到入口并使用相关接口。

### 2.2 设计建议（结合本仓库现状）

当前后端已具备：

- **统一 JSON 响应**：`ApiResponse<T>`，`code === 0` 表示成功（见 `org.example.atuo_attend_backend.common.ApiResponse`）。
- **平台监测台会话**：`/api/platform/*`，Cookie `aa_platform_session`，由 `PlatformAuthFilter` 保护（与 `docs/平台化-阶段0-交付物.md`、`docs/平台监测台-SaaS化扩展设计-审阅稿.md` 一致）。
- **平台级键值配置**：`SystemConfigService` + `tenant_id = 0`（`PLATFORM_SYSTEM_CONFIG_TENANT_ID`），已有先例：`GET/PUT /api/platform/settings/mail`、`GET/PUT /api/platform/settings/report-mail`（见 `PlatformSettingsController`）。

建议将「是否开放外包信息模块」落地为：

| 维度 | 说明 |
|------|------|
| **总开关** | 平台级布尔配置（`tenant_id=0`），例如键名 `marketplace.project_info.enabled`；为 `false` 时，所有对外接口返回 **403** 或业务码 **40300**，前端隐藏菜单与路由。 |
| **范围策略（二选一或组合，按迭代选择）** | **A. 按租户**：仅 `tenant_id ∈ 白名单` 的租户可使用（白名单存 JSON 或独立表）。**B. 按用户**：仅 `user_id ∈ 白名单` 的用户可使用（适合内测）。**C. 全平台开放**：总开关为真且不做白名单（上线推广期）。 |

**产品决策点（需在开发前确认）**：白名单维度（租户 / 用户 / 二者）、以及未登录游客是否可浏览公开列表（若允许，需单独「公开读」接口与脱敏规则）。

### 2.3 前端行为

- **平台监测台**：新增配置区块「项目信息发布」，可维护总开关与白名单（与现有「邮件 / 日报调度」同一套 `/api/platform/settings/*` 风格）。
- **租户侧 / C 端**：应用启动或进入模块前调用 **只读能力探测接口**（见 §5.2），根据 `visible` / `enabled` 决定展示；**禁止**仅依赖前端隐藏路由（后端必须校验）。

## 3. 用户角色体系

### 3.1 角色定义

| 角色 | 权限 | 功能范围 |
|------|------|----------|
| 游客 | 仅浏览首页（若产品允许） | 平台介绍、部分公开信息；**默认无写操作** |
| 企业用户 | 发布项目、查看程序员公开资料 | 项目管理、企业信息维护 |
| 程序员用户 | 查看项目详情、发起「申请联系」 | 项目检索、收藏、申请记录 |
| 管理员 | 内容审核、用户管理、数据统计 | 后台管理系统 |

### 3.2 注册流程（逻辑）

```
游客 → 选择角色（企业/程序员）→ 手机/邮箱验证 → 完善资料 → 审核通过 → 正式用户
```

> **说明**：本仓库现状以租户内「管理员 / 协作成员」为主；若外包平台独立站与现有租户体系合并，需在账号模型中增加 `role` 与审核状态，并与 `TenantContext` 隔离边界写清（本期文档仅约定接口与数据字段）。

## 4. 核心功能模块（业务摘要）

### 4.1 企业端：项目管理

| 功能 | 说明 |
|------|------|
| 发布项目 | 标题、描述、技术要求、预算范围、联系方式策略 |
| 项目编辑 | 未接单 / 招募中状态下可编辑（状态机见字段） |
| 项目关闭 | 手动关闭或到期自动关闭 |
| 申请记录查看 | 查看申请本项目的程序员（仅公开资料） |
| 企业资料管理 | 企业认证、历史项目展示 |

**项目字段（逻辑 JSON，可与 DB 列名 camelCase/snake_case 映射）**

```json
{
  "projectId": "唯一标识",
  "title": "项目标题",
  "description": "详细描述（支持 Markdown）",
  "techStack": ["技术栈标签"],
  "budgetRange": "预算范围（选填）",
  "duration": "预计周期",
  "location": "工作地点/远程",
  "contactType": "站内信|邮箱|电话",
  "contactValue": "具体联系方式（仅对申请通过或企业指定对象可见，需加密存储）",
  "status": "招募中|已关闭|已完成",
  "publishTime": "发布时间（ISO 8601）",
  "expireTime": "过期时间（默认 30 天）",
  "viewCount": 0,
  "applyCount": 0
}
```

### 4.2 程序员端：检索、详情、申请

- **检索**：关键词、技术栈、预算、工作方式、排序（最新 / 热度 / 预算）。
- **详情**：展示项目信息 + 企业公开信息（认证标识、历史发布数等统计字段可二期）。
- **申请流程**：程序员发起申请 → 系统记录 → 企业端通知 → 双方线下对接（平台不参与交易）。

```
程序员点击「申请联系」 → 系统记录申请 → 企业收到通知（含程序员公开资料）
→ 企业自行决定是否联系 → 双方线下沟通 → 平台流程结束
```

### 4.3 平台机制与免责声明

信息流转：平台仅提供发布、检索与意向匹配（公开资料传递），**商务洽谈、合同、资金均由用户线下完成**。

建议在以下位置强制展示免责文案（具体文案由法务审核）：

- 注册页：用户协议勾选（含免责声明）。
- 项目详情页顶部：「平台仅提供信息展示，交易风险自行承担」。
- 「申请联系」弹窗：再次提示核实资质、谨慎交易。

## 5. 接口设计（结合现有项目重新梳理）

以下约定与 **`docs/MVP接口文档.md`** 一致：

- **Base URL**：与现有网关相同（如 `https://<host>`）。
- **响应体**：`{ "code": 0, "message": "ok", "data": ... }`；失败时 `code != 0`（错误码建议沿用 `40000/40100/40300/40400/50000` 体系）。
- **鉴权**：
  - 平台监测台：`/api/platform/*`，`Cookie: aa_platform_session=...`（与现网一致）。
  - 业务用户：建议 **Bearer JWT**（若与现有 `aa_admin_session` / collab 并存，需独立 `marketplace` 或 `public` 前缀，避免过滤器冲突；实现阶段在 `AdminWebConfig` / 新增 `MarketplaceWebConfig` 中划分白名单路径）。

### 5.1 平台监测台：模块开关与范围（新增）

| 方法 | 路径 | 说明 |
|------|------|------|
| `GET` | `/api/platform/settings/project-marketplace` | 查询总开关、白名单策略与摘要（白名单可返回数量不回显全部 ID，避免响应过大） |
| `PUT` | `/api/platform/settings/project-marketplace` | 更新总开关、白名单（租户 ID 列表 / 用户 ID 列表）、可选「游客可读列表」布尔 |
| `GET` | `/api/platform/project-marketplace/audit-logs` | （可选）查询该模块相关配置变更审计，与 `PlatformOpsAuditMapper` 写入动作对齐 |

**`PUT` 请求体示例**

```json
{
  "enabled": true,
  "scope": "tenant_whitelist",
  "tenantIds": [1001, 1002],
  "allowGuestBrowseList": false
}
```

**`scope` 建议枚举**：`off` | `all` | `tenant_whitelist` | `user_whitelist`（与 `enabled` 组合关系需在实现时单一化，避免歧义）。

### 5.2 能力探测（业务前端 / 公开端）

在进入菜单或路由前调用：

| 方法 | 路径 | 鉴权 |
|------|------|------|
| `GET` | `/api/public/project-marketplace/status` | 可选登录；返回当前主体是否可使用模块 |

**响应示例**

```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "moduleEnabled": true,
    "visibleToCurrentPrincipal": true,
    "guestCanBrowse": false,
    "disclaimerVersion": "2026-04-01"
  }
}
```

当 `moduleEnabled === false` 或 `visibleToCurrentPrincipal === false` 时，前端不展示入口；对应写接口一律返回 **403** / `40300`。

### 5.3 企业侧：项目 CRUD 与申请处理

| 方法 | 路径 | 说明 |
|------|------|------|
| `POST` | `/api/marketplace/enterprise/projects` | 发布项目 |
| `GET` | `/api/marketplace/enterprise/projects` | 本企业项目列表（分页、状态筛选） |
| `GET` | `/api/marketplace/enterprise/projects/{projectId}` | 详情（含完整联系方式） |
| `PUT` | `/api/marketplace/enterprise/projects/{projectId}` | 编辑 |
| `POST` | `/api/marketplace/enterprise/projects/{projectId}/close` | 关闭 |
| `GET` | `/api/marketplace/enterprise/projects/{projectId}/applications` | 本项目收到的申请 |

> 路径前缀 `/api/marketplace` 为建议命名空间，与现有 `/api/admin`、`/api/collab`、`/api/platform` 区分；落地时可调整为 ` /api/app/marketplace` 等，但需**全局唯一**。

### 5.4 程序员侧：检索、详情、申请、收藏

| 方法 | 路径 | 说明 |
|------|------|------|
| `GET` | `/api/marketplace/projects` | 项目列表 + 检索参数（`q`、`tech`、`budget`、`location`、`sort`、`page`） |
| `GET` | `/api/marketplace/projects/{projectId}` | 详情；**默认不返回** `contactValue`，或返回掩码 |
| `POST` | `/api/marketplace/projects/{projectId}/applications` | 发起申请联系 |
| `GET` | `/api/marketplace/me/applications` | 我的申请记录 |
| `POST` | `/api/marketplace/projects/{projectId}/favorites` | 收藏 |
| `DELETE` | `/api/marketplace/projects/{projectId}/favorites` | 取消收藏 |
| `GET` | `/api/marketplace/me/favorites` | 收藏列表 |

### 5.5 管理员：审核与风控（可与现有 Admin 合并）

| 方法 | 路径 | 说明 |
|------|------|------|
| `GET` | `/api/marketplace/admin/projects/pending` | 待审核项目队列 |
| `POST` | `/api/marketplace/admin/projects/{projectId}/approve` | 审核通过 |
| `POST` | `/api/marketplace/admin/projects/{projectId}/reject` | 驳回并记录原因 |

若统一走现有租户管理员体系，也可挂载到 `/api/admin/...` 并由 `AdminAuthFilter` 鉴权；本文档保持领域前缀清晰为主，合并时迁移路径即可。

### 5.6 与用户原文档 REST 示例的映射

| 原文草案 | 本文对齐说明 |
|----------|----------------|
| `POST /api/projects` | 拆分为企业命名空间 `POST /api/marketplace/enterprise/projects`，并增加模块开关校验 |
| `GET /api/projects` | `GET /api/marketplace/projects`（程序员检索）与企业列表拆分，避免权限语义混淆 |
| `GET /api/projects/{id}` | `GET /api/marketplace/projects/{id}` + 企业侧详情接口 |
| `POST /api/applications` | `POST /api/marketplace/projects/{projectId}/applications` |
| `GET /api/applications/received` | `GET /api/marketplace/enterprise/projects/{projectId}/applications` |

## 6. 数据存储要点

### 6.1 核心表（逻辑模型，与原文一致并略作命名规范）

- `users`：`user_id`, `role`, `phone`, `email`, `status`, `create_time`
- `enterprise_profiles`：`user_id`, `company_name`, `license_no`, `verify_status`, …
- `developer_profiles`：`user_id`, `nickname`, `skills`, …
- `marketplace_projects`：`project_id`, `enterprise_user_id` 或 `tenant_id`, 项目字段与 `status`
- `marketplace_applications`：`application_id`, `project_id`, `developer_user_id`, `status`（`pending` / `ignored` / `accepted` 等，**不接单仅意向**）

### 6.2 配置存储

- 平台总开关与白名单：优先复用 **`system_config`（tenant_id=0）** 与现有 `SystemConfigService.upsert` 模式，键名在实现时固化并写入迁移说明。

### 6.3 安全要求（继承原文并补充）

- 全站 **HTTPS**；联系方式等敏感字段**加密存储**、传输最小化。
- **防 SQL 注入 / XSS**（Markdown 渲染需消毒）；接口 **限流**；关键操作 **审计日志**（平台侧已有审计写入范式可参考 `PlatformSettingsController` 内 `audit`）。

## 7. 用户协议关键条款（摘要）

1. 平台服务性质：仅提供信息发布与检索，不参与任何交易环节。  
2. 免费声明：不向发包方/接包方收取信息发布或查看费用（若未来变更需单独公示）。  
3. 交易风险：线下纠纷与损失，平台不承担责任。  
4. 信息真实性：用户应确保信息真实，平台仅形式审核。  
5. 禁止虚假、诈骗、违法违规内容。  
6. 违规账号平台有权封禁并配合监管调查。
