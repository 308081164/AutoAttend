# 软件外包信息平台 —「项目信息发布」功能设计

> **文档类型**：功能设计（产品 / 后端 / 前端对齐）  
> **版本**：v0.2  
> **关联仓库**：`atuo_attend_backend`（`/api/*`）、`platform_monitor_frontend`（平台监测台）、主业务前端（租户侧，若单独仓库则仅约定契约）  
> **约束**：本模块为**非经营性信息中介**能力，**全程免费、平台不介入交易**；合规条款以法务定稿为准。  
> **本系统账号模型（重要）**：**不区分「企业端 / 程序员端」**。业务侧仅存在**管理员账号**；其中**仅部分管理员**具备**发布**项目信息的权限，**全体管理员**均可**查看**已被发布出来的项目信息（含其他具备发布权限的管理员所发布内容）。

---

## 1. 平台定位与核心原则

| 项目 | 说明 |
|------|------|
| 平台性质 | 非经营性信息中介平台 |
| 核心原则 | 全程免费、平台不介入交易 |
| 服务对象 | 对外文案可仍面向「发包 / 接包」受众；**系统实现上**由**管理员账号**完成信息发布与浏览（见文首说明） |
| 合规基础 | ICP 备案、用户协议与免责声明 |

## 2. 与现有系统对齐：功能可见性（监测台开关）

### 2.1 需求摘要

「项目信息发布」**不得在默认情况下对全量用户展示**。仅当**平台监测台**对**指定范围**打开开关后，对应用户或租户才能在产品内看到入口并使用相关接口。

### 2.2 设计建议（结合本仓库现状）

当前后端已具备：

- **统一 JSON 响应**：`ApiResponse<T>`，`code === 0` 表示成功（见 `org.example.atuo_attend_backend.common.ApiResponse`）。
- **平台监测台会话**：`/api/platform/*`，Cookie `aa_platform_session`，由 `PlatformAuthFilter` 保护（与 `docs/平台化-阶段0-交付物.md`、`docs/平台监测台-SaaS化扩展设计-审阅稿.md` 一致）。
- **平台级键值配置**：`SystemConfigService` + `tenant_id = 0`（`PLATFORM_SYSTEM_CONFIG_TENANT_ID`），已有先例：`GET/PUT /api/platform/settings/mail`、`GET/PUT /api/platform/settings/report-mail`（见 `PlatformSettingsController`）。

建议将「是否开放外包信息模块」落地为：

| 维度 | 说明 |
|------|------|
| **总开关** | 平台级布尔配置（`tenant_id=0`），例如键名 `marketplace.project_info.enabled`；为 `false` 时，所有对外接口返回 **403** 或业务码 **40300**，前端隐藏菜单与路由。 |
| **范围策略（二选一或组合，按迭代选择）** | **A. 按租户**：仅 `tenant_id ∈ 白名单` 的租户可使用（白名单存 JSON 或独立表）。**B. 按用户**：仅 `user_id ∈ 白名单` 的用户可使用（适合内测）。**C. 全平台开放**：总开关为真且不做白名单（上线推广期）。 |

**产品决策点（需在开发前确认）**：白名单维度（租户 / 用户 / 二者）、以及未登录游客是否可浏览公开列表（若允许，需单独「公开读」接口与脱敏规则）。

### 2.3 前端行为

- **平台监测台**：新增配置区块「项目信息发布」，可维护总开关与白名单（与现有「邮件 / 日报调度」同一套 `/api/platform/settings/*` 风格）。
- **租户侧 / C 端**：应用启动或进入模块前调用 **只读能力探测接口**（见 §5.2），根据 `visible` / `enabled` 决定展示；**禁止**仅依赖前端隐藏路由（后端必须校验）。

## 3. 用户角色体系（与本系统对齐）

### 3.1 角色定义

| 角色 / 权限 | 说明 |
|-------------|------|
| **管理员（默认）** | 登录管理后台；可**浏览**本模块内已发布的项目列表与详情（含他人发布）。 |
| **具备发布权限的管理员（子集）** | 在管理员身份之上，额外拥有 **创建 / 编辑 / 关闭** 自己（或本租户策略下）的项目信息；具体判定由账号权限位、角色组或租户策略配置。 |
| **平台监测台 / 超级管理员** | 维护 §2 中的模块总开关与白名单；若存在「内容审核」流程，也可由该层或租户管理员承担（与现网 RBAC 对齐即可）。 |
| **游客 / 非管理员** | 若产品允许公开访问，仅可走只读能力（可选）；**默认以管理员登录后使用为主**。 |

**明确排除的表述**：本文档**不使用**「企业端」「程序员端」作为系统内角色或接口划分依据，避免实现时误做成两套端侧应用。

### 3.2 账号与权限（逻辑）

```
管理员登录 → （若被授权）发布/维护项目信息 → 全体管理员均可检索与查看已发布项目
```

> **说明**：本仓库现状以租户内「管理员 / 协作成员」为主；本模块只需在现有管理员模型上增加 **「是否可发布项目信息」** 权限位（或等价角色），并与 `TenantContext`、鉴权过滤器对齐。若需对外独立注册「发包/接包」用户，属于**另一套账号体系或后续迭代**，不在 v0.2 默认范围内。

## 4. 核心功能模块（业务摘要）

### 4.1 具备发布权限的管理员：发布与维护

| 功能 | 说明 |
|------|------|
| 发布项目 | 填写标题、描述、技术要求、预算范围、联系方式等并发布 |
| 项目编辑 | 在允许的状态下编辑（如「招募中」；状态机见字段） |
| 项目关闭 | 手动关闭或到期自动关闭 |
| 「我发布的」列表 | 发布者查看与管理本人发布的条目（权限边界由租户策略定义） |

**项目字段（逻辑 JSON，可与 DB 列名 camelCase/snake_case 映射）**

```json
{
  "projectId": "唯一标识",
  "publisherAdminUserId": "发布者管理员用户 ID",
  "title": "项目标题",
  "description": "详细描述（支持 Markdown）",
  "techStack": ["技术栈标签"],
  "budgetRange": "预算范围（选填）",
  "duration": "预计周期",
  "location": "工作地点/远程",
  "contactType": "站内信|邮箱|电话",
  "contactValue": "具体联系方式（默认：全体可查管理员可见；存储需加密）",
  "status": "招募中|已关闭|已完成",
  "publishTime": "发布时间（ISO 8601）",
  "expireTime": "过期时间（默认 30 天）",
  "viewCount": 0
}
```

> **说明**：`applyCount` 等字段仅在**未来**若引入「意向登记 / 站外对接记录」等功能时启用；**当前系统模型下不作为默认必备字段**，避免开发误以为存在「接包方账号体系」。

### 4.2 全体管理员：浏览与检索

- **检索**：关键词、技术栈、预算、工作方式、排序（最新 / 热度 / 预算等）——**所有管理员**使用同一套列表与详情能力。
- **详情**：展示项目信息与发布者**可公开的展示名**（如昵称、组织简称，字段由实现期定）；**不**按「查看方角色」区分接口或字段。
- **线下对接**：若业务上仍需记录「谁对哪条信息感兴趣」，可作为**可选增强**（站内通知、备注），**非**本 v0.2 文档的默认双端流程。

### 4.3 平台机制与免责声明

信息流转：平台仅提供信息发布与检索，**商务洽谈、合同、资金均由用户线下完成**。

建议在以下位置强制展示免责文案（具体文案由法务审核）：

- 注册 / 登录相关页：用户协议勾选（含免责声明）（若对外访客开放）。
- 项目详情页顶部：「平台仅提供信息展示，交易风险自行承担」。
- 发布前确认、或涉及联系方式展示时的二次提示：核实资质、谨慎交易。

## 5. 接口设计（结合现有项目重新梳理）

以下约定与 **`docs/MVP接口文档.md`** 一致：

- **Base URL**：与现有网关相同（如 `https://<host>`）。
- **响应体**：`{ "code": 0, "message": "ok", "data": ... }`；失败时 `code != 0`（错误码建议沿用 `40000/40100/40300/40400/50000` 体系）。
- **鉴权**：
  - 平台监测台：`/api/platform/*`，`Cookie: aa_platform_session=...`（与现网一致）。
  - 管理后台用户：建议 **Bearer JWT**（与现有 `aa_admin_session` 等一致）；**写接口**须校验 **「发布项目信息」** 权限，**读接口**仅需**管理员登录**（与上文「全体管理员可浏览」一致）。若与 `collab` 并存，可用独立前缀（如 `/api/marketplace`）并在 `AdminWebConfig` 或新增配置中划分白名单路径。

### 5.1 平台监测台：模块开关与范围（新增）

| 方法 | 路径 | 说明 |
|------|------|------|
| `GET` | `/api/platform/settings/project-marketplace` | 查询总开关、白名单策略与摘要（白名单可返回数量不回显全部 ID，避免响应过大） |
| `PUT` | `/api/platform/settings/project-marketplace` | 更新总开关、白名单（租户 ID 列表 / 用户 ID 列表）、可选「游客可读列表」布尔 |
| `GET` | `/api/platform/project-marketplace/audit-logs` | （可选）查询该模块相关配置变更审计，与 `PlatformOpsAuditMapper` 写入动作对齐 |

**`PUT` 请求体示例**

```json
{
  "enabled": true,
  "scope": "tenant_whitelist",
  "tenantIds": [1001, 1002],
  "allowGuestBrowseList": false
}
```

**`scope` 建议枚举**：`off` | `all` | `tenant_whitelist` | `user_whitelist`（与 `enabled` 组合关系需在实现时单一化，避免歧义）。

### 5.2 能力探测（业务前端 / 公开端）

在进入菜单或路由前调用：

| 方法 | 路径 | 鉴权 |
|------|------|------|
| `GET` | `/api/public/project-marketplace/status` | 可选登录；返回当前主体是否可使用模块 |

**响应示例**

```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "moduleEnabled": true,
    "visibleToCurrentPrincipal": true,
    "guestCanBrowse": false,
    "disclaimerVersion": "2026-04-01"
  }
}
```

当 `moduleEnabled === false` 或 `visibleToCurrentPrincipal === false` 时，前端不展示入口；对应写接口一律返回 **403** / `40300`。

### 5.3 管理后台：项目信息（统一资源，读写分流）

**读能力（全体已登录管理员）** — 可查看**任意已发布**项目（含其他发布者所发）。

| 方法 | 路径 | 说明 |
|------|------|------|
| `GET` | `/api/marketplace/projects` | 项目列表 + 检索参数（`q`、`tech`、`budget`、`location`、`sort`、`page`） |
| `GET` | `/api/marketplace/projects/{projectId}` | 详情（含联系方式等字段；若策略要求可对部分字段脱敏，但**不因「查看者非发布者」而默认隐藏**，与本节权限模型一致） |

**写能力（仅具备发布权限的管理员）**

| 方法 | 路径 | 说明 |
|------|------|------|
| `POST` | `/api/marketplace/projects` | 发布项目 |
| `PUT` | `/api/marketplace/projects/{projectId}` | 编辑；**仅发布者或租户策略允许的超级管理员**（实现时统一鉴权） |
| `POST` | `/api/marketplace/projects/{projectId}/close` | 关闭 |
| `GET` | `/api/marketplace/me/published-projects` | （可选）「我发布的」列表，便于发布者管理 |

无发布权限的管理员调用写接口返回 **403** / `40300`。

> 路径前缀 `/api/marketplace` 为建议命名空间，与现有 `/api/admin`、`/api/collab`、`/api/platform` 区分；落地时可挂载在 `/api/admin/marketplace/...` 等，但需**全局唯一**、且过滤器与权限位约定清晰。

### 5.4 可选增强（非默认）

若产品后续需要「意向登记 / 收藏 / 站内通知」等，再**单独**增加接口（例如 `.../interest`、`.../favorites`），**不要**与「企业 / 程序员」角色绑定；本节 v0.2 **不**作为必做接口。

### 5.5 内容审核与风控（可与现有 Admin 合并）

| 方法 | 路径 | 说明 |
|------|------|------|
| `GET` | `/api/marketplace/admin/projects/pending` | 待审核项目队列 |
| `POST` | `/api/marketplace/admin/projects/{projectId}/approve` | 审核通过 |
| `POST` | `/api/marketplace/admin/projects/{projectId}/reject` | 驳回并记录原因 |

若统一走现有租户管理员体系，建议挂载到 `/api/admin/...` 并由 `AdminAuthFilter` 鉴权；本文档保持资源名一致，仅路径前缀随实现调整。

### 5.6 与早期草案 REST 示例的映射（防歧义）

| 早期草案 | v0.2 对齐说明 |
|----------|----------------|
| `POST /api/projects` | `POST /api/marketplace/projects`，且**仅**具备发布权限的管理员可调用，并走模块开关校验 |
| `GET /api/projects` | `GET /api/marketplace/projects`，**全体管理员**可调用 |
| `GET /api/projects/{id}` | `GET /api/marketplace/projects/{id}`，**全体管理员**可调用；**不再**拆成「企业详情 / 程序员详情」两套 |
| `POST /api/applications` 等 | **非默认**；若要做意向类能力，见 §5.4，且不与「程序员端」挂钩 |

## 6. 数据存储要点

### 6.1 核心表（逻辑模型）

- `admin_users`（或复用现有用户表）：`user_id`, `tenant_id`, `can_publish_project_info`（布尔或权限组 ID）, …
- `marketplace_projects`：`project_id`, `tenant_id`, `publisher_user_id`, 项目字段与 `status`
- **可选**：`marketplace_*_audit` 审核队列表、操作审计表（与现网风格一致即可）

**已弃用本模块默认范围的表（除非未来单独做对外双端账号）**：`enterprise_profiles` / `developer_profiles` 等**不作为** v0.2 实现前提，避免开发误建「企业 / 程序员」两套资料体系。

### 6.2 配置存储

- 平台总开关与白名单：优先复用 **`system_config`（tenant_id=0）** 与现有 `SystemConfigService.upsert` 模式，键名在实现时固化并写入迁移说明。

### 6.3 安全要求（继承原文并补充）

- 全站 **HTTPS**；联系方式等敏感字段**加密存储**、传输最小化。
- **防 SQL 注入 / XSS**（Markdown 渲染需消毒）；接口 **限流**；关键操作 **审计日志**（平台侧已有审计写入范式可参考 `PlatformSettingsController` 内 `audit`）。

## 7. 用户协议关键条款（摘要）

1. 平台服务性质：仅提供信息发布与检索，不参与任何交易环节。  
2. 免费声明：不向信息发布方或浏览方收取费用（对外话术仍可为「发包/接包」；若未来变更需单独公示）。  
3. 交易风险：线下纠纷与损失，平台不承担责任。  
4. 信息真实性：用户应确保信息真实，平台仅形式审核。  
5. 禁止虚假、诈骗、违法违规内容。  
6. 违规账号平台有权封禁并配合监管调查。
