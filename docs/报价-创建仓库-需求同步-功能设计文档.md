# 报价 → 创建仓库 → 需求同步 → Webhook 接入（MVP）— 功能设计文档

## 1. 文档说明

- **功能名称**：报价页创建 GitHub 仓库 + 将需求写入多维表 + 自动配置 Webhook + 需求清单写入仓库
- **版本**：v0.2（MVP + 双表用途、报价回写 `link_table_id`、反向从多维表带入报价）
- **目标**：在 AutoAttend 内打通「**报价—需求明确—项目创建**」链路，实现从报价项目的一键落地：
  - 在 GitHub **个人账号**下创建远程仓库（public/private、简介、可选初始化 README）
  - 将报价系统中收集到的「功能模块/功能点」同步为该项目的「项目协作-多维表」记录
  - 自动为新仓库配置 Webhook 接入当前 AutoAttend（push 即触发提交记录、Diff 拉取、成员同步）
  - 将需求清单生成一个 `.md` 文件并写入仓库（便于研发直接在仓库查看/评审）

> **实现约束（符合现状）**：系统已具备 GitHub Token 配置与 GitHub API 调用能力（用于拉 Diff/读仓库信息），且协作模块项目的主键 `repoId` 采用 `repoFullName`（如 `owner/repo`）。本功能在此基础上新增“创建仓库/写文件/建 webhook”的 API 调用与报价到协作的数据映射。

## 2. 现状与可复用能力（来自代码/文档）

### 2.1 GitHub Token 配置（已实现）

- 管理后台接口：`GET/PUT /api/admin/config/github`
- 存储位置：`SystemConfigService`（DB 优先，环境变量可兜底）
- 可复用点：本功能直接复用同一 Token（**最简单认证方式**：PAT）

对应代码：
- `atuo_attend_backend/src/main/java/org/example/atuo_attend_backend/admin/AdminConfigController.java`
- `atuo_attend_backend/src/main/java/org/example/atuo_attend_backend/config/SystemConfigService.java`

### 2.2 协作项目与多维表自动创建（已实现）

- Webhook push 或「创建仓库」provision 时会调用：`CollabSyncService.ensureProjectAndTable(repoFullName)`
- 该方法会创建：
  - `biz_project.repo_id = repoFullName`
  - `biz_project_table` 一条，**用途 `purpose = issue_tracking`**，表名展示为 **「项目调整」**（侧重售后阶段的 Bug、需求变更、验收问题；默认列含 **问题描述 / 归属模块 / 图像展示 / 负责人 / 重要程度 / 当前状态 / 验收结果 / 创建时间** 等，与历史「任务表」一致）
  - **「待开发功能清单」表（`feature_backlog`）不随仓库自动创建**，由报价页 **「导入多维表格」** 或后端 `ensureFeatureBacklogTable` **按需创建**，列侧重 **功能名称、功能描述、归属模块、开发进度、图像、负责人、重要程度、预计完成、创建人、创建时间**，供开发阶段使用；与 issue 表 **共用留言、附件、记录模型**，仅 `table_id` 不同。

对应代码：
- `atuo_attend_backend/src/main/java/org/example/atuo_attend_backend/collab/service/CollabSyncService.java`（含 `ensureFeatureBacklogTable`）
- `atuo_attend_backend/src/main/resources/db/schema_collab_mysql.sql`（`biz_project_table.purpose`，唯一约束 `uk_project_purpose`）
- 协作前端：`CollabTableView.vue` 左侧栏切换 `?purpose=issue_tracking` / `feature_backlog`

### 2.2.1 报价项目与协作项目的关联（已实现）

- Provision 成功并确保协作项目与默认表存在后，将 **`biz_project.id` 回写至报价表 `link_table_id`**，使报价与多维表 **不依赖用户再手工绑定仓库名** 即可调用 `link-table-requirements` 与 `import-feature-table`。
- 与 `docs/需求-报价-合同半自动化产出-功能设计文档.md` 中「从多维表带入需求」共用同一关联字段。

### 2.3 Webhook 接入点（已实现）

- 接收入口：`POST /api/webhooks/github`
- 当前处理：仅处理 `push`；写入 commit；拉 diff；同步协作用户与项目成员

对应代码：
- `atuo_attend_backend/src/main/java/org/example/atuo_attend_backend/webhook/GithubWebhookController.java`

> 注意：当前实现未看到对 `X-Hub-Signature-256` 的严格验签逻辑；MVP 可先配置 Webhook 但不强制验签，后续版本补齐验签与幂等。

### 2.4 报价项目结构化需求（已实现）

- 报价项目保存：模块 `modules[]`，模块下功能点 `items[]`（complexity、quantity）

对应代码/页面：
- `auto_attend_forntend/src/views/QuoteProjectView.vue`
- `atuo_attend_backend/src/main/java/org/example/atuo_attend_backend/quote/service/QuoteService.java`
- `docs/需求-报价-合同半自动化产出-功能设计文档.md`

## 3. MVP 用户故事

### 3.1 管理员视角

在「报价项目」页（`/quote/:id`）完成需求拆解与确认后：

1. 点击 **「创建仓库」**
2. 选择仓库可见性（public/private），填写仓库名（可自动建议），可选填写简介
3. 勾选：
   - 初始化 README（可选）
   - 同步需求清单到多维表（默认开启）
   - 将需求清单写入仓库 `docs/需求清单.md`（默认开启）
   - 自动配置 Webhook（默认开启）
4. 提交后系统自动完成上述操作，并回显：
   - 仓库地址、Webhook 状态、多维表同步结果（创建了多少条记录）
5. 管理员可点“进入项目协作”直接查看多维表中的需求任务

## 4. 端到端流程设计（MVP）

### 4.1 主流程（推荐顺序）

1. **创建 GitHub 仓库（个人）**
2. **写入需求清单 MD（可选）**
3. **在协作模块创建项目与表（如尚不存在）**
   - 调用 `CollabSyncService.ensureProjectAndTable(repoFullName)` 以确保协作项目存在
4. **将需求写入多维表记录（可选）**
5. **创建 Webhook（可选）**
   - hook url 指向 AutoAttend 的 `/api/webhooks/github`

### 4.2 异常与回滚策略（MVP 可做“软回滚”）

由于涉及跨系统（GitHub + DB），严格事务回滚成本高，MVP 采用：

- **阶段性落库状态**：每一步落库 `status` 与 `errorMessage`，允许管理员重试失败步骤
- **不做自动删除仓库回滚**（避免误删已使用仓库）；失败后给出「手动删除仓库」提示与链接
- **幂等策略**：
  - 若仓库已存在：提示并允许绑定到现有仓库（后续版本）；
  - 若 webhook 已存在：检测相同 url 则复用或提示；
  - 多维表写入：支持“只写入一次”的标记（避免重复导入）

## 5. 前端交互设计（管理员端）

### 5.1 入口位置

- 页面：`QuoteProjectView.vue`（报价项目编辑页）
- 在 “项目基础/功能模块/报价计算”模块附近新增一个 **「项目创建」卡片**，包含：
  - **创建仓库**按钮
  - 当前绑定信息展示（如果已创建）：repo full name、repo url、webhook 状态、多维表同步状态
  - 快捷入口：进入项目协作（跳转 `/collab/projects/:projectId/table` 或项目列表）

### 5.2 弹窗（CreateRepoModal）

字段：

- **仓库归属**：个人（固定，MVP 不支持组织）
- **仓库名**：默认建议 `autoattend-quote-{quoteProjectId}` 或 `quote-{nameSlug}`；支持编辑
- **可见性**：public/private
- **简介**：默认取报价项目名 + 简要描述
- **初始化 README**：默认开
- **写入需求清单 md**：默认开（路径建议 `docs/需求清单.md`）
- **同步到多维表**：默认开
- **创建 webhook**：默认开
- **Webhook Secret**：MVP 可自动生成并隐藏；后续可支持自定义

按钮：
- **开始创建**（loading + 分步进度）
- **关闭**

进度展示（分步）：
- 创建仓库 → 写入 MD → 创建协作项目 → 同步多维表 → 配置 webhook

## 6. 后端接口设计（管理员端）

### 6.1 创建仓库并联动（单一入口，服务端分步执行）

> 推荐一个接口完成全链路，服务端内部按步骤执行并返回每步结果，便于前端展示进度与可重试。

- **POST** `/api/admin/quote/projects/{id}/provision`

请求体（MVP）：

- `repoName`：string（必填）
- `private`：boolean（必填）
- `description`：string（可选）
- `autoInit`：boolean（可选，默认 true）
- `syncMd`：boolean（默认 true）
- `syncCollabTable`：boolean（默认 true）
- `createWebhook`：boolean（默认 true）

响应体（建议）：

- `repoFullName`、`repoHtmlUrl`
- `steps[]`：每步 `{ key, ok, message, data? }`
  - `createRepo`
  - `writeRequirementMd`
  - `ensureCollabProject`
  - `syncCollabRecords`
  - `createWebhook`

### 6.2 可选：分步重试接口（后续）

- `POST /api/admin/quote/projects/{id}/provision/retry?step=createWebhook`

MVP 可先不做，先用单接口 + 重复点击（服务端做幂等判断）。

## 7. GitHub API 调用清单（MVP：PAT 最简认证）

### 7.1 认证方式（最简单）

- 复用系统已存在的 GitHub Token（PAT），从 `SystemConfigService.getGitHubToken()` 读取
- 权限要求：
  - 创建仓库（个人）：`repo`（私有）或 `public_repo`（公有）
  - 创建 webhook：同样需要对仓库的管理权限（PAT 一般包含）

### 7.2 需要调用的 API（REST）

1) 创建个人仓库  
- `POST https://api.github.com/user/repos`
- body：`name, private, description, auto_init`

2) 写入需求清单文件  
- `PUT https://api.github.com/repos/{owner}/{repo}/contents/docs/需求清单.md`
- body：`message, content(base64)`

3) 创建 webhook  
- `POST https://api.github.com/repos/{owner}/{repo}/hooks`
- body 示例：
  - `name: "web"`
  - `active: true`
  - `events: ["push"]`
  - `config: { url, content_type:"json", secret, insecure_ssl:"0" }`

> 备注：MVP 可先仅配置 `push`；后续可扩展 `pull_request` 等。

## 8. 报价需求 → 多维表记录映射设计

### 8.1 目标

将报价项目中的 `modules[].items[]` 转成协作多维表 `biz_record` + `biz_record_field` 行数据，便于研发以任务表方式推进。

### 8.2 建议映射规则（MVP）

多维表默认列（见 `CollabSyncService.createDefaultColumnsAndOptionGroups`）：

- **问题描述**：写入：`【模块】功能点名称` + 辅助描述（复杂度/数量/估算人天）
- **归属模块**：写入模块名（若该项目的“归属模块”选项组为空，MVP 可先直接写 value_text，不强制校验选项；后续版本完善选项组同步）
- **重要程度**：MVP 默认空（或统一设为“下一阶段待办”）
- **当前状态**：默认“已创建”
- **验收结果**：默认“未验收”
- **创建人**：写入“报价系统/管理员”（文本列）
- **创建时间**：写入当前时间

一条功能点一条记录：

- record.title（无该列时）使用“问题描述”承载
- quantity > 1：在描述中体现，MVP 不拆多条记录（后续可按 quantity 拆分）

### 8.3 去重策略（MVP）

为避免重复同步：

- 在报价项目落库一个 `provisionSyncVersion` 或 `syncedAt`（见数据模型建议），若已同步过则默认不重复导入
- 或者在协作记录里写入一段隐藏标识（MVP 暂不实现隐藏字段，建议走前者）

### 8.4 反向：从协作「项目调整」表带入报价（MVP，与 8.2 方向相反）

- **数据源**：仅 **`issue_tracking`** 表；读取列 **「问题描述」「归属模块」** 作为摘要行。
- **接口**：`GET/POST .../link-table-requirements`（见报价设计文档）；**不**从 `feature_backlog` 表自动导入报价（功能清单表用于研发排期，与报价模块的映射规则可后续单独定义）。
- **与 provision 关系**：无仓库时若仅有本地协作项目 ID，仍可通过 `link_table_id` 拉取；创建仓库后 `link_table_id` 与 `repoFullName` 指向同一协作项目。

## 9. 数据模型变更建议（MVP）

为把“报价项目”与“仓库/协作项目/同步状态”关联起来，建议给 `biz_quote_project` 增加字段（增量 migration）：

- `github_repo_full_name`（如 `owner/repo`）
- `github_repo_html_url`
- `github_webhook_id`（可选）
- `github_webhook_secret`（可选，若未来要验签）
- `provision_status`（draft/provisioning/done/failed）
- `provision_last_error`
- `provision_synced_to_collab`（bool）/ `provision_synced_at`

> 说明：协作项目表 `biz_project` 已使用 `repo_id` 作为关联键（`repoFullName`），因此无需新增跨表 join，只需要在报价项目里保存 `github_repo_full_name` 即可定位到协作项目。  
> **补充（已实现）**：另增 **`link_table_id` → `biz_project.id`**，用于报价与多维表 API 的稳定关联（provision 后自动写入）。

## 10. 权限与安全（MVP）

- **权限**：仅管理员可执行“创建仓库/配置 webhook/同步需求”
- **Token**：
  - PAT 存储于系统配置（已存在）
  - 输出给前端仅展示 masked，不下发明文
- **Webhook Secret**：
  - 即便当前未验签，也建议生成并存储，后续补验签无需迁移历史仓库
- **审计**：
  - 记录谁触发了 provision（管理员 userId / adminName）
  - 记录每一步 GitHub API 的响应摘要（不含 token）

## 11. MVP 范围与后续扩展

### 11.1 MVP（本期）

- 个人仓库创建（public/private）
- 可选 auto_init README
- 写入 `docs/需求清单.md`
- 确保协作项目与表存在（复用 `ensureProjectAndTable`）
- 同步需求到多维表（按功能点生成记录）
- 创建 webhook（push）
- 状态回显 + 可重试（至少提示“哪一步失败”）

### 11.2 后续版本（规划）

- 支持创建到组织（org）与团队权限分配
- webhook 验签与 delivery 幂等
- 需求与协作双向同步（多维表更新可回写到仓库 issue/PRD 等）
- quantity 拆分多条任务、按负责人/模块智能分配
- 需求清单 MD 模板化（带目录、里程碑、验收标准、DoD）

## 12. 实现建议（落地策略）

1) 先实现后端 `provision` 单接口，内部按步骤执行并返回 step 结果  
2) 前端先做“按钮 + 弹窗 + 进度展示 + 回显 repo url”  
3) 多维表同步先写最小字段集合（问题描述/归属模块/状态/创建时间）  
4) webhook 最后做（不影响本地协作与文档沉淀）
