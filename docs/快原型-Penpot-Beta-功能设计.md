## 1. 背景与目标

### 1.1 问题

现有「快原型」在 **受约束 Spec** 与 **有限交互类型** 下，已能控制安全与一致性，但用户感知上易出现 **产出质量波动**（结构单调、过校验但观感一般、迭代方向不稳定等）。需在 **不推翻现有能力** 的前提下，验证 **Penpot 作为设计宿主** 是否能带来 **更稳定的版式结构、可导出设计源文件、可预览画布**。

### 1.2 Beta 目标（可验收）

1. **双轨并存**：用户在同一快原型项目内，可区分 **经典模式（Spec/Mockup）** 与 **Penpot Beta（设计文件）**（具体入口可为 Tab、开关或独立子页，实现阶段再定）。  
2. **生成链路**：自然语言 / 现有 prompt → **沿用平台 LLM** → **结构化计划** → **Penpot REST/Plugin 能力写入** → **预览/导出**。  
3. **部署**：Penpot **与系统一并 Docker 部署**，服务发现走 Compose 网络；运维只需拉起栈，**无需为 Penpot 单独暴露一套新的「用户侧 API Key 配置」**。  
4. **失败可解释**：任务状态、重试次数、校验错误对用户可见，避免「黑盒失败」。

### 1.3 非目标（Beta 明确不做）

- 替换现有 Spec 渲染与导出为主唯一路径（避免一刀切迁移风险）。  
- 用户为 Penpot Beta **单独配置** 一套 LLM API Key（与需求 **「仍使用现有系统基座」** 一致）。  
- 完整设计系统管理平台（组件库 CRUD、多主题引擎）—— **仅最小化** 引用 Penpot 文件内变量/组件（若需要）。

## 2. 与现有系统架构的衔接

### 2.1 现有模块（仓库现状）

- **前端**：`auto_attend_forntend`，路由 `/prototype`、`PrototypeProjectView.vue`，调用 `/admin/ui-prototype/projects/...`。  
- **后端**：`atuo_attend_backend` 中 `UiPrototypeService`、异步 Job（避免 LLM 超时）、Spec/Mockup 校验器。  

### 2.2 Beta 增量（逻辑）

在 **不改变既有鉴权与租户模型** 的前提下，增加：

1. **Penpot 适配层**（独立 Java `@Service` 或独立进程：由实现阶段权衡；**推荐首期为后端内嵌 HTTP 客户端 + 异步任务**，降低运维面）。  
2. **配置**：`PENPOT_BASE_URL`（如 `http://penpot-frontend:9001`）、服务账号 Token（由部署环境注入，**非终端用户配置**）。  
3. **数据**：快原型项目与 Penpot **file / project** 的映射表（或扩展现有项目表字段），以及异步任务状态扩展。

### 2.3 架构示意（Beta）

```
用户 → 现有前端「快原型」→ 现有 Backend API
                              │
                              ├─► 既有：Spec / Mockup 生成（不变）
                              │
                              └─► 新增：Penpot Beta 任务
                                      → LLM（平台基座）
                                      → Penpot Adapter（HTTP / 官方 API）
                                      → 返回 fileId + 预览 URL / 导出
```

## 3. 核心能力（Beta 范围）

### 3.1 用户可见能力

| 能力 | 说明 | Beta |
|------|------|------|
| 从需求生成 Penpot 文件 | 单页或多 Frame 的初稿；允许后续迭代指令 | ✅ 优先单页 |
| 在应用内打开/嵌入预览 | iframe 或跳转至 Penpot 前端（同源或受信域下配置） | ✅ |
| 导出 | SVG/PNG 等以 Penpot 导出能力为准；**不要求** Beta 内完成「React 代码全栈导出」 | 分阶段 |
| 与 Spec 互转 | 可选：Spec → 指导 Penpot 布局；或 Penpot 截图 → 辅助说明 | ❌ P1 |

### 3.2 稳定性策略（与原文档对齐并落地）

1. **两阶段生成**：Planning（JSON 计划：页面/Frame/组件引用）→ Execution（仅执行允许的 Penpot 操作集合）。  
2. **校验**：结构 Schema 校验 + **可选** 截图/缩略图对比（成本允许时）。  
3. **重试**：固定上限（如 3 次），失败返回 **可解析错误码**（网络、Penpot 4xx、LLM 空输出等）。  
4. **与模板思想结合**：Planning 阶段可注入 **默认布局模板 ID**（与《快原型功能优化文档》中「模板壳」概念对齐），减少版式随机性。

## 4. API 与数据（原则）

Beta 阶段 **优先复用** 现有 `/admin/ui-prototype/projects/{id}/...` 风格，新增例如：

- `POST .../penpot/jobs`：提交生成任务（body：prompt、可选 templateId）。  
- `GET .../penpot/jobs/{jobId}`：轮询状态。  
- `GET .../penpot/link`：返回只读预览或跳转信息（**不**在响应中泄露服务端 Penpot 管理员密码）。

具体路径与 DTO 在 **接口设计评审** 时定稿；数据表字段需包含：`tenant_id`、`project_id`、`penpot_file_id`、`status`、`last_error`、`retry_count`。

## 5. AI 与密钥

### 5.1 平台基座

- **所有 LLM 调用** 走现有后端已接好的 **统一 AI 配置**（与报价、日报等模块一致）：**终端用户无需也不应** 为 Penpot Beta 单独配置 Key。  
- 若需 **按租户** 区分模型或额度，沿用现有 **租户级 AI 策略**（若仓库中已有表/配置则复用；若无则 Beta 先用系统级默认模型）。

### 5.2 Prompt 资产

保留原文档中 **分角色 Prompt**（planner / executor / validator）思想，代码库中以 **资源文件** 或 **配置中心** 管理，支持 **版本号** 与回滚。

## 7. 部署与端口（Docker）

### 7.1 Penpot 与端口

- **目标**：将 **官方 Penpot Docker 编排**（或经团队裁剪的 Compose）作为 **独立 service** 加入仓库根目录或 `deploy/` 下 Compose 文件。  
- **端口**：**使用 Penpot 官方文档中的默认 HTTP 端口**（通常为 **9001**；以选用镜像/Compose 模板为准）。  
- **冲突说明**：本仓库现有 `docker-compose.yml` 中 MinIO 控制台曾映射主机 **9002**，**不应** 与 Penpot 默认端口在 **容器内** 冲突；主机侧端口映射需在实施时 **显式检查** 宿主机占用情况。

### 7.2 网络

- `backend` 通过服务名访问 `http://penpot-frontend:<默认端口>`（或官方 Compose 中的前端服务名）。  
- 可选：仅 `backend` 访问 Penpot API；浏览器 iframe 通过 **反向代理同源** 或 **配置 CORS/可信域名**。

### 7.3 密钥与账号

- Penpot **团队/服务账号** 由部署时创建，Token 存 **环境变量/密钥管理**，不写死在前端。

## 8. 实现阶段规划（无日历估算）

下列顺序表示 **依赖关系**，不代表工期。

### 阶段 A — 基础设施

1. 在 Compose 中 **拉起 Penpot 依赖栈**（含其 PostgreSQL/Redis 等，按官方模板），并文档化 **环境变量清单**。  
2. 验证从 `backend` 容器 **curl** Penpot 健康与登录/API 可用性。  
3. 数据库 **迁移**：项目与 Penpot 文件映射、异步任务表（或与现有 job 表扩展）。

### 阶段 B — 适配层与最小闭环

1. 实现 **Penpot Adapter**：创建文件、创建 Page/Board、基础形状/文本（以官方 API 能力为准）。  
2. 接入 **平台 LLM**：实现 Planning → 指令执行 → 持久化 `penpot_file_id`。  
3. **API + 轮询**：与前端打通一条 **最小生成链路**。

### 阶段 C — 产品化与稳定性

1. 错误码、重试、进度百分比。  
2. 预览入口（iframe/新标签）与权限校验。  
3. 与《快原型功能优化文档》对齐 **模板 ID**、**默认通用模板** 在 Penpot 路径上的映射（若采用）。  
4. 导出（SVG/PNG）与 **经典 Spec 导出** 并在 UI 上 **并列说明差异**。

### 阶段 D — 运营与清理

1. 特性开关（按租户/全局）。  
2. 监控与告警（Penpot 宕机降级文案）。  
3. Beta 反馈收集后，再评估 **是否** 将 Penpot 路径提升为默认。

## 9. 风险与缓解

| 风险 | 缓解 |
|------|------|
| Penpot API 与插件能力与假设不一致 | Adapter 单测 + 对官方版本 **锁版本号** |
| 资源占用高 | 仅内网部署、限制并发任务数、可选独立节点 |
| 双轨维护成本 | Beta 功能开关、文档明确边界 |
| LLM 仍不稳定 | 两阶段 + 模板 + 严格 JSON Schema + 重试上限 |
