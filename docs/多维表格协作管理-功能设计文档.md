# 多维表格协作管理（类飞书多维表）— 功能设计文档

## 1. 文档说明

- **功能名称**：多维表格协作管理（项目需求/功能在线协作、任务看板）
- **版本**：v0.5
- **目标**：在现有 AutoAttend 系统中集成一套**类飞书多维表格**的协作能力，支持按项目管理需求/功能、任务清单与工作看板；用户通过**系统自有账号登录**（账号来源于项目推送中的提交人信息，默认密码 123456，支持用户与管理员重置密码）；每个协作项目**至少绑定一张多维表**，并可按 **用途（purpose）** 扩展第二张表（**项目调整** vs **待开发功能清单**）；支持**附件上传**（MinIO，与现有 Docker 部署一起编排）、**记录级讨论/留言**、**负责人多选**、**快捷选填选项**等。
- **依赖**：现有 AutoAttend 技术栈（Vue 前端、Spring Boot 后端、MySQL）；MinIO 与现有 docker-compose 一同编排；**不依赖 GitHub OAuth**，使用系统内置账号体系。
- **参考材料**：`docs/多维表格辅助材料/邻聚惠任务清单_数据表.csv` 作为任务清单列结构参考（该 CSV 无讨论功能，本系统在每条记录下增加「讨论/留言」能力）。

---

## 2. 功能目标与范围

### 2.1 目标

- **统一入口**：与现有系统同域，通过右上角「切换项目」与项目页顶部的**「项目协作」**入口进入；首页点击「项目协作」时选择有权限的项目进入其表格，项目详情页点击「项目协作」时直接进入当前项目的多维表。
- **按项目维度**：每个项目对应一个协作空间；**随仓库首次同步自动创建**一张默认多维表（**用途 `issue_tracking`，界面称「项目调整」**），侧重售后阶段的 **Bug / 需求变更 / 验收问题**。同一项目可 **按需** 再建一张 **「待开发功能清单」**（**`feature_backlog`**），侧重研发阶段的 **功能描述与开发进度**；两张表 **共用留言、附件、权限与成员**，仅 **表结构（列）与业务语义** 不同。**项目均根据仓库自动创建**（系统识别到几个被记录的仓库就有几个对应项目），**不需要管理员手动创建项目**。
- **自有账号与权限**：超级管理员可查看所有项目并可创建**子管理员**、为成员**分配指定项目的管理权限**；**子管理员仅可查看与管理被分配了管理权限的项目**；普通成员仅能查看**自己参与的项目**（由仓库作者自动同步）及其表格。
- **协作能力**：每条记录支持负责人（多选）、状态/优先级等快捷选项、附件、**讨论留言**（参考 CSV 无此能力，本系统新增）。

### 2.2 范围（本期）

| 在范围内 | 不在范围内（可后续迭代） |
|----------|---------------------------|
| 系统自有账号登录（来源于推送 commit 作者，默认密码 123456，可重置） | GitHub OAuth、飞书/企业微信登录 |
| 项目与多维表绑定、自动创建默认「项目调整」表；可选第二张「待开发功能清单」 | 多视图（甘特图、日历视图） |
| 表格字段：文本、多选负责人、单选/多选选项、附件、日期等 | 公式、关联表、自动化 |
| 附件上传与下载（MinIO，Docker 编排） | 大文件分片、在线预览 |
| 每条记录的讨论/留言与查看记录 | @提醒、富文本、表情 |
| 负责人多选、快捷选填选项 | 自定义字段类型、拖拽排序 |
| 项目协作入口（首页选项目 / 项目详情页直进）、权限（管理员全量、成员仅参与项目） | 独立移动端 App |
| 子管理员创建、项目级管理权限分配 | 更细粒度 RBAC |
| **CSV + AI 智能导入**：上传 CSV 后经 **DeepSeek** 按当前表结构分片清洗为任务草稿，确认后写入（**不**直接按 CSV 列硬导入） | 任意格式 Excel 无结构导入 |
| **仪表盘视图**：多维表内一键切换，图形化展示任务状态、周新增、周解决、重要程度分布 | 词云算法与问题平均解决时长（本期预留占位，后续版本实现） |

## 3. 用户角色与场景

| 角色 | 操作 |
|------|------|
| **超级管理员** | 编辑/归档项目（项目由仓库自动创建，不可手动新建）；创建**子管理员**账号；为成员**分配指定项目的管理权限**；查看全部项目与多维表；管理表格字段与选项组；重置任意用户密码。 |
| **子管理员** | **仅可查看与管理被分配了管理权限的项目**；在被分配的项目内权限同管理员（管理成员、管理表格与选项等）；不可创建子管理员或分配项目权限（仅超级管理员）。 |
| **项目管理员**（被分配了某项目管理权限的普通成员） | 对该项目及其多维表拥有管理权限；可添加/移除项目成员、编辑表格结构（可选）。 |
| **普通成员** | 仅能查看**自己参与的项目**及其多维表；在有权项目内新建/编辑/删除记录、上传附件、留言、选择负责人与选项。 |
| **系统** | 从 commit 推送中同步作者为系统用户（若不存在则创建，默认密码 123456）；**当仓库被系统记录时自动创建对应项目并生成并绑定一张多维表**；附件存 MinIO；讨论与记录落库。 |

## 4. 功能需求详述

### 4.1 账号体系（不接入 GitHub OAuth）

- **账号来源**：在现有项目推送（Webhook）流程中，根据 commit 的 **author_email / author_name** 自动创建或关联**系统用户**。若系统中不存在该邮箱对应用户，则自动创建账号，**默认密码为 123456**（存储时加密）。
- **登录方式**：使用**邮箱 + 密码**登录（与现有管理员登录类似，可复用或扩展登录接口）；不再使用 GitHub OAuth，减少外部依赖与费用。
- **密码重置**：  
  - **用户**：可在「个人设置」中自行修改密码（需验证旧密码或预留邮箱验证，MVP 可简化为登录后直接改密码）。  
  - **管理员**：超级管理员/子管理员可为任意用户**重置密码**（如重置为 123456 或临时密码）。
- **与现有管理员关系**：现有「管理员账号密码登录」保留；多维表模块与现有后台可共用同一套会话（如 JWT），通过角色区分「超级管理员 / 子管理员 / 项目管理员 / 普通成员」。

### 4.2 项目与多维表/工作看板

- **项目与仓库（根据仓库自动创建）**：**现版本所有项目均根据仓库自动创建，不需要管理员手动创建项目**。系统识别到有几个仓库被记录，就认为有几个对应的项目。实体「项目」与「仓库」**严格一一对应**（一个项目绑定一个仓库，一个仓库对应一个项目）。项目具备名称、描述、关联仓库、创建时间、状态（进行中/已归档等）。**触发**：当某个仓库被系统记录时（例如现有业务中「添加/接入仓库」或首次同步该仓库时），在持久化该仓库的同时自动创建对应项目并同时创建该项目的 **默认** 多维表（及默认列）。
- **多维表与项目绑定（双表模型）**：  
  - **`issue_tracking`（项目调整）**：**随项目自动创建**，每个项目 **恰好一张**；列以 **问题描述、当前状态、验收结果** 等为主，适合 **售后 / 缺陷 / 变更跟踪**。  
  - **`feature_backlog`（待开发功能清单）**：**按需创建**（如报价页「导入多维表格」或后端 `ensureFeatureBacklogTable`）；列以 **功能名称、功能描述、开发进度、预计完成** 等为主，适合 **开发阶段排期**；与上表 **共用** `biz_record` / 留言 / 附件，**不共用** `table_id`。  
  - 数据库层通过 `biz_project_table.purpose` + **`uk_project_purpose(project_id, purpose)`** 保证同一项目下同用途至多一张表。
- **入口与动线**：  
  - **首页**：顶部提供**「项目协作」**入口；点击后进入「选择项目」页，仅展示**当前用户有权限的项目**（超级管理员全部，子管理员仅被分配了管理权限的项目，普通成员仅参与的项目即仓库作者自动同步的项目）；选择某一项目后进入该项目的多维表/工作看板。  
  - **项目详情页**：在现有项目详情页顶部提供**「项目协作」**入口；点击后**直接进入当前项目的多维表**，无需再选项目。  
  - 与现有「右上角切换项目」配合：切换项目后，若再点「项目协作」，应进入**当前已选项目**的表格（若当前选中的是项目详情上下文，则同上直进该项目的表）。

### 4.3 多维表结构（表与列）

- **表**：一个项目可有 **多张** 物理表（`biz_project_table` 多行），按 **`purpose`** 区分语义；生命周期随项目。
- **默认列 — `issue_tracking`（参考 CSV）**：参考 `docs/多维表格辅助材料/邻聚惠任务清单_数据表.csv`，建议默认列包括（列名与类型可配置）：  
  - **问题描述**（多行文本）  
  - **归属模块**（单选，**选项组按项目配置**）  
  - **图像展示**（附件）  
  - **负责人**（多选）  
  - **重要程度**（单选）  
  - **创建人**、**当前状态**、**验收结果**、**创建时间** 等  
- **默认列 — `feature_backlog`（待开发功能清单）**：  
  - **功能名称**、**功能描述**、**归属模块**、**开发进度**（单选：待开发 / 开发中 / 联调中 / 测试中 / 已完成 / 阻塞）、**图像展示**、**负责人**、**重要程度**、**预计完成**、**创建人**、**创建时间**  
  - 与 issue 表 **同样** 支持记录详情中的 **讨论/留言** 与 **附件**。  
  具体列名与选项组可在附录与实现对齐。
- **列类型**：文本（单行/多行）、负责人（多选）、单选/多选（绑定选项组）、日期、附件、数字等；支持快捷选填选项组（见 4.6）。

### 4.4 记录级讨论（留言）

- **说明**：参考 CSV 中**无**聊天/沟通功能；本系统在**每条记录**下增加**讨论区**，用于任务/需求相关的留言与沟通。
- **行为**：每条记录对应一条任务/需求；记录下有**留言列表**；用户可发表留言（纯文本 MVP）；每条留言记录发言人、时间、内容；按时间排序；支持分页。
- **展示**：在记录详情或侧滑/抽屉中提供「讨论」Tab，列出全部留言并可发表新留言；支持「查看讨论记录」。

### 4.5 附件上传与存储（MinIO）

- **方案**：采用 **MinIO** 作为对象存储，**与现有 Docker 部署一起编排**；无需阿里云 OSS，降低费用与外部依赖。
- **MinIO 部署**：  
  - 在 **docker-compose.yml**（本地开发）与 **docker-compose.prod.yml**（生产）中新增 **minio** 服务；使用官方镜像 `minio/minio`，端口 9000（API）与主机 9002 映射容器 9001（Console，可选，避免主机 9001 冲突）；数据卷持久化；启动命令为 `server /data`。  
  - **Bucket 命名规范**：建议使用 **单一 Bucket**，如 `autoattend-attachments`；对象 key 按 `{projectId}/{recordId}/{uuid}_{filename}` 组织，便于按项目/记录隔离与清理。  
  - 后端通过环境变量配置 MinIO 端点（如 `http://minio:9000`）、Access Key、Secret Key、Bucket 名称；应用启动时若 Bucket 不存在则自动创建（可选）。
- **上传与下载**：前端上传至后端接口，后端转存至 MinIO 并落库附件元信息；下载时后端生成临时预签名 URL 或代理转发，避免暴露 MinIO 密钥。
- **落库**：附件元信息（文件名、大小、storage_key、uploaded_by、created_at）写入 `biz_attachment`；记录与附件为一对多。

### 4.6 负责人与快捷选填

- **负责人**：列类型为「负责人」多选；数据来源为**系统用户**（即 commit 作者同步来的用户 + 管理员创建的子管理员等）；每条记录可选 0～N 个负责人。
- **快捷选填**：状态、重要程度、解决情况、验收结果、归属模块等列绑定**选项组**；选项组在项目或全局维护（名称 + 可选值列表）；默认值可与 CSV 对齐（见附录）。

### 4.7 权限与可见范围

- **超级管理员**：可查看所有项目；可创建子管理员；可为任意用户分配「某项目的项目管理员」权限；可重置任意用户密码。
- **子管理员**：**仅可查看与管理被分配了管理权限的项目**；不可创建子管理员或分配项目权限；在被分配的项目内可管理成员、表格与选项等。
- **项目管理员**：仅对被分配的项目拥有管理权限（成员、表格结构等）；可查看并管理该项目的多维表。
- **普通成员**：仅能查看**自己参与的项目**（见下「项目成员来源」）；可在此类项目的表格中增删改记录、上传附件、留言、选负责人与选项。
- **项目成员来源**：项目与仓库**严格一一对应**；**项目成员由该仓库的 commit 作者自动同步**（即某仓库的 push 中出现的 author_email 对应用户自动成为对应项目的成员）；管理员/项目管理员可在此基础上增删成员或设置项目管理员角色。

### 4.8 CSV + AI 智能导入（DeepSeek，分片清洗）

- **动机**：外部 CSV 的列名、顺序、枚举值常与项目多维表不一致，**直接导入**易污染数据或无法写入；因此采用「**先 AI 对齐结构、再人工确认、最后批量插入**」。
- **流程**：  
  1. 用户在表格页选择 **「CSV 智能导入」**（或与「AI 录入」同一弹窗内切换 **CSV + DeepSeek** 页签），上传 `.csv`（建议 **UTF-8**，支持 BOM）。  
  2. 后端读取 CSV **首行作为表头**，其余为数据行；**按可配置上限分片**（默认每片约 **12** 条数据行、不含表头，配置项 `app.collab.csv-ai-import.max-rows-per-chunk`，范围 **5～30**，环境变量 `COLLAB_CSV_AI_CHUNK_ROWS`），通过 **会话 + 多请求** 依次调用 **DeepSeek**（与报价等共用管理后台 **AI 配置** 中的 Key；**非**通义千问）。每片请求携带：当前项目的**列名、类型、选项组可选值**、**项目成员邮箱列表**（用于 `owners` 字段匹配）、本片的 **Markdown 表**（表头 + 行）。  
  3. 模型输出与「AI 录入」一致的任务草稿 JSON：`{"items":[{ title, description, module, importantLevel, status, acceptResult, owners[], attachmentIds:[] }]}`。服务端合并各片 `items` 返回前端；若某片失败，在响应 `warnings` 中提示，其余片结果仍返回（若全部失败则报错）。  
  4. 用户在前端 **编辑草稿** 后，调用既有 **`POST /api/collab/projects/{id}/ai-tasks/commit`** 落库（与通义 AI 录入共用插入逻辑）；**创建人、创建时间**仍由系统按当前用户与时间写入。  
- **限制（建议）**：单文件体积上限约 **6MB**（可配置）；数据行总数上限约 **3000**（防止超长耗时与费用）；仅接受 **`.csv`** 扩展名；分片行数 **30** 为服务端允许的配置上限；**列很多或 DeepSeek 较慢时**，宜调小单批行数并配合 Nginx `proxy_read_timeout`（见 `docs/nginx-api-timeout-snippet.conf`）。  
- **与「AI 录入（通义）」关系**：**文本+附件** 走通义多模态；**CSV 清洗** 走 DeepSeek，二者入口可并列或 Tab 切换，**提交接口统一**。

### 4.9 与现有 AutoAttend 的集成

- **入口**：  
  - 现有系统**项目页顶部**增加**「项目协作」**入口；  
  - **首页**点击「项目协作」→ 选择有权限的项目 → 进入该项目的多维表；  
  - **项目详情页**点击「项目协作」→ 直接进入当前项目的多维表。  
- **与「切换项目」**：右上角切换项目后，再点「项目协作」时进入的即为当前选中项目的表格。  
- **数据**：新建独立业务表（用户、项目、表、列、记录、讨论、附件）；**项目与仓库严格一一绑定且项目随仓库被记录而自动创建**，项目成员由仓库 commit 作者自动同步，便于后续扩展「需求—commit」关联。

### 4.10 多维表仪表盘视图（本期新增）

- **适用范围**：仅 **`issue_tracking`（项目调整）** 表显示「仪表盘」入口；**`feature_backlog`** 以表格推进为主，本期不展示售后向仪表盘（避免与「开发进度」语义混淆）。
- **开关位置**：在「项目协作-多维表」页头工具栏新增 **「仪表盘」开关按钮**（与「新建记录」「AI 智能录入」同级）。
- **切换行为**：  
  - 默认显示「表格视图」。  
  - 点击「仪表盘」后切换为「仪表盘页面」。  
  - 仪表盘页面支持点击「返回表格」切回原表格视图。  
- **数据范围**：基于当前项目当前表的记录（与当前用户权限一致）；统计口径与表格数据保持一致。
- **图表与卡片（本期）**：  
  1. **任务完成情况饼图**：已完成 / 未完成（完成判定优先参考「解决情况 / 验收结果 / 当前状态」中的完成类值，如“已解决”“已完成”“通过任务关闭”等）。  
  2. **近期新增任务折线图（按周）**：按记录创建时间聚合最近 N 周新增数量。  
  3. **问题解决折线图（按周）**：按周统计“进入完成态”的任务数量（本期默认按当前状态判定完成，后续可升级为按状态变更历史口径）。  
  4. **重要程度统计柱状图**：按「重要程度」字段选项分组计数。  
  5. **问题词云图（预留）**：本期展示占位卡片与“即将上线”提示。  
  6. **问题平均解决时长计数图（预留）**：本期展示占位卡片与“即将上线”提示。  
- **交互要求**：  
  - 图表支持在数据更新后自动刷新。  
  - 无数据时显示空态提示，不报错。  
  - 保持与现有筛选/编辑逻辑兼容（切回表格后上下文不丢失）。

### 4.11 提交联动任务状态更新（项目级 AI 开关）

- **目标**：当开发者产生新 commit 且触发 AI 自动分析时，结合项目多维表候选任务，推断并建议（或自动）更新任务状态字段（如「解决情况」「验收结果」），形成「代码提交 -> 任务状态」闭环。
- **开关位置**：在 `/collab/projects/{id}/table` 的「开发与数据看板」新增配置区（项目级，默认关闭）。
- **配置项建议**：  
  - `启用提交联动任务状态更新`（布尔）  
  - `联动模式`（`confirm` / `auto` / `suggest_only`，默认 `confirm`）  
  - `最小置信度`（可选）
- **提示文案（开启时必须展示）**：  
  - **“AI 的分析可能不准确，请仔细辨别。”**  
  - 在开关区域常驻展示；`auto` 模式建议再弹一次确认。
- **行为约束**：  
  - 仅更新约定字段，不改任务主体描述。  
  - 仅对当前项目（仓库映射到该项目）生效。  
  - 关闭开关后仍保留历史建议与历史更新记录，不再产生新联动更新。
- **审计建议**：记录来源 `source=commit_ai_linkage`，保留 commit_sha、record_id、old_value/new_value、置信度与处理人（自动或人工）。

### 5.1 用户与登录（自有账号）

| 表名 | 说明 |
|------|------|
| `biz_user` | id、email（唯一，来自 author_email）、name、password_hash、role（super_admin/sub_admin/member）、created_at、updated_at。 |

| 表名 | 说明 |
|------|------|
| `biz_project` | id、name、description、**repo_id**（关联仓库，严格一一对应；**项目随仓库被记录时自动创建**，非管理员手动创建）、status、created_at、updated_at。 |
| `biz_project_member` | id、project_id、user_id、role（admin/member）、source（sync/manual）。role=admin 表示该项目管理员；source=sync 表示由仓库作者自动同步，manual 表示管理员手动添加。 |
| `biz_project_table` | id、tenant_id、project_id、name、**purpose**（`issue_tracking` / `feature_backlog`）、created_at。同一 `project_id` 下 **每种 purpose 至多一行**（`uk_project_purpose`）。项目自动创建时插入 **issue_tracking** 一行；**feature_backlog** 按需插入。 |

| 表名 | 说明 |
|------|------|
| `biz_table_column` | id、table_id、name、column_type、option_group_id、sort_order。 |
| `biz_option_group` | id、name、options（JSON 数组）、scope（project/global）。scope=project 时关联 project_id，用于「归属模块」等按项目配置的选项组。 |
| `biz_record` | id、table_id、created_by、created_at、updated_at。 |
| `biz_record_field` | id、record_id、column_id、value_text、value_number、value_date、value_json（负责人 id 列表、选项、附件 id 列表等）。 |
| `biz_record_comment` | id、record_id、user_id、content、created_at。 |
| `biz_attachment` | id、**project_id**、record_id、file_name、file_size、storage_key、uploaded_by、created_at。project_id 为所属项目（支持项目级上传/AI 录入未绑定记录时）；storage_key 为 MinIO 对象 key。 |

### 5.4 子管理员与项目权限

- **子管理员**：`biz_user.role = sub_admin`；由超级管理员在「用户管理」中创建（邮箱+初始密码），或指定某现有用户为子管理员。  
- **项目管理员**：通过 `biz_project_member` 的 `role=admin` 表示；超级管理员/子管理员可为某用户添加 `project_id + user_id + role=admin`，即该用户对该项目拥有管理权限。

## 6. MinIO 与 Docker 编排

### 6.1 服务定义（示例）

在 **docker-compose.yml** 与 **docker-compose.prod.yml** 中增加：

```yaml
  minio:
    image: minio/minio:latest
    container_name: autoattend-minio
    restart: unless-stopped
    command: server /data
    environment:
      MINIO_ROOT_USER: ${MINIO_ROOT_USER:-minioadmin}
      MINIO_ROOT_PASSWORD: ${MINIO_ROOT_PASSWORD:-minioadmin}
    volumes:
      - autoattend_minio_data:/data
    ports:
      - "9000:9000"
      - "9002:9001"   # Console，主机 9002 避免与 9001 冲突
```

在 **volumes** 中增加：`autoattend_minio_data`。

### 6.2 后端配置

- **账号与密钥**：**使用 MinIO 默认配置即可**（用户名 `minioadmin`、密码 `minioadmin`），开发与生产均可不单独配置 `MINIO_ROOT_USER` / `MINIO_ROOT_PASSWORD`；若需修改则通过环境变量覆盖。
- 环境变量（示例）：`MINIO_ENDPOINT=http://minio:9000`、`MINIO_ACCESS_KEY=minioadmin`、`MINIO_SECRET_KEY=minioadmin`、`MINIO_BUCKET=autoattend-attachments`。
- 生产环境中 backend 的 `depends_on` 增加 `minio`；backend 内通过 MinIO Java SDK 或 S3 兼容接口上传/下载，Bucket 在首次使用时自动创建（若 MinIO 支持）。

### 6.3 Bucket 命名与 Key 规范

- **Bucket**：`autoattend-attachments`（固定一个，便于备份与策略统一）。  
- **Object Key**：`{projectId}/{recordId}/{uuid}_{原始文件名}`，便于按项目/记录隔离与排查。

| 方法 | 路径 | 说明 |
|------|------|------|
| GET  | /api/projects/{id}/records | 分页列表，支持组合筛选（query：`filters` 为 JSON 数组）。本版本支持筛选列：`重要程度`、`当前状态`、`验收结果`；多条规则之间为 AND（全部满足才返回）。 |
| POST | /api/projects/{id}/records | 新增记录。 |
| GET  | /api/records/{recordId} | 单条记录详情（含讨论、附件列表）。 |
| PUT  | /api/records/{recordId} | 更新记录。 |
| DELETE | /api/records/{recordId} | 删除记录。 |
| GET  | /api/records/{recordId}/comments | 分页留言。 |
| POST | /api/records/{recordId}/comments | 发表留言。 |
| POST | /api/records/{recordId}/attachments | 上传附件（后端写 MinIO）。 |
| GET  | /api/attachments/{id}/download | 获取下载 URL 或代理下载。 |
| DELETE | /api/attachments/{id} | 删除附件。 |

**组合筛选（本版本：AND + 重要程度/当前状态/验收结果）**：

- `filters` 参数：JSON 数组，每条规则结构：
  - `columnId`：列 ID
  - `op`：操作符（`eq` / `ne` / `contains` / `not_contains` / `empty` / `not_empty`）
  - `value`：除 `empty/not_empty` 外必填（按系统选项组配置的实际值匹配）
- 逻辑：多条规则之间使用 AND（同时满足所有规则的记录才会被返回）。
- 操作符语义（默认）：
  - `eq/ne`：大小写不敏感相等 / 不等
  - `contains/not_contains`：子串包含 / 不包含（大小写不敏感）
  - `empty/not_empty`：空 / 非空（空=字段值为 `null/''/[]` 之一；具体以服务端默认解析为准）

**已实现协作路径（`/api/collab/...`，JWT）补充：**

## 8. 前端页面结构建议

- **登录**：使用现有或扩展的登录页，支持「邮箱+密码」；登录后根据角色进入首页或管理后台。
- **首页 / 项目区**：顶部**「项目协作」**入口；点击 → 若有多个有权限项目则进入「选择项目」页，选择后进入该项目多维表；若仅一个项目则可直接进入。
- **项目详情页**：顶部**「项目协作」**；点击 → 直接进入当前项目的多维表。
- **多维表页**：左侧 **侧边栏** 在 **`issue_tracking` / `feature_backlog`** 之间切换（路由 query **`purpose`**）；主区为表格视图（行=记录，列=字段）；支持单元格编辑、记录详情抽屉（含讨论 Tab、附件）；负责人与选项为下拉/多选；工具栏提供 **组合筛选**（`issue_tracking`：重要程度/当前状态/验收结果；`feature_backlog`：重要程度/开发进度）、**AI 智能录入（通义）**、**CSV 智能导入（DeepSeek）**、**仪表盘开关**（仅 `issue_tracking`）。切换到仪表盘后展示 2 行图表卡片：任务完成情况饼图、周新增折线图、周解决折线图、重要程度柱状图，以及词云/平均解决时长两个预留卡片。
- **管理**：超级管理员可创建子管理员、分配项目管理员、重置密码；子管理员/项目管理员在各自范围内管理项目与成员。

## 9. 非功能要点

- **安全**：密码加密存储；MinIO 密钥仅后端持有；附件下载鉴权或临时签名 URL。  
- **权限**：所有接口按「当前用户角色 + 项目成员/项目管理员」做可见与可写校验。  
- **性能**：记录列表分页；附件大文件可考虑后端生成预签名 URL 由前端直传 MinIO（可选）。  
- **网关**：单批 `/chunk` 若超过 Nginx **`proxy_read_timeout`**（常见默认 **60s**）仍会 **504**（与前端 axios 超时无关）。处理：**①** 将 `COLLAB_CSV_AI_CHUNK_ROWS` 调小（如 **8**）；**②** 在反代 `location` 提高 `proxy_read_timeout` / `proxy_send_timeout`（示例见 **`docs/nginx-api-timeout-snippet.conf`**）；**③** 后端 `app.deepseek.http.read-timeout-seconds` 不宜小于 Nginx 等待时间量级。

## 附录 B：已确认约定

1. **默认列与选项**：**「项目调整」** 默认表结构见附录 A；「归属模块」的选项组为**按项目配置**，后期根据不同项目由管理员/项目管理员维护。**「待开发功能清单」** 列见 **4.3**，与附录 A 并存为第二张表。
