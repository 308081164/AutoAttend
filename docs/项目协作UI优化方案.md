# 项目协作 UI 优化方案

**文档性质**：产品 / 体验设计评审稿（面向项目经理与研发协同评审）  
**范围**：协作空间内「开发与数据看板」首页及多维表切换导航的整体信息层次与布局策略  
**关联实现线索**：前端路由 `collab-table`（路径 `/collab/projects/:projectId/table`）当前在同一页面内承载「首页看板 + 表格视图」，首页区域包含传送门、协作类配置区块，以及内嵌的控制台数据看板能力。

---

## 1. 现状与问题

### 1.1 现状概览

当前「开发与数据看板」将传送门、邮件与 AI 联动、客户阅览看板等多类能力纵向堆叠在同一视图中，并内嵌数据看板以呈现仓库维度的开发指标。左侧另设可折叠导航，用于在「首页看板 / 项目调整表 / 功能清单」等视图之间切换。

### 1.2 核心问题（与用户反馈对齐）

| 问题类型 | 描述 |
|---------|------|
| **信息层次扁平** | 页面像「功能清单式堆砌」，项目经理优先需要的一眼结论（昨日进展、风险、里程碑）未形成视觉与阅读上的第一屏焦点。 |
| **高频与低频混排** | 「昨日开发进展总结」「传送门」等高频入口未获得足够权重；低频、一次配置长期不改的「协作能力配置」却与主工作流同屏、同权重，常态占用大量纵向篇幅。 |
| **项目身份弱** | 「项目基本信息」未稳定置于页面顶端作为认知锚点，跨日回顾时增加心智负担。 |
| **每日总结形态** | 「项目每日进展总结」若以长列表展示，会加剧纵向滚动与扫读成本。 |
| **核心数据呈现** | 项目经理关心的关键指标缺少「仪表盘式」聚合呈现，指标散落在各模块或内嵌看板中，难以形成管理视角的闭环。 |
| **导航占用横向空间** | 可折叠侧边栏在展开态仍挤占主内容区宽度；折叠态依赖缩写与反复展开，对「表格为主」的工作场景不友好。 |

---

## 2. 设计原则与目标

### 2.1 设计原则

1. **角色优先**：默认视角按「项目经理 / 负责人」的晨会与跟进节奏组织信息，开发者深度操作通过二级入口或专用视图承接。  
2. **频率驱动布局**：日更、日点检内容靠近首屏与顶栏；月级或一次性配置下沉或收纳。  
3. **锚点一致**：项目标识、阶段、关键干系人信息在协作空间内各子页保持同一顶栏或同一信息带宽，降低「我在哪个项目」的认知成本。  
4. **纵向吝啬、横向慷慨**：表格与看板为主内容，应尽可能获得全宽；辅助导航不长期占用主栅格列。  
5. **渐进披露**：配置类能力默认折叠、收纳或独立页，避免「打开页面即面对表单墙」。  

### 2.2 目标（可验收）

- 首屏 1～2 屏内可见：**项目身份条**、**昨日/今日摘要**、**传送门**、**核心 KPI 仪表盘**、**最近数日每日总结卡片**。  
- 「协作能力配置」默认不占主叙事区；展开后仍可通过锚点或独立路由完整编辑。  
- 取消「长期占用横向宽度的可折叠侧边栏」作为默认方案，改为对表格更友好的导航模式（见第 6 节）。

## 3. 信息架构（优先级分层）

建议采用四层结构，自上而下阅读优先级递减。

| 层级 | 内容 | 说明 |
|------|------|------|
| **L0 身份与上下文** | 项目名称、编号/仓库、当前阶段、负责人、环境标签 | 全页顶栏或粘性标题区，所有子视图复用。 |
| **L1 今日行动** | 昨日开发进展总结（或「最近一次同步摘要」）、阻塞与风险一句话、传送门 | 项目经理打开页面 10 秒内应完成「是否健康、今天要盯什么」的判断。 |
| **L2 管理仪表盘** | 进度、吞吐、缺陷/需求分布、周期类指标等 4～8 个 KPI 卡或小图 | 聚合呈现，避免与 L1 重复叙述；可链接到明细表或下钻。 |
| **L3 过程与明细** | 每日进展卡片（近 N 天）、内嵌/跳转的完整数据看板、多维表 | 卡片控高；完整看板可「在页面内折叠区」或「独立子路由全屏」。 |
| **L4 配置与治理** | 邮件日报、提交与表联动（AI）、客户阅览看板等 | 默认收纳；与 L1～L3 物理或视觉隔离。 |

## 4. 页面布局（线框级说明）

### 4.1 推荐主结构（ASCII 线框）

```
┌──────────────────────────────────────────────────────────────────────────────┐
│ L0 顶栏：项目基本信息（名称 / 仓库 / 阶段 / 负责人）          [设置▼] [帮助]   │
├──────────────────────────────────────────────────────────────────────────────┤
│ L1 摘要横带：昨日进展摘要（1～3 行） │ 风险/阻塞 │ 传送门（横向图标+文字）      │
├──────────────────────────────────────────────────────────────────────────────┤
│ L2 仪表盘： [KPI] [KPI] [KPI] [KPI] [趋势缩略]     （栅格，可 2×2 或 1×4）    │
├──────────────────────────────────────────────────────────────────────────────┤
│ L3 左：每日总结卡片（近 3～7 天，横向滑动或 2 列网格）                          │
│      右：可选「展开完整数据看板」或占位说明（嵌入/新开）                         │
├──────────────────────────────────────────────────────────────────────────────┤
│ L4 折叠区：「协作与通知配置」──────── 点击展开 ────────                        │
│         （邮件 / AI 联动 / 客户看板等表单，默认收起）                          │
└──────────────────────────────────────────────────────────────────────────────┘
```

### 4.2 与表格视图的关系

- 进入「项目调整表」「功能清单」等表格态时：**保留同一 L0 顶栏**；L1 可收缩为一行「返回概览」+ 当前表用途标签，避免表格列被导航挤压。  
- 表格工具栏（筛选、新建、AI 录入等）置于表格正上方，不与 L0 混排。

## 5. 关键模块说明

### 5.1 顶栏（L0）

- **内容**：项目名称、仓库标识、阶段/迭代、负责人与关键角色入口（只读展示为主）。  
- **行为**：右侧提供「项目设置」聚合入口（内含原协作配置、成员、权限等），避免在首页重复铺开表单。  

### 5.2 摘要区（L1）

- **昨日开发进展总结**：自动生成或半自动摘要，支持一键复制到晨会文档。  
- **传送门**：外链以图标 + 短标签横向排列，支持「常用置顶」；配置入口弱化到「管理传送门」链接。  

### 5.3 仪表盘（L2）

- **定位**：项目经理关心的核心数据以卡片或小型图表呈现，与控制台完整看板形成「摘要 vs 明细」关系。  
- **实现策略**：可复用现有数据看板的数据源，在本页仅取聚合指标；完整图表与多维度分析保留「展开」或独立入口。  

### 5.4 传送门

- 与摘要区同带或紧邻，保证**不折叠、不二次点击**即可到达常用系统（文档、仓库、CI、设计稿等）。  

### 5.5 项目每日进展总结（卡片）

- **范围**：默认展示最近 **3～7 天**（可配置），每张卡片：日期、一句话结论、关键数字、展开查看全文。  
- **布局**：横向滑动（移动端友好）或 2 列网格，**限制单卡高度**，避免长文撑开整页。  

### 5.6 协作能力配置（收纳）

- **原则**：一次配置、长期不改的能力不应与每日跟进抢首屏。  
- **形态**：  
  - **折叠面板**：默认收起，标题区显示「已启用 / 未配置」状态徽标；  
  - 或 **独立子路由**「项目设置 → 协作与通知」，首页仅保留状态摘要与「去配置」链接。

## 6. 侧边栏替代方案对比

**背景**：当前协作表格页使用左侧可折叠侧栏切换「首页看板 / 不同用途表格」，展开态占用横向空间，与「以表格为主」的密集信息界面存在张力。以下方案均不要求用户长期让出一列给导航。

| 方案 | 描述 | 优点 | 缺点 | 适用 |
|------|------|------|------|------|
| **A. 顶栏 Tab（或分段控件）** | 在 L0 下方或主标题行使用 Tab：`概览` \| `项目调整` \| `功能清单` | 零侧栏宽度损失；与常见后台一致，学习成本低；当前路由已通过 `query.purpose` 区分用途，可映射为 Tab 切换。 | Tab 数量不宜多（建议 ≤4）；项目名称很长时需处理截断与 Tooltip。 | **首选 MVP**：改动集中、与现有 `collab-table` 查询参数模型契合。 |
| **B. 主区顶部分段 + 溢出菜单** | 前 2～3 项用分段按钮，其余收入「更多」下拉 | 兼顾「常用两项一键达」与扩展性。 | 次要入口多一次点击；需定义「常用」规则。 | 表格类型继续增加时的演进形态。 |
| **C. 抽屉 / 独立「视图切换」** | 汉堡或「切换视图」打开右侧/左侧抽屉列出全部入口；或单独「视图选择」轻量页再进入表格 | 主区全宽；适合入口很多、或未来接入更多子表。 | 多一次交互打开抽屉；需明确关闭态与键盘可达性。 | 子视图显著增多或移动端优先时。 |
| **D. 设置页与作业页分离** | 「概览首页」与「各表格作业页」拆成不同路由（或不同布局壳），通过顶栏或列表进入 | 心智模型清晰：「看」与「改」分离。 | 路由与权限拆分工作量较大；需处理返回路径与状态保持。 | **二期**：信息架构大改版时。 |

**评审建议**：短期以 **方案 A** 为主交付；若产品规划明确会扩展多种「表用途」，在 A 基础上预留 **方案 B** 的溢出位；**方案 D** 与大型设置收纳（第 5.6 节）可合并规划为二期。

## 7. 与现有路由 / 页面的衔接建议

| 现有能力 | 衔接方式 |
|----------|----------|
| 路由 `collab-table`（`/collab/projects/:projectId/table`） | 保留为协作空间核心入口；首页与表格仍可通过 `query`（如 `purpose`）区分，**UI 上由侧栏改为顶栏 Tab**，避免破坏书签与分享链接。 |
| 项目列表 `collab-projects` | 从列表进入表格页时，默认落在「概览」Tab；上次访问的 Tab 可本地记忆（可选）。 |
| 内嵌数据看板 | L2 仪表盘可引用同源数据；完整内嵌看板移至 L3 折叠/全屏或子路由，减少与 L1 抢首屏。 |
| 控制台全局看板 | 保持独立路由能力；协作页仪表盘定位为「项目内摘要」，避免与控制台全量能力重复建设文案。 |

## 8. 分阶段实施建议

### 8.1 MVP（建议 1～2 个迭代）

1. **信息重组**：实现 L0 顶栏「项目基本信息」；将 L1（摘要 + 传送门）上移至首屏。  
2. **导航改造**：以顶栏 Tab 替代默认可折叠左侧导航；保留键盘与路由可访问性。  
3. **配置收纳**：协作类配置区默认折叠，展示配置状态徽标。  
4. **每日总结卡片**：仅展示近 N 天，卡片化 + 高度上限。  
5. **轻量仪表盘**：在概览 Tab 增加 4～6 个 KPI 卡（数据可来自现有看板接口的聚合字段）。  

### 8.2 二期

1. **路由级拆分**（可选）：概览与「深度表格作业」分壳，强化「看板 vs 作业」心智。  
2. **个性化**：项目经理自定义 KPI、传送门排序、摘要模板。  
3. **通知与摘要联动**：邮件日报与页面摘要双向引用（例如从邮件深链回某日卡片）。  
4. **移动端专项**：顶栏 Tab 滚动、卡片横滑、抽屉方案 C 的针对性优化。
