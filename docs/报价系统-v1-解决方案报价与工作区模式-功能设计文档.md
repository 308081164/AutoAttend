## 2. 背景与问题陈述

### 2.1 现状痛点（来自实际使用）

1. **交付物粒度偏细、偏单栈**  
   现有交互与数据结构容易使管理员按「一个网站 / 一个 APP / 一个后端」分别思考，而真实外包项目往往是 **Web + 移动端 + 后端 + 集成** 的**解决方案（Solution）**，需要 **一张报价视图下多子系统、共享风险与工期假设**，而不是多个彼此弱关联的报价单简单相加。

2. **报价结果与商务现实脱节**  
   客户与服务商在报价前往往已有 **参考总价区间** 或 **目标价**。系统若只输出「模型算出的唯一数字」，且 **明细行锁定不可调**，会导致用户不得不在 Excel 里重做一遍，平台价值被削弱。需要的是：**保留人天/复杂度逻辑作为「成本与合理性解释」**，同时允许在 **总价或子域层面做受控的等比例缩放**，并记录原因（审计与复盘）。

3. **租户类型多样，单一「报价优先」路径造成割裂**  
   - **创业团队 / 自研产品**：核心诉求是协作、代码、运维、客户看板等，**报价不是主任务**；若每次进入平台都被引导向「报价项目」，会产生 **工具链不连贯** 的感受。  
   - **同一团队接外包**：同一主体仍可能需要 **报价 → 合同** 能力，不能简单关掉模块，而应 **按需出现、弱打扰**。

## 3. 设计目标与原则

### 3.1 目标

| 编号 | 目标 | 说明 |
|------|------|------|
| G1 | **解决方案级报价** | 支持将一次商务机会建模为 **一个「解决方案报价」**，其下包含 **多个交付物（子系统）**，共享项目级假设（工期、风险、单价策略），并生成 **统一汇总与单一报价单视图**。 |
| G2 | **价格锚定与等比例调整** | 在 **计算基线价** 之后，允许用户设定 **目标总价** 或 **缩放系数（%）**，系统对 **可配置范围内的明细**（如按人天计价行）做 **等比例调整**，并保留 **调整前后对照** 与 **可选备注**（如「客户预算上限 28 万」）。 |
| G3 | **工作区/项目模式** | 在租户或工作区维度区分 **产品自研** / **外包交付** / **混合**，默认首页与导航 **按模式展示**，报价作为 **可选能力模块** 嵌入，而非唯一主线。 |
| G4 | **向后兼容** | 现有「单报价项目」数据与流程可映射为「仅含一个交付物的解决方案」，避免推倒重来。 |

### 3.2 原则

- **解释性优先**：调价不改变原始 **人天与基准单价** 的存档（或仅存「基线快照」），缩放作为 **派生层**，便于复盘「为什么当时报这个价」。  
- **受控自由**：等比例缩放可限制在 **总价约束** 或 **分交付物分组约束**（例如只对「开发实施」类行缩放，不动「第三方授权费」）。  
- **场景可选**：不自研的团队可走 **传统报价路径**；自研团队 **默认不展示** 报价入口，但可在 **项目设置** 或 **模板市场** 中一键启用。  
- **同一团队双场景**：通过 **项目类型标签** + **同一协作空间** 切换，避免维护两套账号体系。

## 4. 用户细分与典型路径

### 4.1 角色保持不变

仍兼容现有文档中的 **租户管理员、员工** 等；新增 **工作区/项目意图** 标签（见 6.2）。

### 4.2 三类典型路径

| 路径 | 描述 | 报价模块 |
|------|------|----------|
| **A. 外包服务商（主线）** | 以接单、报价、合同为主 | 默认展示「解决方案报价」入口；与协作、仓库强关联。 |
| **B. 自研产品团队** | 以版本、协作、运维为主 | 报价 **默认收起** 或仅在 **显式创建「商业/外包类项目」** 时出现。 |
| **C. 混合** | 同一租户既有产品线又有外包单 | 项目级选择 **产品交付** / **客户项目**；导航根据当前项目切换侧栏 emphasis。 |

## 5. 解决方案级报价（多交付物统一模型）

### 5.1 概念层级

建议采用三层概念（命名可与现有表名映射，此处为产品语义）：

```
解决方案报价（Solution Quote / 商务机会）
├── 项目级参数：客户名、币种、税率、整体风险档位、工期假设、备注
├── 交付物 A（例如：管理后台 Web）
│   └── 模块 / 功能点 / 人天…（与现有一致）
├── 交付物 B（例如：用户端小程序）
│   └── …
├── 交付物 C（例如：API 与后台服务）
│   └── …
└── 汇总：分交付物小计 + 项目级费用（如驻场、差旅）+ 总价
```

### 5.2 与「一个网站/一个 APP」的关系

- **不再强制**用户按「单栈」创建多个彼此独立的报价项目；**推荐路径**为在一个 **解决方案** 下 **新增交付物**。  
- **兼容路径**：允许从模板 **复制某一交付物** 到另一解决方案，或 **将历史单交付物报价升级为多交付物**（迁移向导：新建解决方案 → 导入现有报价项目为交付物 1 → 继续添加）。

### 5.3 AI 与结构化录入

- **AI 智能录入**：提示词与输出结构扩展为支持 **`deliverables[]`**，每个元素内含 `modules[]`；前端支持 **按交付物分 Tab** 审阅与合并。  
- **多维表联动**：可按交付物 **分别映射** 不同用途的表（若未来扩展），MVP 可仍为 **项目级统一导入** + 手工拆到各交付物。

### 5.4 报价单与合同

- **报价单 PDF**：封面为 **解决方案总价**；目录级列出 **各交付物小计**；附录可附各交付物明细。  
- **合同**：标的描述支持 **「整体系统」+ 分系统清单** 引用，避免法律文本只绑定单一 URL。

## 6. 价格锚定与等比例调整

### 6.1 场景

- 与客户已口头约定 **约 25～30 万**，模型算出 **32 万**，需整体 **压到 28 万** 仍保持明细结构可解释。  
- 客户要求 **各模块同比例让利 8%**（统一折扣）。  
- 部分金额为 **固定第三方成本**（如短信、地图、证书），**不参与**缩放。

### 6.2 功能要点

| 能力 | 说明 |
|------|------|
| **基线价** | 由人天 × 基准单价 × 风险等计算出的 **未缩放总价**，持久化存档。 |
| **缩放模式** | ① 按 **总价反算系数** `k = 目标总价 / 基线总价`；② 用户直接输入 **系数** 或 **折扣%**。 |
| **缩放作用域** | 默认：对所有 **「内部实施」类明细行** 等比例调整；**标记为例外的行**（第三方实报实销、固定授权费）**不参与**。 |
| **展示** | 每行可展示：**基线金额**、**系数**、**调整后金额**；汇总区展示 **基线总价 / 调整后总价**。 |
| **审计** | 记录：**操作者、时间、目标价或系数、备注**（选填）。 |

### 6.3 与「小计/组件不可调」的冲突处理

- 产品规则明确为：**锁定的是「基线计算规则」的可追溯性**，而非 **最终商务价**。  
- 若需 **单行议价**（非等比例），可作为 **Phase 2**：在等比例层之上允许 **单行覆盖**，并显示与基线差异。

## 7. 降低非报价用户的割裂感（工作区模式）

### 7.1 租户或工作区设置

建议增加 **默认项目意图**（可后续再做 UI）：

| 设置项 | 选项 | 影响 |
|--------|------|------|
| **主业类型** | 自研产品 / 外包服务 / 混合 | 控制台默认首页、快捷创建、侧栏排序 |
| **报价模块** | 开启 / 关闭 / 仅管理员可见 | 关闭时隐藏主导航中「报价」入口；数据保留 |

### 7.2 项目级标签

创建协作或报价相关项目时可选：

- **内部产品研发**：弱化报价，强调路线图、版本、缺陷。  
- **客户交付项目**：展示报价、合同、验收。  

同租户可在不同项目间切换，**无需**多个租户账号。

### 7.3 引导与空状态

- 自研为主租户：首次登录 **不** 强推「创建报价项目」；主推 **仓库 / 看板 / 发布**。  
- 在 **帮助中心** 说明：**何时启用报价**（例如接外包单时一键启用模块）。

## 8. 数据与接口（高层）

> 具体表名与字段在开发阶段与现有 `biz_quote_*` 对齐；此处仅列逻辑实体。

| 逻辑实体 | 关键字段（示例） |
|----------|------------------|
| `SolutionQuote` | id、租户、客户、状态、基线总价、调整后总价、缩放系数、缩放备注 |
| `Deliverable` | solution_id、名称、类型（web/app/api/…）、排序 |
| `QuoteLine` | 归属 deliverable、是否参与缩放、基线金额、调整后金额 |
| `PriceAdjustmentLog` | solution_id、旧总价、新总价、系数、操作者、原因 |

对外 API 宜区分：

- `GET/POST .../solutions` — 解决方案 CRUD  
- `POST .../solutions/{id}/apply-scale` — 应用等比例缩放  
- 现有 `.../quote/projects/...` 可 **别名或迁移** 为单交付物解决方案。

## 10. 分期实施建议

| 阶段 | 范围 | 价值 |
|------|------|------|
| **Phase A** | 数据模型：解决方案 + 多交付物；UI：Tab/步骤式编辑；汇总报价单 **展示** 分交付物小计 | 解决「一整套系统」主痛点 |
| **Phase B** | 基线价存档 + 等比例缩放 + 例外行 + 调整日志 | 解决「议价区间、统一折扣」 |
| **Phase C** | 租户/项目 **工作区模式**、导航与空状态策略 | 解决自研团队割裂感 |
| **Phase D** | AI 录入结构化输出 `deliverables[]`；历史项目迁移向导 | 提升效率与迁移体验 |

## 11. 非目标与风险（本期不承诺）

- 自动根据市场价爬虫调价。  
- 复杂 **非线性** 调价（仅部分模块打折）需单独评估，默认先做 **全局/分组等比例**。  
- 多币种税务引擎（可在解决方案层仍用单一币种 MVP）。

---

## 12. 验收要点（供测试编写参考）

- 创建一个含 **≥3 个交付物** 的解决方案，汇总价等于各交付物小计 + 项目级费用。  
- 对可缩放行应用 **系数 0.9**，第三方固定行金额不变，**调整后总价** 与手工验算一致。  
- 租户关闭「报价模块」后，主导航无入口，但 **管理员** 仍可通过直达链接或设置页开启（若产品如此定义）。  
- **混合租户**：项目 A 为自研、项目 B 为客户交付，侧栏与默认动作符合项目标签。
