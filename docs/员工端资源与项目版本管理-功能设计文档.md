# 员工端资源与项目版本管理 — 功能设计文档

## 1. 文档说明

- **功能名称**：员工端资源与项目版本管理（OSS 资源区 + 类 Git 项目推送）
- **版本**：v0.1
- **目标**：为建模师等员工提供登录后可用的「资源下载」与「个人 3D 项目版本推送」能力；资源存储采用 OSS，项目推送采用类 Git 的版本记录，并基于推送数据实现类似 AutoAttend 的工作情况可见与监控。
- **依赖**：现有 AutoAttend 后端与前端技术栈、管理员后台；需接入企业 OSS（如阿里云 OSS、腾讯云 COS、MinIO 等）。

---

## 2. 功能目标与范围

### 2.1 目标

- **资源区**：以 OSS 为底层存储，员工登录后像使用 NAS 一样**浏览、下载**企业共享资源（参考模型、贴图、规范文档等），无需直接暴露 OSS 控制台或密钥。
- **项目推送**：员工可创建「我的项目」，以**类 Git 的版本形式**推送 3D 项目文件；每次推送记录「谁、何时、哪个项目、版本说明、文件清单」，便于追溯与统计。
- **工作情况可见**：基于推送记录做活动时间线、按人/按项目统计、长期无推送预警等，与现有 AutoAttend 开发侧监控形成统一的可视能力。

### 2.2 范围（本期）

| 在范围内 | 不在范围内（可后续迭代） |
|----------|---------------------------|
| 员工登录与权限 | 资源区在线预览（如 3D 预览） |
| OSS 资源区：目录浏览、文件下载 | 资源区全文检索 |
| 我的项目：创建、推送新版本、版本历史查看/下载 | 项目内二进制 diff、合并 |
| 推送记录落库，供看板/统计/预警 | 多人协作编辑、冲突解决 |
| 管理员查看员工/项目活动（可选与现有看板整合） | 移动端独立 App |

---

## 3. 用户角色与场景

| 角色 | 操作 |
|------|------|
| **员工（建模师等）** | 登录 → 浏览/下载资源区（OSS）；创建/管理「我的项目」；推送新版本（上传文件 + 版本说明）；查看本人项目的版本历史并下载任意版本。 |
| **管理员** | 配置 OSS 与资源区目录；管理员工账号与权限；查看全员/全项目活动与推送统计；可选与现有 AutoAttend 看板整合。 |
| **系统** | 对接 OSS 列表/下载；存储项目版本文件与元数据；提供活动数据给看板与预警。 |

## 4. 功能需求详述

### 4.1 员工认证与权限

- **登录**：员工使用独立账号密码（或与现有统一认证对接）登录员工端；与管理员后台隔离（不同入口或不同角色）。
- **权限**：登录后仅可访问「资源区」「我的项目」及本人推送记录；不可访问其他员工的项目内容或管理员配置。
- **实现建议**：复用或扩展现有用户表，增加角色（如 `employee`）及员工端会话；资源区与项目 API 均校验当前用户身份。

### 4.2 资源区（OSS 存储，NAS 式使用）

- **定位**：企业统一发放的**只读资源**，如参考模型、贴图库、规范文档等，底层存放在 OSS，对员工表现为「像 NAS 一样的目录 + 下载」。
- **存储**：使用一个或多个 OSS Bucket（或 Bucket 内前缀），由管理员在后台配置 Bucket 名称、地域、访问密钥（或 STS 临时凭证）；建议使用**服务端签名 URL** 或**后端代理下载**，不把 OSS 密钥暴露给前端。
- **目录结构**：在 OSS 内按业务约定前缀组织，例如 `resources/参考模型/`、`resources/贴图/`、`resources/规范/`；后端根据前缀列出「目录」与「文件」，并生成下载链接或通过后端流式转发。
- **能力**：
  - **目录浏览**：按层级展示目录树或面包屑；列出当前目录下的子目录与文件（名称、大小、最后修改时间等）。
  - **文件下载**：点击文件生成临时下载链接（如 OSS 签名 URL 有效期 15 分钟）或由后端从 OSS 读流并返回给浏览器；大文件建议使用签名 URL 减轻后端带宽。
- **权限**：仅登录员工可访问；若需细分（如按部门可见不同目录），可在后端按员工属性过滤可见前缀。

### 4.3 我的项目与类 Git 版本推送

- **项目**：员工可创建多个「项目」，每个项目有名称、创建时间、归属人；仅创建者可见与管理（列表、推送、历史）。
- **推送新版本**：
  - 员工在项目内点击「推送新版本」，填写**版本说明**（必填，类似 Git commit message），并**上传本次版本包含的文件**（可多文件，如 .blend、.max、贴图等）。
  - 系统为本次推送生成**版本号**（建议自增整数或「项目内版本序号」），记录**提交人、提交时间、版本说明、文件列表（路径 + 存储 key 或 size）**。
  - 文件存储：建议上传至 OSS 独立前缀，如 `projects/{projectId}/versions/{versionId}/`，或落库元数据 + 文件存 OSS，通过 URL/key 关联。
- **版本历史**：
  - 列表展示该项目的所有版本：版本号、版本说明、提交人、提交时间；支持分页。
  - 每个版本可「下载」：打包该版本下所有文件为 zip，或提供文件列表逐一下载；下载通过后端生成签名 URL 或代理下载，不暴露 OSS 密钥。
- **与 Git 的类比**：一次「推送」≈ 一次 commit；无分支/merge，仅线性版本历史；不实现二进制文件的 diff，仅做「版本快照 + 元数据」管理。

### 4.4 工作情况可见与监控（类似 AutoAttend）

- **数据来源**：上述「推送」记录（谁、何时、哪个项目、版本说明）。
- **能力**：
  - **活动时间线**：按人或按项目展示「最近推送」列表。
  - **统计**：每人推送次数、每项目版本数、最近 N 天活跃人数等。
  - **简单预警**：如某人在某项目上超过 N 天无推送，或某项目长期无人更新（规则可配置或写死首版）。
- **展示**：可在现有管理员看板中增加「员工项目活动」Tab，或单独页面；与开发侧 Git 提交看板并列，形成「开发 + 建模」统一视图（可选）。
- **实现**：推送记录落库（见下文数据模型），查询接口支持按人、按项目、按时间范围筛选；前端看板调用这些接口做图表与列表。

### 4.5 管理员侧配置与视图

- **OSS 配置**：Bucket（及地域）、AccessKey/SecretKey 或 STS 角色；资源区根前缀（如 `resources/`）；可选：多个 Bucket 或前缀映射为不同「资源库」名称。
- **员工账号**：员工列表、启用/禁用、重置密码等；若已有统一账号体系，可只做角色与权限映射。
- **活动视图**：按人/按项目/按时间的推送记录与统计；可选导出。

## 5. 数据模型建议

### 5.1 员工与认证（若新建）

| 表名 | 说明 |
|------|------|
| `aa_employee` | 员工 id、登录名、密码哈希、姓名、部门、状态、创建/更新时间。 |

（若与现有用户表合并，可仅扩展角色与员工端权限字段。）

### 5.2 资源区（可选落库，用于权限或缓存）

| 表名 | 说明 |
|------|------|
| `aa_resource_config` | id、bucket、region、root_prefix、credentials 引用或加密存储、创建/更新时间。 |

资源区目录与文件列表可**仅从 OSS 实时列举**，不落库；若需「目录树缓存」或「可见范围配置」可再增加表（如 `aa_resource_visible_prefix` 按角色/部门配置）。

### 5.3 项目与版本

| 表名 | 字段概要 |
|------|----------|
| `aa_employee_project` | id、name、owner_employee_id、created_at、updated_at。 |
| `aa_employee_project_version` | id、project_id、version_number（项目内自增）、message（版本说明）、created_by、created_at；可选：file_count、total_size。 |
| `aa_employee_project_version_file` | id、version_id、file_name、oss_key 或 path、file_size；用于列表与打包下载。 |

项目文件实际存储在 OSS，例如：`projects/{project_id}/v/{version_id}/{file_name}`。

### 5.4 与现有 AutoAttend 的关系

- **数据**：员工项目推送独立表，不修改现有 `aa_commit`、`aa_webhook_delivery` 等。
- **看板**：可共用「活动」概念：开发侧展示 commit，员工侧展示 project_version，同一个人若同时有 Git 与项目推送，可在统一视图中合并展示（可选）。

## 6. OSS 使用约定

### 6.1 Bucket 与前缀规划

| 用途 | 建议前缀 | 说明 |
|------|----------|------|
| 资源区 | `resources/` 或配置的 root_prefix | 只读，由后端列举并生成下载链接。 |
| 员工项目 | `projects/{project_id}/v/{version_id}/` | 每个版本一个目录，内容为该版本上传的文件。 |

- 资源区与项目可使用同一 Bucket 不同前缀，或不同 Bucket（需在配置中区分）。

### 6.2 访问方式与安全

- **后端持有 OSS 密钥**：AccessKey/SecretKey 存配置或环境变量，仅后端使用；生产环境建议使用 **RAM 角色 + STS** 或 **服务端签名 URL**，限制权限与有效期。
- **列表**：使用 OSS ListObjects（按 prefix/delimiter）得到目录与文件，映射为前端树或列表。
- **下载**：
  - **推荐**：后端生成**预签名 URL**（如 15 分钟有效），前端重定向到该 URL 完成下载。
  - **备选**：后端从 OSS 读流并 proxy 到客户端（适合强审计或二次处理）。
- **上传**：员工推送时，前端将文件上传到后端 API，后端再写入 OSS 对应 prefix；或后端生成**上传用预签名 POST/PUT URL**，前端直传 OSS（需严格控制 prefix 与 overwrite 策略）。

### 6.3 大文件与配额

- 单文件大小限制：可在后端与前端约定（如 500MB），避免超时与内存占用。
- 单版本文件数量：可限制（如 50 个），防止滥用。
- 存储用量：可选按 project 或按员工统计，用于配额或告警（后续迭代）。

## 7. 接口设计建议

### 7.1 员工端 — 认证

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | /api/employee/auth/login | 员工登录，返回 token 或 session。 |
| POST | /api/employee/auth/logout | 登出。 |
| GET  | /api/employee/me | 当前员工信息。 |

### 7.2 员工端 — 资源区（OSS）

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | /api/employee/resources/entries | 列出某前缀下目录与文件；query: prefix, delimiter（可选）。 |
| GET | /api/employee/resources/download | 获取单个文件下载 URL 或触发代理下载；query: key 或 path。 |

### 7.3 员工端 — 项目与版本

| 方法 | 路径 | 说明 |
|------|------|------|
| GET  | /api/employee/projects | 当前员工的项目列表。 |
| POST | /api/employee/projects | 创建项目。 |
| GET  | /api/employee/projects/{id} | 项目详情。 |
| GET  | /api/employee/projects/{id}/versions | 版本列表（分页）。 |
| POST | /api/employee/projects/{id}/versions | 推送新版本（multipart 上传 + version message）。 |
| GET  | /api/employee/projects/{id}/versions/{vid}/download | 获取该版本打包下载 URL 或文件列表及各自下载链接。 |

### 7.4 管理员端（可选与现有 admin 合并）

| 方法 | 路径 | 说明 |
|------|------|------|
| GET/PUT | /api/admin/oss-config | 资源区 OSS 配置。 |
| GET     | /api/admin/employee-activity | 按人/项目/时间范围的推送记录与统计。 |
| GET     | /api/admin/employees | 员工列表与管理。 |

## 8. 前端页面结构建议

- **员工端**（独立入口或与现有前端同域不同路由）：
  - **登录页** → **首页/工作台**（快捷入口：资源区、我的项目）。
  - **资源区**：左侧树或面包屑，右侧当前目录下的子目录与文件；点击文件即下载。
  - **我的项目**：项目卡片列表 → 进入项目 → Tab「版本历史」+ 按钮「推送新版本」；推送时弹窗：版本说明 + 文件选择上传。
  - **版本详情**：版本说明、时间、文件列表、下载全部/单文件。
- **管理端**：在现有管理员后台增加「OSS 配置」「员工项目活动」等菜单与页面。

---

## 9. 非功能要点

- **安全**：OSS 密钥仅后端持有；所有员工端接口鉴权；下载链接临时有效。
- **性能**：大文件下载优先签名 URL；列表接口可对 OSS 列举结果做短期缓存（如 1 分钟）减轻 OSS 调用量。
- **可维护性**：OSS 配置与项目/版本表分离，便于日后更换存储或扩展多 Bucket。
