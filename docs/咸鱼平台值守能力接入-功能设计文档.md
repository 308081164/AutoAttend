## 2. 可行性分析

### 2.1 咸鱼平台技术现状

咸鱼（闲鱼）是阿里巴巴旗下二手交易平台，目前**没有公开的官方开放 API**。接入咸鱼能力只能通过以下方式：

| 方式 | 可行性 | 风险与限制 |
|------|--------|-----------|
| **咸鱼网页版（Web）** | ✅ 可行 | 咸鱼有 Web 版（`https://2.taobao.com`），登录依赖淘宝/阿里系账号体系（扫码或 Cookie）。页面结构可能随版本更新变化。 |
| **咸鱼 APP 协议模拟** | ⚠️ 高难度 | APP 接口经过加密（风控严格），模拟难度极高，且违反用户协议可能导致封号。 |
| **咸鱼开放平台** | ❌ 不存在 | 咸鱼未提供官方开放 API。 |
| **RPA/浏览器自动化** | ✅ 可行 | 通过 Puppeteer/Playwright 等无头浏览器模拟人工操作，可绕过无 API 的限制。 |

### 2.2 推荐技术方案：浏览器自动化（Puppeteer/Playwright）

**核心思路**：在服务端使用无头浏览器（Headless Browser）模拟用户登录咸鱼网页版，通过 DOM 操作实现消息监听、商品管理、文件发送等操作。

| 维度 | 说明 |
|------|------|
| **技术栈** | Node.js（Puppeteer/Playwright）+ Java 后端（Spring Boot）通过 HTTP 或消息队列通信 |
| **部署** | 独立容器（`xianyu-worker`），与主应用同 Docker 网络 |
| **多账号** | 每个账号一个独立的浏览器上下文（Browser Context），隔离 Cookie 与会话 |
| **消息提醒** | 轮询监听页面 DOM 变化 → 通过 WebSocket 推送到前端 |
| **文件发送** | 通过浏览器自动化操作文件上传 DOM 元素 |

### 2.3 风险与限制

| 风险 | 等级 | 缓解措施 |
|------|------|---------|
| **账号被封** | 🔴 高 | 控制操作频率、模拟真实人类行为（随机延迟、鼠标轨迹）、避免批量操作 |
| **页面结构变更** | 🟡 中 | 使用语义化选择器 + 定期自动化测试 + 降级提醒 |
| **登录态过期** | 🟡 中 | 检测登录态失效后自动暂停并通知管理员重新扫码 |
| **法律合规** | 🟡 中 | 仅用于已授权账号的自动化辅助，不涉及爬取他人数据；用户协议风险需法务评估 |
| **多账号并发** | 🟢 低 | 每个账号独立浏览器上下文，资源消耗可控 |

```

│                    AutoAttend 平台                        │

│  │  WebSocket   │   │  │  ├─ Account 2 (Context) │  │    │
│  │              │   │  │  ├─ ...                 │  │    │
│  └──────────────┘   │  └────────────────────────┘  │    │
│                     └──────────────────────────────┘    │
│  ┌──────────────────────────────────────────────┐       │
│  │  前端 (Vue)                                   │       │
│  │  ├─ 咸鱼值守面板（多账号管理）                  │       │

│  │  ├─ 商品管理（上架/编辑/下架）                  │       │
│  │  └─ 文件发送                                  │       │
│  └──────────────────────────────────────────────┘       │
└─────────────────────────────────────────────────────────┘
```

### 3.1 组件说明

| 组件 | 职责 | 技术选型 |
|------|------|---------|
| **xianyu-worker** | 独立 Node.js 服务，管理 Puppeteer 浏览器实例，执行咸鱼页面操作 | Node.js + Puppeteer + Express |
| **主后端** | 提供 REST API 供前端调用，通过 HTTP 调用 xianyu-worker 执行操作，通过 WebSocket 推送消息提醒 | Spring Boot + WebSocket |
| **前端面板** | 咸鱼值守管理界面，多账号切换、消息查看、商品管理 | Vue 2 + Element UI |

### 3.2 通信协议

```
前端 → 主后端: REST API / WebSocket
主后端 → xianyu-worker: HTTP REST (内部 API)
xianyu-worker → 主后端: Webhook / 轮询结果
主后端 → 前端: WebSocket 推送
```

## 4. 分期实施计划

### Phase 1：基础值守（MVP）— 预计 3-4 周

**目标**：实现多账号同时登录咸鱼网页版，实时监听消息并推送提醒。

#### 4.1 功能清单

| 功能 | 说明 |
|------|------|
| **账号管理** | 添加/删除咸鱼账号，每个账号独立存储 Cookie（加密） |
| **扫码登录** | 首次添加账号时生成二维码，管理员扫码后自动保存登录态 |
| **多账号同时在线** | 每个账号独立浏览器上下文，可同时保持登录态 |
| **消息监听** | 轮询检测新消息（每 30-60 秒），检测到新消息时触发提醒 |
| **消息提醒** | 前端页面内通知 + 可选系统通知/邮件提醒 |
| **消息列表查看** | 在平台内查看咸鱼会话列表与消息内容（只读） |
| **快捷回复** | 预设回复模板，一键发送文字消息 |
| **登录态检测** | 定期检测 Cookie 有效性，失效时通知管理员重新登录 |

#### 4.2 数据模型


#### 4.3 前端界面

- **咸鱼值守面板**：左侧账号列表（显示在线状态），右侧消息面板
- **消息提醒**：新消息时页面标题闪烁 + 右下角弹窗通知
- **快捷回复**：预设回复模板管理（管理员可配置常用回复）

## 8. 技术实现要点

### 8.1 Puppeteer 操作核心逻辑

```javascript
// 伪代码示例
class XianyuAccountSession {
  constructor(accountId, cookie) {
    this.browser = await puppeteer.launch({ headless: true });
    this.page = await this.browser.newPage();
    await this.page.setCookie(...cookie);
  }

  // 监听新消息
  async startPolling(callback) {
    setInterval(async () => {
      await this.page.goto('https://2.taobao.com/message/');
      const messages = await this.page.evaluate(() => {
        // 解析 DOM 获取消息列表
        return parseMessages();
      });
      if (messages.length > 0) callback(messages);
    }, 30000);
  }

  // 发送消息
  async sendMessage(conversationId, text) {
    await this.page.goto(`https://2.taobao.com/message/${conversationId}`);
    await this.page.type('.chat-input', text);
    await this.page.click('.send-btn');
  }

  // 上传文件
  async sendFile(conversationId, filePath) {
    const input = await this.page.$('input[type="file"]');
    await input.uploadFile(filePath);
    await this.page.click('.send-btn');
  }
}
```

### 8.2 反检测策略

| 策略 | 实现 |
|------|------|
| **随机行为模拟** | 每次操作前随机延迟 500-3000ms，模拟鼠标移动轨迹 |
| **浏览器指纹** | 使用 `puppeteer-extra-plugin-stealth` 插件隐藏自动化特征 |
| **User-Agent 轮换** | 每次会话使用不同的 UA 字符串 |
| **操作频率限制** | 单账号每分钟操作不超过 5 次，消息轮询不低于 30 秒间隔 |
| **代理 IP** | 可选为每个账号配置独立代理 IP |

### 8.3 登录态管理

```
首次添加账号:
1. 后端生成一次性登录链接
2. 管理员打开链接，看到咸鱼登录二维码
3. 扫码登录后，xianyu-worker 捕获 Cookie
4. Cookie 加密存储到数据库

登录态检测:
1. 每 30 分钟检测一次 Cookie 有效性
2. 尝试访问咸鱼首页，检测是否跳转到登录页
3. 若失效，标记账号状态为 expired 并通知管理员
```

## 9. 非功能需求

| 需求 | 说明 |
|------|------|
| **资源消耗** | 每个账号约 100-200MB 内存（Puppeteer 实例），建议单机不超过 20 个账号 |
| **高可用** | xianyu-worker 崩溃后自动重启，账号会话自动恢复 |
| **安全性** | Cookie 加密存储（AES-256），仅运行时解密 |
| **可观测性** | 记录所有自动化操作日志，便于排查问题 |
| **降级策略** | xianyu-worker 不可用时，前端展示离线状态，不影响平台其他功能 |

## 10. MVP 验收标准

- [ ] 管理员可添加咸鱼账号（扫码登录）
- [ ] 多个账号可同时保持在线状态
- [ ] 新消息到达时前端有提醒（页面内通知）
- [ ] 管理员可在平台内查看消息列表与内容
- [ ] 管理员可发送预设的快捷回复
- [ ] 登录态失效时自动通知管理员
