# 基于 GitHub Webhook 的员工开发状态推断智能体（Vue + Spring Boot）PRD v2

## 1. 背景与目标
研发管理中常见痛点是“进展不可见、状态同步成本高、口径不一致”。本项目通过接入 **GitHub Webhook** 事件流，结合仓库数据（commit/diff、PR、CI）与智能体推断规则（MVP以规则为主，后续可接入大模型），自动推断员工与团队的“开发状态”，并提供 **仅管理员可见** 的管理看板、查询与预警能力。

- **目标**：用“客观事件 + 可解释规则 +（可选）AI分析/总结”替代人工同步。
- **非目标**：不对个人绩效做直接裁决（仅做事实与推断、给出依据与置信度）。MVP阶段以“管理员侧效率提升”为主，不面向员工开放UI。

## 2. 用户画像与使用场景
- **管理员（MVP唯一前台用户）**：配置GitHub Webhook/仓库、维护员工映射关系、查看各员工状态与证据（含完整diff）、处理预警与审计。
- **研发负责人/主管（后续版本开放）**：查看团队状态与风险。
- **开发者本人（后续版本开放）**：查看系统推断、纠错/补充、生成日报。

典型场景（管理员视角）：
- 每天/每周查看：各员工当前状态（开发/评审/修复CI/阻塞/无信号）及**证据链（含commit diff）**。
- 风险控制：长时间无提交、无效提交占比异常、PR堆积、CI持续失败。

## 3. 核心概念与术语
- **事件（Event）**：GitHub Webhook 推送到系统的原始事件（push、pull_request、pull_request_review、check_suite/workflow_run 等）。
- **活动（Activity）**：对事件的规范化抽象（一次 commit、一次 PR 评论/审批、一次 CI 失败等）。
- **开发状态（Dev Status）**：系统推断的员工当前状态（见 3.1）。
- **状态证据（Evidence）**：导致该推断的事件/活动及其关键字段（可追溯）。
- **置信度（Confidence）**：对推断可靠性的量化（0~1 或 0~100）。
 - **Diff 存档（Diff Archive）**：对每个 commit 的 `git diff` 完整文本保存，用于管理员查看与后续AI分析。

### 3.1 员工开发状态（建议状态机）
系统对“员工-仓库/项目”维度维护一个当前状态（可按需汇总到员工全局状态）。

- **Active Coding（活跃开发）**：近期有有效代码提交或在推进分支。
- **In Review（评审中）**：近期主要活动是发起/更新 MR、回复评审、处理review意见。
- **Reviewing Others（评审他人）**：近期主要活动是审核他人 MR、给出建议/批准。
- **CI Fixing（修复构建/测试）**：近期 pipeline 失败并持续提交修复。
- **Blocked（阻塞）**：Issue/MR 被标记阻塞、或长期等待他人review/依赖、或持续失败且无推进。
- **Idle / No Signal（无信号）**：在工作时间窗口内缺少可观测事件（需要谨慎展示）。

> 注：状态不是“好坏”，而是“在做什么”。UI需展示证据与时间窗口，避免误解。

## 4. 产品范围（MVP 与迭代）
### 4.1 MVP（必须交付）
- **仅接入 GitHub 的 Webhook 与事件落库**
  - 支持 GitHub：`push`、`pull_request`（可选：`pull_request_review`、`workflow_run`/`check_suite`）。
  - 事件规范化（Event → Activity），并按 `delivery_id` 幂等去重。
- **有效活动识别**
  - 识别“有效提交/无效提交/机械提交”（如仅格式化、仅改锁文件、仅改注释等）。
  - 统计：提交次数、影响文件数、增删行、涉及模块。
- **完整 Diff 拉取、保存与可查看（MVP关键约束）**
  - 对每个 commit 生成并保存完整 `git diff` 文本（before→after 或单commit diff，见第11章实现建议）。
  - 管理员可在后台按员工/仓库/时间/commit 查询并查看完整diff。
- **开发状态推断引擎（规则优先，可解释）**
  - 基于时间窗口与事件权重推断状态，产出：状态 + 置信度 + 证据列表。
- **管理员后台（唯一UI）**
  - 团队概览：按状态分布、最近24h/7d 活跃度、异常列表。
  - 员工详情：状态时间线、最近活动、指标 + **完整diff查看**。
- **预警**
  - 无信号、无效提交占比异常、PR长期未推进、CI持续失败。
- **权限与审计（MVP简化）**
  - RBAC：**仅管理员**（后续扩展主管/成员）。
  - 审计：管理员查看diff与敏感数据的审计日志。

### 4.2 V1.1（增强）
- 多角色与权限：主管/成员视图，组织架构（部门/小组）管理。
- 大模型分析（可选开关）：基于 **已存档的完整diff** + 活动上下文，输出代码质量/有效性/风险点与日报周报总结（见附录预留设计）。
- 手工纠错/补充（开发者可标注“在做设计/沟通/排障”，并设置有效期）。

### 4.3 V2（高级）
- 需求流转闭环：与 Jira/禅道/飞书项目/工单系统对接。
- 预测：基于历史节奏预测风险（review瓶颈、CI不稳定、交付延期概率）。

## 5. 业务流程（端到端）
1. GitHub 触发事件（push/PR/CI等）→ Webhook 发到后端。
2. 后端校验签名、解析事件、入库（Event）。
3. 事件规范化为活动（Activity），计算基础指标（变更规模、文件类型分布、是否疑似无效提交等）。
4. 后端在本地仓库执行 `git fetch`，生成并保存完整 `git diff`（Diff Archive）。
5. 推断引擎在“员工×仓库”维度更新状态（Status），生成证据（Evidence）与置信度。
6. 管理员后台展示状态分布、个人时间线、预警列表与diff详情。
7. 定时任务生成预警（后续可扩展推送到企业微信/飞书/邮件）。

## 6. 状态推断规则（可解释、可配置）
### 6.1 时间窗口与衰减
- 默认窗口：最近 **4小时**（实时状态）、最近 **24小时**（日活跃）、最近 **7天**（趋势）。
- 对活动赋权重，随时间衰减：\(score = weight \times e^{- \lambda \Delta t}\)。

### 6.2 活动类型与权重（示例，可配置）
- **Push/Commit**
  - 有效提交：+6
  - 小修复（<10行）：+2
  - 疑似无效提交（仅空白/格式化/版本号/锁文件）：+0.5
- **MR/PR**
  - 新建 MR：+5（倾向 In Review）
  - 更新 MR（push 到同分支）：+4（In Review 或 Active Coding）
  - 回复review / 处理建议：+4（In Review）
  - 审核他人 MR（comment/approve/request changes）：+4（Reviewing Others）
- **CI/Pipeline**
  - Pipeline 失败：触发 CI Fixing 候选
  - 失败后短时间内连续修复提交：CI Fixing 权重 +5
  - Pipeline 持续失败且无修复推进：Blocked 候选 +6
- **Issue/Label**
  - 标记 blocked / dependency：Blocked +8
  - 需求进入 in progress：Active Coding 候选 +3

### 6.3 状态决策（示例）
在同一窗口内汇总各状态得分，取最高者为当前状态；同时给出：
- **置信度**：最高分与次高分差距、证据数量、事件来源多样性决定。
- **证据**：最近N条关键活动（默认5条）。

优先级兜底：
- 若出现显式 blocked 标签/字段 → **Blocked** 最高优先级。
- 若 pipeline 失败 + 有修复提交 → **CI Fixing**。
- 若最近以 MR 交互为主 → **In Review / Reviewing Others**。
- 若仅有零碎小提交且无MR/Issue → 置信度降低，可能仍为 Active Coding。
- 若工作时间窗口内无事件 → **Idle / No Signal**（必须标注“可能在离线工作/会议/设计”等提示）。

### 6.4 “有效提交”识别（MVP版规则）
默认不引入复杂静态分析，采用可解释规则：
- **强无效**（命中即判为无效/低价值，仍记录）：仅改空白、仅改注释、仅改版本号、仅改 lock 文件（可配置白名单/黑名单）。
- **弱无效**：单文件且<5行，且文件类型为文档/配置（可降权）。
- **有效**：涉及业务代码（java/js/ts/vue等）且变更规模达到阈值，或与MR/Issue关联。

可选增强：对 diff 片段做轻量模型/大模型归类（“格式化/重构/修复bug/新增功能/测试”等），但必须输出结构化结论并保留证据。

## 7. 功能需求（前端：仅管理员后台）
### 7.1 管理员登录
- 本地账号 + JWT（MVP）；后续可接公司SSO。

### 7.2 员工状态看板（管理员）
- 状态分布：Active Coding / In Review / Reviewing / CI Fixing / Blocked / Idle
- 列表：员工、当前状态、置信度、最近活动时间、今日有效提交数、无效提交占比（MVP指标可先简化）
- 筛选：仓库/时间窗口（4h/24h/7d）、员工

### 7.3 员工详情（管理员）
- 状态时间线（最近7天）
- 活动/证据卡片：commit/PR/CI摘要（含跳转到 GitHub）
- Commit 列表：按时间展示每次提交的指标（新增/删除行、文件数、有效性判定）
- **Diff 查看器（MVP必做）**
  - 查看完整diff（支持折叠文件、搜索、按文件分段）
  - 大diff分页/懒加载（后端按文件或按块返回）

### 7.4 预警中心（管理员）
- 无信号（可配置阈值，如工作日 > 6h）
- 无效提交占比异常（如24h内 > X%）
- PR长期未推进（如 > 24h 无更新/无review）
- CI持续失败（连续 N 次失败）

### 7.5 配置中心（管理员）
- GitHub Webhook 配置指引（secret、事件类型、回调URL）
- 仓库绑定：org/repo、clone地址、访问token（如需私有仓库）
- 员工映射：GitHub username/email ↔ 系统员工
- 规则配置：有效提交判定规则、阈值、工作时间窗口

## 8. 功能需求（后端）
- Webhook 接入（GitHub）

- git 本地仓库管理（clone/fetch/缓存/并发锁）
- 完整diff生成、落库与查询（支持大文本与分页）
- 推断引擎（规则配置化）
- 报表聚合接口（管理员视角：团队/个人/仓库维度）
- 定时任务（预警计算）
- **AI分析预留（不启用）**：分析任务表、结果表、统一的“分析器接口”（见附录）

## 9. 数据模型（建议）
### 9.1 表结构（核心）
- `org`：组织/租户
- `user`：系统用户
- `user_identity`：外部身份映射（git平台userId、username、email）
- `repo`：仓库
- `webhook_event`：原始事件（存payload摘要或加密存储）
- `activity`：规范化活动（类型、actor、repo、时间、外链URL、关键字段）
- `commit_metric`：commit维度指标（插入/删除/文件数、有效性、分类）
- `commit_diff`：commit diff 存档（完整diff文本或分块存储索引）
- `status_snapshot`：状态快照（user、repo/项目、status、confidence、evidence_ids、at_time）
- `alert`：预警（类型、级别、状态、关联对象、证据）
- `audit_log`：审计日志
 - `ai_analysis_job`（预留）：AI分析任务队列表
 - `ai_analysis_result`（预留）：AI分析结果表（结构化JSON + 评分）

### 9.2 字段要点
- 所有实体带：`id`, `org_id`, `created_at`, `updated_at`
- `webhook_event`：`source`(固定github), `event_type`, `delivery_id`, `signature_valid`, `payload_json`(建议加密存储)
- `status_snapshot`：`status`, `confidence`, `reason_summary`, `evidence_json`
 - `commit_diff`（建议字段）
   - `repo_id`, `commit_sha`, `parent_sha`（可选）
   - `diff_text`（LONGTEXT / TEXT / CLOB）
   - `diff_size_bytes`
   - `encrypted`（bool，是否加密存储）
   - `storage_mode`（inline/chunk）
   - `chunk_count`（如分块）

## 10. 接口设计（REST，MVP）
### 10.1 Webhook 接入
- `POST /api/webhooks/github`：接收 GitHub webhook
  - 校验签名/secret
  - 返回 `200 OK`（尽量快速），异步处理事件（建议消息队列；MVP可先本地异步线程/任务表）

### 10.2 看板与详情
- `GET /api/admin/dashboard?range=4h|24h|7d&repoId=`
- `GET /api/admin/users/{userId}/status?range=7d&repoId=`
- `GET /api/admin/users/{userId}/activities?range=7d&repoId=&page=`
- `GET /api/admin/commits?userId=&repoId=&range=7d&page=`
- `GET /api/admin/commits/{commitSha}/diff`（返回完整diff；如过大支持分块：`?chunk=`）
- `GET /api/admin/alerts?range=7d&status=open|closed&type=`

### 10.3 配置（管理员）
- `GET /api/admin/rules`
- `PUT /api/admin/rules`（状态权重、阈值、文件黑白名单、工作时间窗口）
- `POST /api/admin/repos/bind`（绑定仓库与组织/团队）
- `POST /api/admin/users/identity/bind`（绑定外部身份）
 - `POST /api/admin/github/webhook/test`（可选，验证secret与连通性）

### 11.1 前端
- Vue 3 + Vite + TypeScript + Pinia + Vue Router

- ECharts 用于趋势图/分布图

### 11.2 后端

- Web：Spring Web MVC（Webhook 与 REST API）

- DB：MySQL 8 / PostgreSQL（二选一）
- 任务：Spring Scheduler（MVP）→ Quartz（需要更强调度时）
- 异步：Spring @Async（MVP）→ MQ（Kafka/RabbitMQ，后续）

### 11.3 代码平台对接
- Webhook：接收事件（GitHub）
- API（可选）：补拉 PR/commit 详情、补齐关联信息（需要 token）
- Git 操作（MVP必须）：本地 `git clone/fetch` + `git diff` 生成**完整diff**（注意性能、安全、并发锁）

> 存储建议（符合你的MVP要求）：**完整diff必须可保存与可查看**。生产环境建议默认开启“加密存储 + 严格RBAC + 审计 + 最小暴露（仅管理员）”。

## 12. 安全、合规与隐私
- **敏感数据保护**：MVP需要落库完整diff，因此必须：
  - 仅管理员可访问
  - 数据库加密（字段级或表空间/磁盘加密），并配合应用层二次加密（可选但推荐）
  - 对diff访问做审计（谁、何时、看了哪个commit）
- **权限隔离**：MVP仅管理员；后续扩展主管/成员时需按团队隔离。
- **审计**：记录查看diff、导出、删除等关键行为。
- **Webhook 安全**：签名校验、防重放、IP白名单（可选）。
- **数据保留**：事件原文可设置保留期（如30/90天）并支持清理。

## 13. 交付物与验收标准（MVP）
- 能成功接入 **GitHub** webhook，稳定落库并去重。
- 管理后台能展示员工当前状态、置信度、证据（可追溯到 GitHub 链接）。
- **能查询并查看每次提交的完整 `git diff`**（支持大diff的分块/分页传输）。
- 推断规则可配置，且同一事件在重复投递下不会重复计数。
- 预警至少覆盖：无信号、无效提交占比异常、PR长期未推进、CI持续失败。
- 权限体系可用：**仅管理员可登录与查看diff**；未授权请求返回401/403。

## 14. 风险与对策
- **事件不全导致误判**：UI必须展示“证据与时间窗口”，并对 Idle/No Signal 明确提示。
- **diff落库带来合规/泄露风险**：强制管理员RBAC、加密与审计；支持保留期与清理；禁止前端缓存diff。
- **性能与存储**：diff 不宜全量存储；对大仓库使用异步与限流。
 - **性能与存储（在必须全量diff的前提下）**：对diff采用分块存储/压缩（如gzip）与按需加载；仓库本地缓存与并发锁；为单仓库设置队列避免git目录竞争。

你希望后续加入“DeepSeek 判断代码质量、是否有效工作”等能力。为避免未来大改，建议在MVP就预留以下结构，但**不开启实际调用**：

### A. 分析器接口（后端抽象）
- `Analyzer` 接口（或策略模式）：输入 `commitSha + diff_text + context`，输出结构化JSON
- 通过配置启用：`analysis.enabled=false`（MVP默认关闭）

### B. 分析任务与结果数据（表已预留）
- `ai_analysis_job`
  - `job_id`, `repo_id`, `commit_sha`, `status(pending/running/success/failed)`, `model`, `prompt_version`, `created_at`
- `ai_analysis_result`
  - `commit_sha`, `workload(低/中/高)`, `is_valid_commit`, `quality_comment`, `level(初/中/高)`, `risk_flags(json)`, `raw_response(json)`, `created_at`

### C. 后续对 DeepSeek 的调用形态（建议）
- 输入：**完整diff（来自`commit_diff`）** + 关键元数据（作者、时间、文件类型分布、PR链接）
- 输出：严格JSON（便于存储与筛选），并记录 `prompt_version` 以便回溯
- 风控：diff截断策略（超大diff分文件/分块分析），调用限流与重试，敏感信息脱敏（可选）
